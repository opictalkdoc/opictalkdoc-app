### P-5. 리브랜딩: 오픽톡닥 → 하루오픽 ⏳

**핵심 질문**: 서비스 이름과 도메인을 변경할 것인가?

**배경 (2026-02-25)**:
- "오픽톡닥" 문제점: 4음절 발음 어려움, 도메인 11자(opictalkdoc) 타이핑 오타 잦음
- 메타포 혼재: 이름/로고=의료, 랜딩=연극/무대, 메뉴=학습 — 서비스 본체와 이름이 동떨어짐
- 실제 메뉴(시험후기, 스크립트, 모의고사, 튜터링)는 "체계적 훈련 과정"이며 의료 메타포와 무관

**검토 과정**:
1. 의료 메타포 유지 (오픽닥, 오픽룸) → 서비스 본체와 안 맞음
2. 훈련/연구소 계열 (오픽랩, 오픽캠프) → 랜딩 톤(따뜻한, 위로)과 안 맞음
3. 일상/감성 계열 → 랜딩 카피("평범한 하루가 대본")와 직결

**결정 (2026-02-25)**: **하루오픽 (HaruOPIc)** 확정
- **이름**: 하루오픽
- **도메인**: haruopic.com (8자, 구매 가능 확인)
- **발음**: 하-루-오-픽 (4음절, 부드러운 진입)
- **의미**: "하루의 OPIc" — 일상이 곧 대본, 매일의 OPIc 준비
- **컨셉**: "나는 내 삶의 주인공이다" (유지)
- **태그라인**: "말하다, 나답게." (유지)
- **도메인 가독성**: haru + opic → 음절 경계 명확 (opicharu는 "오피차루" 오독 위험)

**왜 "하루"가 앞인가**:
- "하루"로 시작 → 일상/따뜻함이 먼저 전달 → 20대 감성
- "오픽"으로 끝 → 서비스 성격 확인
- 랜딩 카피 "평범한 **하루**가 가장 완벽한 대본입니다"와 직결

**변경 범위**:
| 항목 | 내용 | 난이도 |
|------|------|--------|
| 도메인 | haruopic.com 구매 + DNS(Vercel/Spaceship) | 낮음 |

| 로고 | 반창고 아이콘 + "하루오픽" 텍스트 재생성 | 중간 |

**도메인 전환 전략 (양쪽 동시 운영)**:
- **주 도메인**: haruopic.com (새 브랜드)
- **리다이렉트**: opictalkdoc.com → haruopic.com (301 리다이렉트)

| # | 문제 | 위험도 | 결정 |

| A-6 | JSONB 과다 사용 | 낮음 | question_id 배열만 저장, JOIN으로 조회 |

### T-9. 하이브리드 백엔드 전략 ✅ (큰 방향만 확정, 모듈별 상세는 미정)

**핵심 질문**: Server Actions만 쓸 것인가, Edge Functions도 활용할 것인가?

**배경 (T-8 재검토)**:
- T-8에서 "Edge Functions 사용 안 함"으로 결정했으나, 실제 모듈 분석 결과 AI API 호출(GPT, Whisper, Gemini TTS, Azure Speech)이 다수 존재
- 이런 AI 작업은 10~60초 이상 소요 → Vercel Hobby 서버리스 함수 제한(10초)에 걸릴 수 있음
- Supabase Pro($25)에 Edge Functions 무료 포함 → 추가 비용 없음

**결정 (2026-02-23)**: 하이브리드 — 작업 성격에 따라 분리

| 작업 성격 | 백엔드 방식 | 예시 |
|----------|-----------|------|
| **가벼운 CRUD** | Server Actions | DB 읽기/쓰기, 세션 관리, 목록/상세 조회 |
| **무거운 AI API** | Edge Functions | GPT 생성/채점, Whisper STT, Gemini TTS, Azure 발음 |

**모듈별 상세 적용 (⏳ 초안 — 각 모듈 이관 시 소리담 파악 후 확정)**:

| 모듈 | Server Actions (예상) | Edge Functions (예상) | 확정 |

### F-3. Edge Function 구조 ✅

**핵심 질문**: 모의고사 Edge Function을 어떤 구조로 구성할 것인가?

**결정 (2026-03-02)**: **3개 EF (AI만) + Server Actions (CRUD)**

**변경 근거**:
1. **Vercel 4.5MB Body 제한**: WAV 16kHz 4분 = ~7.5MB → SA/EF 경유 시 긴 녹음 실패
2. **T-9 일관성**: "CRUD = SA, AI = EF" 원칙 유지 (시험후기·스크립트와 동일 패턴)
3. **비용 최적화**: 오디오 대역폭 Vercel 무관 ($0), 폴링도 Supabase 직접 (F-8)
4. **배포 단순화**: CRUD EF 별도 배포 불필요, SA는 Next.js 코드베이스 내 관리

**확정 구조**:

| 계층 | 역할 | 트리거 |
|------|------|--------|
| **Client → Supabase Storage** | 오디오 직접 업로드 (7.5MB, 프록시 없음) | 녹음 완료 |
| **Client → Supabase DB** | 상태 폴링 5초 (F-8) | `setInterval` |
| **Server Actions** | CRUD (세션 생성, 메타데이터 저장, 이력 등) | Client 호출 |
| **SA → fire-and-forget** | AI 처리 체인 시작 | 답변 메타데이터 저장 후 |

| EF | 역할 | 예상 시간 | 트리거 |
|----|------|-----------|--------|
| `mock-test-process/` | Stage A: Whisper STT + Azure 발음 평가 | ~70초 | SA fire-and-forget |
| `mock-test-eval/` | Stage B: GPT-4.1 체크박스 평가 | ~100초 | Stage A fire-and-forget |
| `mock-test-report/` | Stage C: 규칙엔진 + GPT 리포트 생성 | ~45초 | Stage B fire-and-forget (Q15 완료 시) |

**답변 제출 플로우 (최적화)**:
```
[Client] → Supabase Storage 직접 업로드 (WAV, 프록시 없음, 50MB 제한)

[Client] → SA submitAnswer (URL + 메타데이터만, 수 KB)
         → DB INSERT (answers)
         → Q1이면: eval_status='skipped' 끝
         → Q2~15: fire-and-forget HTTP → mock-test-process/
           → fire-and-forget → mock-test-eval/ (체크박스 평가)
             → Q15 완료 시 → fire-and-forget → mock-test-report/ (규칙엔진 + 리포트)
```

### F-8. 상태 모니터링 방식 ✅

**핵심 질문**: 평가 진행 상태를 프론트에서 어떻게 추적할 것인가?

**결정 (2026-03-02)**: **A — 폴링 5초 간격 + Supabase 직접 조회**

- **폴링 경로**: 브라우저 → Supabase 직접 조회 (Vercel 무관, 비용 $0)
- **간격**: 5초 기본, 에러 시 exponential backoff (최대 30초)
- **stuck 복구**: pending 항목 있는데 processing 0이면 EF 자동 재호출
- **단계별 진행 표시**: `eval_status` 변화를 폴링으로 감지 → UI 반영
  ```
  processing → "음성 분석 중..."
  stt_completed → "체크박스 평가 중..."
  completed → "평가 완료!" → 결과 표시
  ```

**Realtime 미채택 근거**:
- 개별 평가 소요 ~170초 → 5초 지연은 전체 대비 ~3% (체감 차이 없음)
- 소리담에서 폴링만으로 검증됨
- Supabase 직접 조회로 Vercel 비용 무관
- 부하: 동시 100명 기준 초당 20쿼리 — 인덱싱된 단일 행 SELECT, DB 부담 없음
- Realtime은 추후 사용자 피드백 기반 개선 옵션으로 보류

**기존 설계 대비 변경**: Supabase Realtime + 폴링 폴백 → **폴링만 (단순화)**

### F-12. 평가 프롬프트 관리 ✅

**핵심 질문**: 개별 평가(8종) + 종합 평가(1종) 프롬프트를 어떻게 관리할 것인가?

#### 소리담 현황 분석

**프롬프트 2종류**:

| 구분 | 개별 평가 (Individual) | 종합 평가 (Comprehensive) |
|------|----------------------|------------------------|
| EF | `mock-test-evaluate-individual-v7` | `mock-test-report-v7` |
| 호출 시점 | 문항 답변 제출 직후 (14회) | Q15 완료 후 (1회) |
| GPT 역할 | 체크박스 pass/fail + 교정/분석 | 진단 리포트 생성 (등급은 규칙엔진) |
| DB 프롬프트 | `eval_v7_{answer_type}_eval` (8종) | `eval_v7_final_evaluation` (1종) |
| 저장 테이블 | `evaluation_prompts` | `evaluation_prompts` (동일) |

**소리담 프롬프트 모드 3종** (`mock_test_eval_settings.prompt_mode`):

**소리담 프롬프트 네이밍** (`answer_type` 기반 8종):
- `eval_v7_description`, `eval_v7_routine`, `eval_v7_comparison`
- `eval_v7_past_experience`, `eval_v7_roleplay_11`, `eval_v7_roleplay_12`
- `eval_v7_advanced_14`, `eval_v7_advanced_15`
- + `eval_v7_final_evaluation` (종합)

**관리 방식**: `.md` 파일 원본 → `register-v7-prompts.js` 스크립트 → DB PATCH → `evaluation_prompts` 테이블

### F-15. 네이밍 일관성 ✅

**핵심 질문**: 소리담의 `answer_type` 기반 네이밍을 `question_type`으로 전환할 때, 영향 범위와 적용 전략은?

#### 현황: 소리담의 `answer_type` 사용처

| 위치 | 사용 방식 | 예시 |
|------|----------|------|
| **EF 프롬프트 조회** | `eval_v7_{answer_type}_eval` | `eval_v7_description_eval` |
| **체크박스 ID** | `{TAG}-{n}-{suffix}` (answer_type과 직접 연관 없음) | `INT-1-1`, `ADV-2-SP` |
| **enabled 토글** | `enabled_{answer_type}` | `enabled_roleplay_11`, `enabled_comparison` |
| **EF 분기 로직** | `switch(answer_type)` | description → INT 체크박스, comparison → ADV 체크박스 |
| **DB 프롬프트 key** | `eval_v7_*` 접두사 | `eval_v7_past_experience` |
| **규칙엔진 V7** | question의 answer_type으로 평가 로직 분기 | `getCheckboxSet(answer_type)` |

#### 오픽톡닥의 `question_type` 체계

questions 테이블에 이미 `question_type_eng` 10종이 존재:
`description`, `routine`, `comparison`, `experience_specific`, `experience_habitual`, `experience_past`, `comparison_change`, `social_issue`, `asking_questions`, `suggest_alternatives`

**핵심 이슈**: 소리담 `answer_type` 8종 → 오픽톡닥 `question_type` 10종 (경험 3분화)

#### 영향도 분석

| 영역 | 변경 사항 | 복잡도 |
|------|----------|--------|
| **프롬프트 ID** | `eval_v7_description_eval` → `eval_description` (v7 제거 + _eval 제거) | 낮 |
| **enabled 토글** | `enabled_roleplay_11` → `enabled_asking_questions` (question_type 기반) | 낮 |
| **EF 분기 로직** | `answer_type` → `question_type` 변수명 치환 + 경험 3종 매핑 추가 | 중 |
| **규칙엔진** | 체크박스 세트 매핑 함수 시그니처 변경 | 중 |
| **DB 프롬프트 key** | 새로운 네이밍 컨벤션 결정 필요 | 낮 |

#### 선택지

| 안 | 전략 | 장점 | 단점 |
|----|------|------|------|
| **A** | **전면 `question_type` 통일** — 모든 곳에서 `question_type_eng` 값 사용 | questions 테이블과 완전 일관, 직관적 | 경험 3종 → 같은 프롬프트 매핑 로직 필요 |
| **B** | **하이브리드** — DB/API는 `question_type`, EF 내부 프롬프트 조회만 8종 그룹 키 유지 | 프롬프트 관리 단순 (9개), EF 내부 로직 깔끔 | 두 체계 혼재 (question_type ↔ prompt_group) |
| **C** | **최소 변경** — `answer_type`을 `question_type`으로 이름만 치환, 값 체계는 8종 유지 | 변경 최소, 소리담 로직 재활용 | questions 테이블의 10종과 불일치 |

**결정 (2026-03-02)**: **A안 — 전면 `question_type` 통일**

**근거**:
- questions 테이블의 `question_type_eng` 10종이 시스템의 SSOT → 모든 모듈이 이 값을 그대로 사용해야 일관성 유지
- 경험 3종 → 1 프롬프트 매핑은 단순한 함수 1개로 해결 가능 (복잡도 낮음)

**구현 명세**:

- **프롬프트 ID**: `eval_{question_type}` 패턴 (예: `eval_description`, `eval_asking_questions`)
- **경험 3종 매핑**: `getPromptKey(questionType)` 함수 — `experience_*` 3종 → `eval_past_experience` 반환
- **enabled 토글**: `enabled_{question_type}` (예: `enabled_asking_questions`, `enabled_suggest_alternatives`)
- **종합 프롬프트**: `eval_comprehensive`

```sql
CREATE TABLE evaluation_prompts (

);
```
