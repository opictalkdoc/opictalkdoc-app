### 2-1. 데이터 로딩 체인 — Before vs After

**Before (7차 적용 전):**
```
브라우저 → /reviews
│
├─ [서버] getStats()                        ← 순차 워터폴 60-110ms
│       ├─ await 쿼리1: submissions COUNT    (20-30ms)
│       ├─ await 쿼리2: combos 전체 SELECT   (20-40ms) → JS Set.size
│       └─ await 쿼리3: submissions 전체 SELECT (20-40ms) → JS Set.size
│
├─ [클라이언트] FrequencyTab useEffect       ← 이중 조회 40-60ms
│       └─ getFrequency() → combos 전체 SELECT
│
├─ [탭 전환] SubmitTab remount → getMySubmissions()   ← 매번 40-60ms
├─ [탭 전환] ListTab remount → getPublicReviews()     ← 매번 40-60ms
└─ [탭 복귀] FrequencyTab remount → getFrequency()    ← 동일 데이터 재요청
```

**After (7차 적용 후 — 현재):**

### 2-2. 문제별 분석 및 해결 상태

#### 문제 A: getStats() — 순차 워터폴 + 비효율 SQL → ✅ 해결 (7차)

**해결:** `getStats()` 삭제 → `getStatsAndFrequency()` 신규
- 3개 `await` 순차 → `Promise.all` 병렬
- combos 쿼리를 stats + frequency 공용으로 통합 (1회 조회)

#### 문제 B: 이중 데이터 로딩 (서버 stats + 클라이언트 frequency) → ✅ 해결 (7차)

**해결:** 서버에서 stats + frequency를 1회 조회 후 `initialData`로 전달
- FrequencyTab: `useQuery({ initialData: initialFrequency })` — 클라이언트 추가 왕복 제거

#### 문제 C: 탭 전환마다 캐시 없는 remount → ✅ 해결 (7차)

**해결:** 모든 탭 `useState+useEffect` → TanStack Query 전환
- FrequencyTab: `useQuery` (staleTime 5분)
- SubmitTab: `useQuery` + `invalidateQueries` (제출/삭제 시 자동 갱신)
- ListTab: `useInfiniteQuery` (필터별 캐시 + 페이지네이션 캐시)
- 탭 unmount/mount 되어도 QueryClient 캐시에 데이터 유지 → 0ms 즉시 표시

#### 문제 D: master_questions 조회도 캐시 없음 → ✅ 해결 (7차)

**해결:** `staleTime: Infinity`로 세션 내 영구 캐시
- TopicPagination: `useQuery({ queryKey: ["topics", category], staleTime: Infinity })`
- QuestionSelector: `useQuery({ queryKey: ["questions", topic, category], staleTime: Infinity, enabled })`

## 6. 교훈 및 원칙

### 반복된 패턴: "서버에서 할 일을 클라이언트에서, SQL에서 할 일을 JS에서"

| 잘못된 패턴 | 올바른 패턴 |
|------------|-----------|
| 전체 행 fetch → JS Set.size | SQL `COUNT(DISTINCT column)` |
| 3개 await 순차 | Promise.all 병렬 |
| 서버 stats + 클라이언트 frequency 이중 로드 | 서버에서 1회 통합 |
| 탭 전환마다 useState+useEffect로 매번 fetch | useQuery로 클라이언트 캐시 |
| 고정 마스터 데이터 매번 Server Action 호출 | staleTime: Infinity 영구 캐시 |

### 앞으로의 원칙

1. **읽기 전용 데이터는 useQuery**: staleTime 설정으로 불필요한 재요청 차단
2. **고정 데이터(master_questions)는 staleTime: Infinity**: 세션 내 1회만 로드
3. **DB 집계는 SQL에서**: JS에서 COUNT/DISTINCT 하지 않음
4. **병렬 가능한 쿼리는 Promise.all**: 순차 await 금지
5. **서버에서 한 번에 내려줄 수 있으면 통합**: 클라이언트 추가 왕복 최소화
