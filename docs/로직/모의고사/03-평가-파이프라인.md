# 03. 평가 파이프라인

> 답변이 제출되면 뒤에서 뭐가 일어나는지 — 4-Stage Edge Function 상세

---

## 전체 구조

```
답변 제출 (submitAnswer)
   │ fire-and-forget
   ▼
┌──────────────────────────────────────┐
│  Stage A: mock-test-process          │
│  Whisper STT + Azure 발음 + 스킵 판정│
│  → eval_status: processing → stt_completed
└────────────┬─────────────────────────┘
             │ fire-and-forget
             ▼
┌──────────────────────────────────────┐
│  Stage B-1: mock-test-eval-judge     │
│  GPT-4.1-mini 체크박스 74개 판정     │
│  + 과제충족 + feedback_branch 결정   │
│  → eval_status: evaluating → judge_completed
└────────────┬─────────────────────────┘
             │ fire-and-forget
             ▼
┌──────────────────────────────────────┐
│  Stage B-2: mock-test-eval-coach     │
│  fulfilled → GPT-4.1 코칭 6-Layer   │
│  partial  → GPT-4.1 축소 코칭       │
│  failed   → 룰 기반 구제 메시지      │
│  → eval_status: → completed
└────────────┬─────────────────────────┘
             │ 세션 completed + 모든 답변 완료 시
             │ fire-and-forget
             ▼
┌──────────────────────────────────────┐
│  Stage C: mock-test-report           │
│  규칙엔진 7-Step + FACT 점수         │
│  + GPT-4.1 종합 리포트 (9섹션)       │
│  + 성장 리포트 (2회차+)              │
│  → holistic_status: processing → completed
└──────────────────────────────────────┘
```

### fire-and-forget 체인 패턴

```typescript
// 각 Stage가 다음 Stage를 호출 (await 없이)
fetch(`${SUPABASE_URL}/functions/v1/mock-test-eval-judge`, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${SUPABASE_SERVICE_ROLE_KEY}`,
  },
  body: JSON.stringify(payload),
}).catch((err) => {
  console.error("fire-and-forget 호출 실패:", err);
});
// ← await 없음. 즉시 다음 코드로 진행.
```

대신 **수동 인증**으로 서비스 롤 키를 직접 검증:

```typescript
const authHeader = req.headers.get("authorization");
```

## Stage A: mock-test-process (STT + 발음)

### 입력

| 파라미터 | 소스 |
|---------|------|
| `session_id` | submitAnswer에서 전달 |
| `question_number` | 1~15 |
| `question_id` | questions 테이블 ID |
| `audio_url` | Storage 업로드 URL |
| `audio_duration` | 초 단위 |

### 처리 흐름

```
입력 수신
   │
   ├─ eval_status → 'processing'
   │
   ├─ ① 1차 스킵: 오디오 길이 판정
   │   → audio_duration ≤ 15초 → 전체 생략 (STT도 안 함)
   │   → eval_status → 'skipped'
   │   → evaluations INSERT (model='skipped', 구제 메시지 포함)
   │   → 종료
   │
   ├─ ② 오디오 다운로드
   │   → audio_url에서 WAV 파일 fetch
   │
   ├─ ③ Whisper STT (OpenAI whisper-1)
   │   → 3회 재시도 (exponential backoff: 1s → 2s → 4s)
   │   → 영어 언어 고정 (language: 'en')
   │
   ├─ ④ 텍스트 분석
   │   → 단어 수 (word_count)
   │   → 필러 단어 카운트 (um, uh, like, you know 등 15패턴)
   │   → WPM 계산 (단어수 / 초 × 60)
   │   → Long Pause 감지 (WPM 기반 휴리스틱)
   │       • WPM < 50 → 3회
   │       • WPM < 80 → 2회
   │       • WPM < 100 → 1회
   │
   ├─ ⑤ 2~3차 스킵: 트랜스크립트 판정
   │   → transcript ≤ 15자 → 스킵 (reason: 'short_transcript')
   │   → Whisper 환청 → 스킵 (reason: 'hallucination')
   │       • 환청 판정: 이모지만 있는 경우
   │   → 스킵 시: evaluations INSERT (구제 메시지 포함)
   │   → 종료
   │
   ├─ ⑥ Azure 발음 평가
   │   → WebSocket 기반 Azure Speech SDK
   │   → 3회 재시도
   │   → 실패해도 GPT 평가는 진행 (발음 데이터 없이)
   │   → 반환값:
   │       • accuracy_score (정확도, 0~100)
   │       • prosody_score (운율, 0~100)
   │       • fluency_score (유창성, 0~100)
   │       • mispronunciations (발음 오류 단어 목록)
   │
   ├─ ⑦ answers 업데이트: eval_status → 'stt_completed'
   │   → transcript, word_count, wpm, filler_word_count,
   │     long_pause_count, pronunciation_assessment 저장
   │
   └─ ⑧ fire-and-forget → Stage B-1 (eval-judge)
       → 페이로드: 위 데이터 + question_type
```

### 스킵 시 구제 메시지

스킵된 답변에도 구제 메시지를 생성하여 evaluations에 저장:

```typescript
const rescue = buildRescueMessage(questionType);

      recovery_tip: "\"One place I can describe is…\"로 시작해보세요.",
    },
  },
  priority_prescription: [{
    action: "\"One place I can describe is…\"로 시작하여 첫 문장 진입 연습",

  }],
```

## Stage B-1: mock-test-eval-judge (체크박스 판정)

### 역할

GPT-4.1-mini가 3가지를 판정:
1. **체크박스 74개** (pass/fail + evidence)
2. **과제충족 체크리스트** (🔴필수 / 🟡심화)
3. **feedback_branch** 결정 (fulfilled / partial / failed)

### 입력

Stage A에서 전달받은 전체 데이터 + question_type

### 처리 흐름

```
입력 수신
   │
   ├─ eval_status → 'evaluating'
   │
   ├─ ① 질문 정보 조회 (question_english, topic, category)
   │
   ├─ ② 사용자 target_level 조회 (user_metadata)
   │
   ├─ ③ 프롬프트 조회 (evaluation_prompts.key = 'eval_judge')
   │   → 프롬프트 미설정 시: evaluations INSERT (skipped) → 종료
   │
   ├─ ④ 동적 데이터 주입
   │   ├─ 체크박스 정의 텍스트 (question_type별)
   │   │   → getCheckboxIdsForQuestionType()
   │   │   → buildCheckboxDefinitionsText()
   │   │   → 표 형식으로 프롬프트에 주입
   │   │
   │   ├─ 과제충족 체크리스트 텍스트
   │   │   → buildTaskChecklistText()
   │   │   → 🔴필수 / 🟡심화 항목 + 목표등급별 threshold
   │   │
   │   └─ 변수 치환 (transcript, scores, question 정보 등)
   │
   ├─ ⑤ GPT-4.1-mini 호출 (JSON 모드)
   │   → temperature: 0.2 (일관성 중시)
   │   → max_tokens: 6000
   │   → 2회 재시도
   │   → truncation 감지 → 더 높은 max_tokens로 재시도 (8000)
   │
   ├─ ⑥ 결과 파싱
   │   ├─ checkboxes: { "INT-1-1": { pass: true, evidence: "..." }, ... }
   │   ├─ task_fulfillment: { status, checklist, completion_rate, reason }
   │   └─ feedback_branch: determineFeedbackBranch(task_fulfillment)
   │       → GPT의 task_fulfillment.status를 그대로 사용
   │       → "failed" → failed
   │       → "partial" → partial
   │       → 나머지 → fulfilled
   │
   ├─ ⑦ evaluations INSERT (B-1 결과)
   │   → checkboxes, pass_count, fail_count, pass_rate
   │   → task_fulfillment, feedback_branch
   │   → judge_model: 'gpt-4.1-mini'
   │
   ├─ ⑧ eval_status → 'judge_completed'
   │
   └─ ⑨ fire-and-forget → Stage B-2 (eval-coach)
       → B-1 판정 결과 전달 (checkboxes_summary, task_fulfillment, feedback_branch)
```

### 프롬프트 인젝션 방어

사용자 답변 (transcript)에 대해 새니타이징:

```typescript
function sanitizeUserInput(text: string): string {
  return text
    .replace(/---USER---|---SYSTEM---|---ASSISTANT---/gi, "[FILTERED]")
    .replace(/ignore\s+(all\s+)?previous\s+instructions/gi, "[FILTERED]")
    .replace(/you\s+are\s+now\s+a/gi, "[FILTERED]")
    .replace(/forget\s+(all\s+)?your\s+(previous\s+)?instructions/gi, "[FILTERED]")
    .slice(0, 10000);
}
```

## Stage B-2: mock-test-eval-coach (코칭 생성)

### feedback_branch별 분기

```
feedback_branch
   │
   ├─ "fulfilled" → GPT-4.1 표준 코칭 (eval_coach_standard 프롬프트)
   │   → 6-Layer 코칭 JSON 생성
   │   → temperature: 0.4, max_tokens: 8000
   │
   ├─ "partial" → GPT-4.1 축소 코칭 (eval_coach_partial 프롬프트)
   │   → 빠진 요소 강조 + 구조 교정
   │   → temperature: 0.4, max_tokens: 8000
   │
   └─ "failed" → 룰 기반 구제 (GPT 호출 없음!)
       → buildRescueMessage(question_type)
       → 비용 0원, 즉시 완료
```

### GPT 코칭 출력 구조 (6-Layer)

```json
{
  "coaching": {
    "one_line_insight": "한줄 핵심 통찰",
    "structure_evaluation": {
      "structure_label": "전개형",
      "structure_comment": "서론-본론-결론 구조가 잘 갖춰졌습니다."
    },
    "key_corrections": [
      {
        "original": "I go to gym every day",
        "corrected": "I go to the gym every day",
        "reason": "관사 the 누락",
        "drill_tag": "vocabulary_variety"
      }
    ],
    "better_version": "교정된 전체 답변 (학습자 어휘 수준 유지)",
    "delivery_insight": "전달력 분석",
    "strength": "잘한 점 1가지"
  },
  "priority_prescription": [
    {
      "action": "관사 사용법 집중 연습",
      "why": "관사 오류가 반복되어 유창성을 저해",
      "example": "the gym, the park, the library"
    }
  ]
}
```

### 처리 흐름

```
입력 수신 (B-1 결과 포함)
   │
   ├─ feedback_branch에 따른 분기 (위 참조)
   │
   ├─ evaluations UPDATE (B-2 코칭 결과 추가)
   │   → coaching_feedback, priority_prescription
   │   → model: 'gpt-4.1' 또는 'rule_based'
   │
   ├─ answers 업데이트: eval_status → 'completed'
   │
   └─ Stage C 트리거 조건 확인:
       ├─ 세션 status = 'completed'?
       │   └─ 아직 active → 트리거 안 함 (다음 답변 또는 completeSession에서)
       │
       └─ 모든 답변 eval_status IN ('completed','skipped','failed')?
           └─ 예 → fire-and-forget → mock-test-report
           └─ 아니요 → 대기 (다음 답변의 B-2에서 재확인)
```

## Stage C: mock-test-report (종합 리포트)

### 입력

- `session_id` 하나만 받음 (나머지는 DB에서 조회)

### 처리 흐름

```
입력 수신
   │
   ├─ holistic_status → 'processing'
   │
   ├─ ① 데이터 조회 (병렬)
   │   ├─ mock_test_sessions (user_id, question_ids)
   │   ├─ mock_test_evaluations (전 문항 평가 결과)
   │   ├─ mock_test_answers (audio_duration, word_count)
   │   └─ questions (question_english)
   │
   ├─ ② 규칙엔진 파라미터 로드 (mock_test_eval_settings → 기본값 폴백)
   │
   ├─ ③ 규칙엔진 7-Step 실행 (→ 04-등급-판정.md 상세)
   │   → 입력: evaluations → EvaluationInput[] 변환
   │   → 출력: final_level, FACT 점수, 체크박스 집계 등
   │
   ├─ ④ reports UPSERT (규칙엔진 결과)
   │   → final_level, floor/ceiling/sympathetic/q12/al 판정
   │   → FACT 점수 (score_f, score_a, score_c, score_t, total_score)
   │   → 발음 평균 (avg_accuracy/prosody/fluency)
   │   → rule_engine_status: 'completed'
   │   → report_status: 'processing' (GPT 리포트 대기)
   │
   ├─ ⑤ GPT-4.1 종합 리포트 생성
   │   │
   │   ├─ 프롬프트 조회 (evaluation_prompts.key = 'eval_comprehensive')
   │   │
   │   ├─ v3 question_summaries 구성
   │   │   → 각 문항의 과제충족/피드백분기/핵심교정/발화통계/발음 요약
   │   │
   │   ├─ 변수 치환 (규칙엔진 결과 + FACT + 체크박스 집계 + 문항요약)
   │   │
   │   ├─ GPT-4.1 호출 (JSON 모드)
   │   │   → temperature: 0.4, max_tokens: 12000
   │   │   → 2회 재시도
   │   │
   │   └─ 출력: 9섹션 JSON (coaching_report)
   │
   ├─ ⑥ reports UPDATE (GPT 리포트 결과)
   │   → coaching_report (9섹션 JSON)
   │   → recurring_mistakes
   │   → overall_comments_ko (snapshot.headline)
   │   → training_recommendations
   │   → tutoring_prescription (자동 변환)
   │   → avg_completion_rate
   │   → report_status: 'completed'
   │
   ├─ ⑦ 성장 리포트 생성 (2회차부터)
   │   → 이전 세션과 비교 (FACT 점수, 등급, 유형별 pass_rate)
   │   → GPT-4.1-mini로 성장 분석 (한국어)
   │   → 3회차+ 성장 패턴 4종 감지
   │   → DB 저장: growth_summary, growth_comparison, growth_analysis
   │
   └─ ⑧ 세션 상태 업데이트
       → holistic_status: 'completed'
       → status: 'completed'
```

### GPT 종합 리포트 9섹션

```json
{
  "snapshot": { "headline": "한줄평", "readiness_percent": 75 },
  "fact_analysis": { "F": "분석", "A": "분석", "C": "분석", "T": "분석" },
  "top3_priorities": [
    { "rank": 1, "area": "영역", "why": "이유", "drill_tag": "태그" }
  ],
  "strengths": ["강점1", "강점2"],
  "question_type_map": [
    { "type": "description", "status": "strong", "priority": false }
  ],
  "recurring_patterns": [
    { "pattern": "패턴", "drill_tag": "태그", "example": "예시" }
  ],
  "training_recommendation": { "focus": "집중 영역", "exercises": [] },
  "roadmap": { "current": "IM2", "next": "IM3", "key_actions": [] }
}
```

### tutoring_prescription 자동 변환

GPT 종합 리포트의 `top3_priorities`, `question_type_map`, `recurring_patterns`에서 자동 추출:

```
top3_priorities → priority_weaknesses (rank, area, drill_tag)
recurring_patterns → error_drill_tags (중복 제거)
question_type_map 중 weak/very_weak → weak_types
question_type_map 중 priority=true → training_order
top3_priorities의 drill_tag → must_fix_for_next_grade
```

## 에러 처리 & 재시도

### 각 Stage 공통 패턴

```
try {
  // 메인 로직
} catch (err) {
  // 1. eval_retry_count 조회
  // 2. retryCount < 3 → eval_status: 'pending' (재시도 가능)
  // 3. retryCount ≥ 3 → eval_status: 'failed' (최종 실패)
  // 4. eval_error에 에러 메시지 기록
}
```

### Stage별 재시도 설정

| Stage | 내부 재시도 | 외부 재시도 (eval_retry_count) |
|-------|-----------|---------------------------|
| A | Whisper 3회, Azure 3회 | 최대 3회 |
| B-1 | GPT 2회 | 최대 3회 |
| B-2 | GPT 2회 | 최대 3회 |
| C | GPT 2회 | 최대 3회 (report_retry_count) |

### 부분 실패 허용

- Azure 발음 실패 → GPT 평가는 발음 데이터 없이 진행
- GPT 종합 리포트 실패 → 규칙엔진 결과는 보존 (report_status 분리)
- 성장 리포트 실패 → 전체 프로세스 중단하지 않음

## eval_status 상태 전이

```
pending
   │ Stage A 시작
   ▼
processing
   │
   ├─ 스킵 판정 (1차/2차/3차) ──────────→ skipped
   │
   │ STT + Azure 완료
   ▼
stt_completed
   │ Stage B-1 시작
   ▼
evaluating
   │ GPT-4.1-mini 판정 완료
   ▼
judge_completed
   │ Stage B-2 완료
   ▼
completed ──── (정상 완료)

─── 에러 발생 시 ───
  retry_count < 3 → pending (재시도)
  retry_count ≥ 3 → failed (최종 실패)
```

## 소요 시간 (문항당)

| 단계 | 소요 시간 | 비고 |
|------|----------|------|
| Stage A | ~5초 | Whisper ~3초 + Azure ~2초 |
| Stage B-1 | ~12초 | GPT-4.1-mini (6K 토큰) |
| Stage B-2 | ~8초 | GPT-4.1 (8K 토큰), 무응답 즉시 |
