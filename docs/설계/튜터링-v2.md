## 1. 핵심 철학: 일타 강사

### 일타 강사가 학원 강사와 다른 점

**진단의 정확성** + **처방의 구체성**

| | 학원 강사 (v1) | 일타 강사 (v2) |
|---|----------|----------|
| 진단 | "문법이 약하시네요" | "15문항 중 8개에서 연결어가 and, so 2종뿐이야" |
| 처방 | "연결어를 다양하게 쓰세요" | "however, moreover, on the other hand 3개 외워서, 30초 답변에 3개 넣어" |
| 완료 | "3번 연습했으니 다음으로" | "3회 중 2회에서 4종 이상 나왔으니 다음으로" |
| 톤 | "~를 연습하세요" (시스템) | "이것만 고치면 IH 간다" (인격) |
| 방법 | 약점별 개별 교정 | **틀 기반 학습** — 만능 틀 장착 → 어휘·연결어 업그레이드 |

### v1의 문제점

| 항목 | v1 현실 | v2 목표 |
|------|---------|---------|
| **진단 근거** | coaching_report.top3_priorities (GPT 주관 요약 3줄) | 15문항 × weak_points 빈도·심각도 통계 분석 |
| **처방 생성** | DRILL_TAG 18개 고정 매핑 → 최대 6개 처방 | 병목 스코어링 → 티어별 맞춤 드릴 매칭 |
| **등급 분화** | 3구간(IL_IM, IM_IH, IH_AL) 단순 분기 | **4티어 체계** — 등급 간 결정적 차이 기반 훈련 분화 |
| **프로토콜 선택** | 모든 약점 → 동일 5프로토콜 순차 | 약점 유형 + 티어별 맞춤 프로토콜 조합 |
| **완료 판정** | 3회 반복 = 기계적 완료 | 성공 기준 충족률 기반 (드릴별 측정 가능한 기준) |
| **교수법** | 약점 교정 중심 | **강지완 메소드** — 틀 장착 → 바꿔 끼기 → 채점표 역공학 |
| **증거 인용** | 없음 | "Q3에서 'and then' 3번 반복한 거 봤지?" 학생 발화 직접 인용 |

### 3모듈 역할 유지

| 모듈 | 역할 | 비유 |
|------|------|------|
| 모의고사 v2 개별평가 | 진찰 — 문항별 증거 수집 | 의사의 검사 |
| 모의고사 v2 종합평가 | 진단서 — 등급 판정 + 근거 | 의사의 소견서 |
| **튜터링 v2** | **치료** — 병목 집어서 고침 | **일타 강사의 과외** |

### "다음 한 칸" 원칙

> **지금 등급에서 바로 다음 등급으로 가는 데 필요한 것만 처방한다.**
> AL 목표라도 현재 IM2면 IM3에 필요한 것만 먼저. IH→AL은 IM3→IH를 달성한 후에.

이유: OPIc은 누적 역량 시험이다. 문단 구성(IH)을 못하면서 어휘 다양성(AL)을 훈련하는 것은 무의미하다. ACTFL 기준도 "하위 등급의 기능을 안정적으로 수행"해야 상위 등급을 부여한다.

## 2. 4-Tier 등급 갭 시스템

### 설계 근거

ACTFL Proficiency Guidelines 2024 + 강지완 OPIc AL 강의 22강 분석에서 도출한 **등급 간 결정적 차이**에 기반한다. AM(Advanced Mid 이상)은 OPIc 범위를 넘어서므로 제외한다.

### v2.1 설계 원칙

v2.1에서는 각 Tier를 **교수용 scaffold**와 **ACTFL 판정 기준**으로 분리해서 정의한다.

- **교수용 scaffold**: 학생에게 장착시킬 틀/훈련 장치 (예: Basic Frame, 4줄 만능 틀, Skeleton Paragraph, Extended Paragraph)
- **ACTFL 판정 기준**: 실제 승급 판단의 기준 (예: create with language, strings of sentences, paragraph-length discourse, sustained paragraph in all major time frames)

즉, Skeleton Paragraph는 Tier 3의 매우 강력한 훈련 도구이지만, 그 자체가 곧 등급 기준은 아니다. 실제 승급 판단은 **문단형 connected discourse가 형성되는가**, **major time frames를 어느 수준으로 다루는가**, **과제 기능을 수행하는가**를 기준으로 한다.

### Tier 정의

| Tier | 목표 구간 | 현재 등급 | 교수용 핵심 틀 | ACTFL 승급 핵심 |
|------|-----------|-----------|----------------|----------------|
| **Tier 1** | → IL (Intermediate Low) | NL, NH | Basic Frame | **create with language**: 외운 문장 말고 자기 문장 만들기 |
| **Tier 2** | → IM1~IM2 | IL | 4줄 만능 틀 | **strings of sentences**: 여러 문장을 이어서 말하기 |
| **Tier 3** | → IM3~IH | IM1, IM2 | Skeleton Paragraph | **paragraph-length discourse**: 문단형 connected discourse |
| **Tier 4** | → AL (Advanced Low) | IM3, IH | Extended Paragraph | **sustained paragraph in all major time frames** + complication handling |

### Tier 1: → IL (문장 만들기)

**현재 상태**: 단어 나열, 짧은 구, 암기 표현에 의존
**목표**: 주어+동사+목적어로 완전한 문장을 만들 수 있다

**ACTFL 근거**: Intermediate Low는 "creating with language"가 핵심. Novice와의 결정적 차이는 **외운 표현이 아닌 자기 문장을 만드는 능력**.

**훈련 초점**:
| # | 훈련 항목 | 설명 | 성공 기준 |
|---|-----------|------|-----------|
| 1 | 기본 문장 생성 | SVO 구조로 완전한 문장 만들기 | 30초 답변 내 완전한 문장 3개+ |
| 2 | 기본 거래 수행 | 질문 이해 → 관련 대답 | 과제 충족률 50%+ |
| 3 | 필수 표현 장착 | I like ~, I usually ~, There is ~ | 만능 패턴 5개 사용 |

**교수용 성공 기준**:
- 30초 답변에서 완전한 문장 3개 이상
- Basic Frame 1회 이상 사용
- 질문에 대한 직접 응답 존재

**ACTFL 승급 확인 기준**:
- 암기문 반복이 아니라 주제에 맞게 문장을 새로 조합함
- simple personal topic에서 meaning-bearing sentence를 안정적으로 생성
- 최소한의 task 수행이 가능함

**강지완 메소드 적용**: 이 티어에서는 강지완 강의의 "만능 템플릿"을 단순화한 **Basic Frame**(기본 틀)을 장착한다.
```
Basic Frame:
  I [동사] [목적어].
  I usually [동사] [빈도].
  I like it because [이유].
```

### Tier 2: → IM1~IM2 (일관성 + 발화량)

**현재 상태**: 문장은 만들지만 짧고, 주제 이탈이 잦고, 긴 침묵이 있다
**목표**: 연속 문장 5개+ 말할 수 있고, 주제를 벗어나지 않는다

**ACTFL 근거**: IM1~IM2는 "strings of sentences"가 핵심. IL과의 결정적 차이는 **여러 문장을 이어서 말하는 지속력**과 **주제 유지**.

**훈련 초점**:
| # | 훈련 항목 | 설명 | 성공 기준 |
|---|-----------|------|-----------|
| 1 | 발화량 확보 | 30초 이상 끊기지 않고 말하기 | 40초+ 연속 발화 (WPM 80+) |
| 2 | 주제 유지 | 질문에서 벗어나지 않기 | 관련 없는 문장 1개 이하 |
| 3 | 기본 연결어 | and, so, but, because 자연스러운 사용 | 연결어 3종+ 사용 |
| 4 | 만능 틀 장착 | 강지완식 4줄 틀 (도입-이유-예시-마무리) | 4줄 구조 완성 |

**교수용 성공 기준**:
- 40초 이상 연속 발화
- 4줄 구조 3회 중 2회 이상 형성
- 기본 연결어 3종 이상 사용

**ACTFL 승급 확인 기준**:
- isolated sentence가 아니라 strings of sentences가 형성됨
- topic drift가 제한적임
- hesitation이 있어도 답변 진행이 유지됨

**강지완 메소드 적용**: "4줄 만능 틀" — 어떤 질문이든 4줄로 답할 수 있는 기본 구조.
```
4줄 만능 틀:
  ① 핵심 답변: I [동사] [주제].
  ② 이유/배경: Because [이유]. / The reason is [배경].
  ③ 구체적 예시: For example, [경험/사례].
  ④ 마무리: That's why I [재확인]. / So, [요약].
```

### Tier 3: → IM3~IH (Skeleton Paragraph + 시제)

**현재 상태**: 문장은 이어지지만 문단 구성이 없고, 과거/현재/미래 시제 전환이 불안정
**목표**: Topic Sentence → Supporting ×3 → Concluding의 **Skeleton Paragraph** 구조를 기반으로 문단형 connected discourse를 형성한다

**ACTFL 근거**: IH는 "paragraph-length discourse"가 핵심. IM과의 결정적 차이는 **문장 나열이 아닌 문단 수준 발화**. 강지완: "IM3는 문단이 보이기 시작하는 등급, IH는 안정적으로 문단을 유지하는 등급."

**훈련 초점**:
| # | 훈련 항목 | 설명 | 성공 기준 |
|---|-----------|------|-----------|
| 1 | **Skeleton Paragraph** | 도입-본론×3-마무리 구조 | 3회 중 2회에서 구조 형성 |
| 2 | 연결어 다양성 | however, moreover, on the other hand 등 | 답변당 연결어 4종+ |
| 3 | 시제 시도 | 과거/현재/미래 중 2개+ 사용 | 시제 전환 2회+ (오류 허용) |
| 4 | 마무리 문장 | 답변을 명확히 끝내기 | 마무리 있는 답변 80%+ |
| 5 | 디테일 추가 | 5감, 감정, 빈도 등 구체화 | 본론당 디테일 2개+ |

**교수용 성공 기준**:
- Skeleton Paragraph 구조가 3회 중 2회 이상 관찰됨
- 연결어 4종 이상
- 마무리 문장 포함
- relevant prompt에서 45~60초 이상 connected discourse 유지

**ACTFL 승급 확인 기준**:
- 문장 나열이 아니라 문단형 담화 단위가 보임
- support/detail/closure가 존재함
- 적어도 일부 relevant prompt에서 major time frame 시도가 나타남
- question type 핵심 기능(과거 서술, 비교 변화, 문제 상황 설명 등)이 최소 수행됨

> **주의**: Skeleton Paragraph는 Tier 3의 최적 scaffold이지만, 승급 판정은 "Skeleton 사용 여부"가 아니라 **paragraph-length discourse 형성 여부**로 한다.

**강지완 메소드 적용**: **Skeleton Paragraph + 9대 연결어** — 이것이 IM→IH의 핵심 무기.
```
Skeleton Paragraph:
  ① Topic Sentence: "There are a few things I want to tell you about [주제]."
  ② Support 1: "First of all, [내용]. Moreover, [디테일]."
  ③ Support 2: "On top of that, [내용]. As a result, [결과]."
  ④ Support 3: "Last but not least, [내용]."
  ⑤ Concluding: "That is pretty much everything about [주제]."

9대 연결어 (강지완):
  나열: First of all / On top of that / Last but not least
  대조: However / On the other hand
  결과: As a result / Therefore
  추가: Moreover / In addition
```

**채점표 역공학 (강지완)**: IH 채점표에서 확인하는 것 = 문단 구조의 존재 여부. "문장이 아무리 좋아도 문단이 안 보이면 IM."

### Tier 4: → AL (문단 유지 + 시제 + 돌발 + 어휘)

**현재 상태**: 문단 구조는 있지만 길어지면 붕괴, 어휘가 제한적, 돌발 상황에서 무너짐
**목표**: 문단을 끝까지 유지하고, major time frames를 통제하며, 돌발 상황을 성공적으로 처리한다

**ACTFL 근거**: AL은 "sustained paragraph-length discourse in all major time frames"가 핵심. IH와의 결정적 차이는 **문단의 지속성**, **major time frames 통제**, **돌발 대처 능력**, **어휘 범위 확장**. 강지완: "IH는 문단을 시작할 수 있고, AL은 문단을 끝까지 유지할 수 있다."

**IH→AL의 4대 승급 과제 (v2.1)**:
| 승급 과제 | 설명 | 채점 영향 |
|-----------|------|-----------|
| **문단 지속성** | 후반부까지 문단 구조 유지 | 지속 실패 시 IH 판정 |
| **major time frames 통제** | 과거/현재/미래 중 필요한 time frame을 안정적으로 유지 | control 부족 시 IH 판정 |
| **complication handling** | 문제 상황에서 해결책·대안·후속조치 제시 | task 수행 미흡 시 IH 판정 |
| **lexical breadth 확장** | 반복적 generic wording 감소 | 어휘 폭 제한 시 IH 판정 |

**훈련 초점**:
| # | 훈련 항목 | 설명 | 성공 기준 |
|---|-----------|------|-----------|
| 1 | 시제 정확성 | went/go 혼동 제거, 일관된 시제 유지 | 시제 오류 2개 이하/2분 답변 |
| 2 | 문단 지속 | 2분 전체를 문단 구조로 유지 | 후반부 문단 붕괴 0회 |
| 3 | 어휘 업그레이드 | good→excellent, like→prefer, a lot→significantly | 반복 어휘 비율 15% 이하 |
| 4 | 돌발 대처 | 롤플레이 문제 해결 + 대안 제시 | 해결책+대안+후속조치 3요소 |
| 5 | 사회적 관점 | 찬반 양립 + 근거 제시 (비교변화/사회이슈) | 양측 관점 + 자기 입장 |

**교수용 성공 기준**:
- 90~120초 답변에서 후반부 문단 붕괴 없음
- 시제 오류가 통제 범위 내
- 반복 어휘 비율 감소
- complication prompt에서 해결책+대안+후속 조치 제시

**ACTFL 승급 확인 기준**:
- 2개 이상 major time frames에서 sustained paragraph 수행
- roleplay/problem-solving에서 complication handling 가능
- 비교/변화/관점 과제에서 reason support 존재
- lexical breadth가 확장되며 generic wording 의존이 줄어듦

**강지완 메소드 적용**:
- **어휘 치환법**: 기본 어휘 → 고급 어휘 1:1 치환표 (좋다 → magnificent, excellent, outstanding)
- **시제 고정 연습**: 하나의 에피소드를 과거→현재→미래 3회 반복 (시제만 바꿔 말하기)
- **AL 4대 감점 항목** (강지완): Agreement(주어-동사 일치), Preposition(전치사 오류), Syntax(어순 오류), Breadth of Vocabulary(어휘 폭)
- **돌발 대처 3단 공식**: ① 상황 인식 + 사과 → ② 해결책 제시 → ③ 대안 + 후속 조치

## 3. 아키텍처: 3-Layer 시스템

```
Layer 1: 증거 수집 (모의고사 v2가 이미 완료)
  15문항 × { fulfillment, weak_points[{code,severity,reason,evidence}], transcript, speech_meta }
       │
       ▼
Layer 2: 병목 판별 (튜터링 v2 핵심 — 새로 구현)
  빈도 집계 → 심각도 가중 → 티어 필터 → 병목 스코어링
  출력: 병목 Top 1~3 (순위 + 점수 + 증거 문항번호 + 증거 발화)
       │
       ▼
Layer 3: 드릴 매칭 (하이브리드 — 카탈로그 + GPT 맞춤화)
  티어별 드릴 카탈로그 (뼈대, 하드코딩) + GPT (학생별 맞춤화)
  출력: 처방 1~3개 (타겟스킬 + 훈련방식 + 성공기준 + 증거인용)
```

### 불변 원칙

- ⛔ **V1 참조 금지**: V2 구현 시 V1 코드/설계 일절 참조하지 않음
- ✅ 모의고사 v2 데이터(consults, reports, answers)는 읽기 전용으로 사용
- ✅ V2 전용 파일/테이블 신규 생성

## 4. Layer 2: 병목 판별 알고리즘

### 입력 데이터 (모의고사 v2 평가 결과)

> **인터페이스 계약**: §12.6 `QuestionEvalV2ForTutoring` 참조
> **코드 사전**: §12.2 weak_point 코드 사전 v2 (36개) 참조

```typescript
// 15문항의 v2 개별 평가 데이터
interface QuestionEvalV2 {
  question_number: number;
  question_type: string;           // "description" | "routine" | "comparison" | ...
  fulfillment: FulfillmentStatus;  // "fulfilled" | "partial" | "unfulfilled" | "skipped"

  weak_points: Array<{
```

### Step 1. 현재 티어 결정

```
session_grade + target_grade → 티어 자동 결정

매핑:
  현재 NL/NH  → Tier 1 (→IL)
  현재 IL     → Tier 2 (→IM1~IM2)
  현재 IM1/IM2 → Tier 3 (→IM3~IH)
  현재 IM3/IH → Tier 4 (→AL)

"다음 한 칸" 원칙:
  사용자 목표가 AL이어도, 현재 IM2면 Tier 3(→IM3~IH)부터 시작.
  Tier 3 목표 달성 후 → Tier 4로 진급.
```

### Step 2. 빈도 집계

```
15문항의 weak_points를 모두 펼쳐서 코드별 출현 횟수를 센다.
v2에서는 primary + secondary 모두 집계하되, secondary는 0.5 가중치.

예시 (mt_8099015f, 현재 IM3 → 목표 IH → Tier 3 적용):
  WP_S03 connector_poverty:      8회 (Q1,Q2,Q3,Q5,Q7,Q10,Q12,Q14)
  WP_S06 closure_missing:        3회 (Q5,Q7,Q14)
  WP_A08 timeframe_control_failed: 3회 (Q4,Q9,Q12)
  WP_C04 viewpoint_support_weak: 3회 (Q10,Q13,Q15)
  WP_C01 detail_shallow:         2회 (Q1,Q3)
  WP_T03 comparison_absent:      2회 (Q10,Q13)
  ...
```

### Step 3. 심각도 가중치

```
v2.1: weak_point 자체의 severity 필드를 기본 가중치로 사용한다.
(GPT가 문항 맥락을 보고 직접 판정 — fulfillment 기반 간접 추정보다 정확)

  severity = "severe"    → × 3.0  (해당 기능이 사실상 수행되지 않음)
  severity = "moderate"  → × 1.5  (시도는 있으나 불완전/불안정)
  severity = "mild"      → × 0.5  (약한 흔적이 일부 존재)
  skipped 문항           → 제외

추가 규칙:
  질문당 동일 코드의 최대 기여치는 3.0으로 cap한다.
  severe가 여러 번 중첩되어도 한 문항이 전체 우선순위를
  과도하게 압도하지 않도록 한다.

이유:
  GPT가 transcript를 직접 보고 판정한 severity가 더 정확하다.
  다만 한 번 크게 망한 단일 문항이 전체 우선순위를
  지나치게 끌어올리는 것은 교육적 우선순위 왜곡을 만들 수 있다.
```

### Step 3-1. fulfillment 보정 (v2.1 신규)

```
v2.1에서는 fulfillment를 severity와 별도로 보정 계수로만 사용한다.

  fulfilled   → × 1.0
  partial     → × 1.1
  unfulfilled → × 1.25
  skipped     → 제외

중요:
  unfulfilled × 3.0 방식은 사용하지 않는다.
  severity에서 이미 기능 실패가 반영될 수 있으므로,
  동일 실패를 다시 ×3.0으로 키우면 이중 가중이 된다.
```

### Step 4. 티어 관련도 필터

```
§12.4 티어 관련도 매트릭스의 가중치를 적용한다.

각 weak_point 코드는 현재 티어에서의 가중치(0~1.0)가 정해져 있다.
  가중치 1.0 = 해당 티어의 핵심 병목 (필수)
  가중치 0.5 = 중요한 보조 병목
  가중치 0.2 = 참고 수준
  가중치 0   = 사실상 무관 → 스코어에서 제외

예시 (Tier 3 기준, 가중치 1.0인 코드들):
  WP_S03 connector_poverty, WP_S04 paragraph_absent,
  WP_S08 narrative_structure_weak, WP_S09 coherence_weak,
  WP_A02 tense_limited, WP_A08 timeframe_control_failed,
  WP_C01 detail_shallow

가중치 0인 비관련 약점은 스코어에서 제외하되, 기록은 유지한다.
```

### Step 4-1. 게이트성 코드 보정 (v2.1 신규)

다음 코드는 단순 누적 점수 외에 **승급 게이트**로 별도 관리한다:

- `WP_T01` task_unfulfilled
- `WP_T02` subtask_omission
- `WP_S04` paragraph_absent
- `WP_S05` paragraph_unstable
- `WP_A08` timeframe_control_failed
- `WP_T04` complication_failed
- `WP_T08` solution_absent

이 코드는 빈도가 낮아도 현재 티어의 ceiling을 만드는 약점이면 `gate_flag = true`로 표시한다.

### Step 5. 병목 스코어 계산

```
v2.1 계산식:

bottleneck_score = Σ(
  min(severity_weight, 3.0)
  × tier_relevance
  × fulfillment_multiplier
)

추가 규칙:
  if gate_flag == true:
    final_priority_bonus = +2.0

예시 결과 (mt_8099015f, Tier 3):
  1위: WP_S03 connector_poverty — score 24.0 (8회 출현 × 높은 severity × Tier 3 가중치 1.0)
  2위: WP_S06 closure_missing  — score  7.5
  3위: WP_A08 timeframe_control_failed — score  6.0 + gate_bonus 2.0 = 8.0

출력: BottleneckResult[]
```

**병목 스코어 운영 원칙**:
1. **빈도 높은 습관성 약점**은 누적 점수로 우선순위를 올린다
2. **승급을 막는 게이트성 약점**은 낮은 빈도여도 상위로 당긴다
3. 목표 AL이어도 현재 티어에서 의미 없는 luxury 항목은 뒤로 미룬다
4. micro-grammar는 ceiling 요인이 아닐 경우 상위 병목으로 과대선정하지 않는다

### 출력 구조

```typescript
interface BottleneckResult {
  rank: 1 | 2 | 3;
  drill_code: string;              // "connector_diversity"
  score: number;                   // 28.5
  frequency: number;               // 8 (15문항 중)
  evidence_questions: number[];    // [1,2,3,5,7,10,12,14]
  sample_evidence: string;         // "Q3에서 'and then... and then...' 3회 반복"
  tier_relevance: "essential" | "helpful" | "luxury";
  current_tier: 1 | 2 | 3 | 4;
  // v2.1 추가 필드
  gate_flag: boolean;              // 현재 티어 승급 게이트 여부
  confidence: number;              // 0.0 ~ 1.0
  delta_priority_reason: string;   // "게이트성 코드", "고빈도 반복", "복합 병목"
  evidence_samples: string[];      // 최대 3개
}
```

### 일타 강사의 한줄 진단 (GPT 생성)

병목 결과를 GPT에게 주고, **한줄 진단**을 생성한다:

```
입력: BottleneckResult[] + 현재등급 + 목표등급 + 현재 티어
출력 (Tier 3 예시):
  "15문항 중 8개에서 연결어가 and, so 2종에 한정됐어.
   IH는 단락을 엮어야 하는데, 연결어 빈곤이 문장 나열로 보이게 만들고 있어.
   Skeleton Paragraph 틀 잡고 연결어 9개만 넣으면 IH 간다."

출력 (Tier 4 예시):
  "문단 구조는 잡혔는데, 2분 답변 후반부에서 시제가 흔들려.
   went 쓰다가 go로 돌아가는 게 6문항에서 보여.
   시제 고정 훈련만 하면 AL 문턱 넘는다."
```

## 5. Layer 3: 드릴 카탈로그

### 하이브리드 접근

- **뼈대 (하드코딩)**: 티어별·약점코드별 훈련법 + 성공 기준 + 강지완 메소드 적용
- **GPT (맞춤화)**: 학생의 실제 발화를 인용한 예시, 연습 문장, 격려 메시지

### 카탈로그 구조

```typescript
interface DrillDefinition {
  code: string;                    // "connector_diversity"
  name_ko: string;                 // "연결어 다양성"
  category: DrillCategory;         // "structure" | "accuracy" | "content" | "delivery" | "task"

  // 티어별 적용
  tier: 1 | 2 | 3 | 4;           // 이 드릴이 속한 티어
  tier_relevance: Map<Tier, "essential" | "helpful" | "luxury">;

  // 일타 강사가 학생에게 설명할 때
  what_is_it: string;              // "답변에서 문장을 잇는 연결어가 2종 이하인 상태"
  why_it_matters: string;          // "IH는 단락 수준 발화가 필수. 연결어 빈곤 → 문장 나열로 판정"

  // 강지완 메소드 기반 훈련
  training_method: {
    approach: TrainingApproach;    // "frame_install" | "swap_drill" | "timed_pressure" | "self_correction"
    description: string;           // "Skeleton Paragraph 틀에 9대 연결어를 끼워넣는 연습"
    template?: string;             // 강지완식 만능 틀 (해당 시)
    target_expressions?: string[]; // 외울 표현 리스트
    rounds: number;                // 3 (연습 횟수)
  };

  // 성공 기준 (측정 가능해야 함)
  success_criteria: {
    metric: string;                // "답변 내 고유 연결어 종류 수"
    threshold: string;             // "3회 중 2회에서 4종 이상"
    measurement: "gpt_count" | "keyword_match" | "speech_meta" | "transcript_analysis";
  };
}

type DrillCategory = "structure" | "accuracy" | "content" | "delivery" | "task";
type TrainingApproach = "frame_install" | "swap_drill" | "timed_pressure" | "self_correction" | "pattern_drill";
```

| `timeframe_sustain` | time frame 유지 🆕 | accuracy | swap_drill | 시작한 시제 프레임을 끝까지 유지 (2회 중 2회) | 시제 프레임 고정 말하기 |
| `reason_chain` | 이유 체인 🆕 | content | swap_drill | why-chain 2단계+(이유→근거→예시) | 왜-왜-예시 3단 공식 |

> **🆕 `thought_progression`**: WP_S09(coherence_weak) 전용. `connector_diversity`가 연결어 **종류 수**를 늘리는 드릴이라면, 이 드릴은 문장 간 **논리적 흐름**(thought flow)을 만드는 훈련. 연결어가 4종 이상이어도 논리 관계가 부자연스러우면 S09에 해당.
> - **훈련법**: Skeleton Paragraph 틀 위에 전환 사인포스트(First of all / On top of that / As a result) 배치 → 각 Support 사이 인과·대비·추가 관계 명시적으로 연결
> - **강지완**: "그냥 나열하지 마. First, 다음에 Moreover, 다음에 As a result — 이 순서가 이유가 있어야 해."

> **🆕 `timeframe_sustain`**: WP_A08(timeframe_control_failed) 전용. `time_frame_variety`가 "2개+ 시제를 시도하라"는 드릴이라면, 이 드릴은 "시작한 time frame을 끝까지 유지하라"는 훈련. A02(시제 미시도)와 정반대 — A08은 시도했으나 유지 실패.
> - **훈련법**: 하나의 에피소드를 과거로 시작했으면 끝까지 과거 유지 → 다음 에피소드에서 현재로 전환 → time frame 전환점을 의식적으로 표시
> - **강지완**: "시제 바꿔 끼기의 다음 단계. 바꿨으면 유지해야 해. 중간에 흔들리면 채점관이 '이 사람 control이 안 된다'고 판단해."

> **🆕 `reason_chain`**: WP_C07(reason_support_weak) 전용. `description_depth`가 감각·감정 디테일을 추가하는 드릴이라면, 이 드릴은 **왜(why)**에 대한 논증력을 키우는 훈련. adv_14(비교변화), adv_15(사회이슈)에서 특히 중요.
> - **훈련법**: "왜-왜-예시" 3단 공식 — ① 주장 ② 이유(because...) ③ 근거(for example...) → 순서대로 말하는 연습 3회
> - **강지완**: "채점관이 'why'를 물으면 'because + for example'이 무조건 나와야 해. 이유 없이 주장만 하면 content 점수 안 나와."

**Tier 3 핵심 교수법 (강지완)**:
1. **틀 먼저, 내용 나중에**: Skeleton Paragraph 구조를 먼저 체화한 후, 연결어·디테일을 채움
2. **바꿔 끼기**: 같은 틀에 다른 주제·연결어·시제를 끼워넣는 반복 훈련
3. **채점표 역공학**: "IH 채점관은 문단 구조를 본다" → 구조가 보이면 IH

### Tier 4 드릴 카탈로그 (→AL) — 3대 붕괴 방지 + 어휘 업그레이드

| 코드 | 이름 | 카테고리 | 접근법 | 성공 기준 | 강지완 메소드 |
|------|------|----------|--------|-----------|-------------|
| `tense_accuracy` | 시제 정확성 | accuracy | self_correction | 시제 오류 2개 이하/2분 | 시제 고정 연습 |
| `paragraph_sustain` | 문단 지속 | structure | timed_pressure | 후반부 문단 붕괴 0회 | 2분 Full Response |

| `complication_handling` | 돌발 대처 | task | frame_install | 해결책+대안+후속조치 | 3단 공식 |
| `social_perspective` | 사회적 관점 | content | frame_install | 양측 관점+자기 입장 | 찬반 틀 |
| `comparison_frame` | 비교 틀 | content | swap_drill | 명확한 대비 구조 | 대비 연결어 |

| `solution_proposal` | 해결책 제안 🆕 | task | frame_install | 해결책+대안+후속조치 3요소 모두 포함 | 해결 3단 공식 |

> **🆕 `solution_proposal`**: WP_T08(solution_absent) 전용. `complication_handling`이 "돌발 상황 전체 대처"(상황인식→해결→대안)라면, 이 드릴은 **해결책 제시 자체**에 포커싱. RP12에서 문제는 잘 설명하지만 해결/대안이 빠지는 패턴에 특화.
> - **훈련법**: 문제 상황 제시 → "I think the best solution would be..." 시작 → 해결책 1개 + "If that doesn't work," + 대안 1개 + "After that, I would..." + 후속조치 — 3요소 반복
> - **강지완**: "문제 설명만 하고 끝내면 안 돼. 채점관은 '그래서 어쩌라고?'라고 생각해. 무조건 솔루션을 3개 말해."

**Tier 4 핵심 교수법 (강지완)**:
1. **3대 붕괴 방지**: 시제 붕괴·문단 붕괴·어휘 축소를 각각 집중 훈련
2. **어휘 치환법**: good→excellent, like→prefer, a lot→significantly (1:1 치환표)
3. **AL 4대 감점 인식**: Agreement, Preposition, Syntax, Breadth of Vocabulary

### 드릴 → 훈련 접근법 매핑 원리

약점의 성격에 따라 **가장 효과적인 접근법이 다르다**:

| 약점 성격 | 최적 접근법 | 이유 | 예시 드릴 |
|-----------|-----------|------|-----------|
| **구조 부재** | frame_install (틀 장착) | 틀 자체가 없어서 설치해야 함 | skeleton_paragraph, thought_progression, solution_proposal |
| **표현 부족** | swap_drill (바꿔 끼기) | 틀은 있는데 안에 넣을 표현이 부족 | connector_diversity, reason_chain, timeframe_sustain |
| **오류 습관** | self_correction (자기 교정) | 틀린 부분을 인지하고 고치는 반복 | tense_accuracy, agreement_accuracy |
| **속도/압박** | timed_pressure (시간 압박) | 여유 시 가능하지만 압박 시 무너짐 | paragraph_sustain, speech_volume |
| **패턴 암기** | pattern_drill (패턴 반복) | 특정 표현/공식을 외워야 함 | multi_part_checklist, negotiation_expressions |

**v1과의 차이**: v1은 모든 약점에 EPP→Var→Timed→Repair 순차 적용. v2는 **약점 유형에 맞는 접근법 + 티어에 맞는 난이도**로 조합한다.

## 7. ACTFL 평가 기준 통합

### 등급별 결정적 차이 (v2.1 병목 판별에 활용)

| 전환 | ACTFL 핵심 기준 | 채점관이 보는 것 | v2.1 매핑 |
|------|---------------|----------------|-----------|
| NL/NH→IL | create with language | 외운 표현 vs 자기 문장 | Tier 1: sentence_formation |
| IL→IM | strings of sentences | 문장 연결 + 지속력 | Tier 2: speech_volume, basic_connectors |
| IM→IH | paragraph-length discourse | **문단형 connected discourse의 형성 여부** | Tier 3: skeleton_paragraph + thought_progression |
| IH→AL | sustained paragraph in all major time frames | **문단 지속 + time frame 통제 + complication handling + lexical breadth** | Tier 4: paragraph_sustain, tense_accuracy, solution_proposal, vocabulary_upgrade |

### Tier와 ACTFL의 관계

Tier는 ACTFL 판정을 단순화한 운영 프레임이다.
따라서 Tier별 교수용 틀(Basic Frame, 4줄 만능 틀, Skeleton, Extended Paragraph)은 **훈련 장치**이고,
실제 판정은 **Text Type, Functions, time-frame control, task completion**을 우선 기준으로 삼는다.
즉 "틀을 외웠다"가 아니라 "그 틀이 실제 발화에서 등급 기능으로 구현되었는가"를 본다.

### ACTFL FACT 4축과 티어 매핑

| FACT 축 | 설명 | Tier 1 | Tier 2 | Tier 3 | Tier 4 |
|---------|------|--------|--------|--------|--------|
| **Functions** | 언어 기능 (묘사/서술/비교) | 기본 거래 | 단순 묘사 | 묘사+서술+기본 비교 | 서술+비교+돌발+문제 해결 |
| **Accuracy** | 정확성 (문법/발음) | 의미 전달 | 기본 문법 | 시제 시도 | 시제 통제 + ceiling grammar 관리 |
| **Content** | 내용 (주제 범위/깊이) | 개인 영역 | 일상 주제 | 확장 주제 | 사회적 관점/비교변화 |
| **Text Type** | 텍스트 유형 (단어→문장→문단) | 문장 | 문장 연결 | **문단** | **지속 문단** |

### 무응답/스킵 처리

모의고사 v2에서 이미 3축 무응답 감지를 수행 (`fulfillment: "skipped" | "unfulfilled"`).

튜터링 v2.1에서는:
- `skipped` 문항: 병목 분석에서 제외 (데이터 없음)
- `partial` 문항: fulfillment 보정만 부여 (×1.1)
- `unfulfilled` 문항: severity와 별도로 **경미한 보정만 부여** (×1.25, ~~×3.0 아님~~)
- 전체 `unfulfilled` 비율이 높을 경우 Tier 자동 강등 후보로 보되, **단일 세션 즉시 강등이 아니라 최근 2회 기준**으로 판정한다

```
tier_reassessment_rule:
  if latest_two_sessions.unfulfilled_rate_avg >= 0.60:
    tier_down_candidate = true
```

> 이유: 단일 모의고사 한 번의 상태 불량으로 Tier를 바로 낮추면 학습 연속성이 흔들릴 수 있다.

### 완료 판정과 진급 판정의 분리 (v2.1)

v2.1에서는 **드릴 완료**와 **Tier 진급**을 분리한다.

- **드릴 완료**: 해당 성공 기준을 충족했는가
- **Tier 진급**: 다음 모의고사 또는 누적 성과에서 승급 게이트가 해소되었는가

즉, `connector_diversity` 드릴을 통과했다고 해서 즉시 Tier 3를 졸업하는 것은 아니다.
Tier 진급은 **게이트성 병목 해소 + 최근 성과 안정성**을 함께 본다.

### 3라운드 규칙 (v2.1)

기본값은 유지한다.

```
default_rounds = 3
default_pass_rule = "3회 중 2회 성공"
max_extension_rounds = 5
```

다만 **접근법별 예외**를 허용한다:

| 접근법 | 기본 운영 |
|--------|-----------|
| frame_install | 3라운드 고정이 적합 |
| swap_drill | 3라운드 고정이 적합 |
| self_correction | 짧은 2~4회 반복이 더 적합 |
| timed_pressure | 2~5회 짧은 라운드가 더 적합 |
| pattern_drill | 체크리스트/패턴 확인형으로 2~3회 적합 |

### 5-Screen 구조 (v2.1)

기존 원칙은 유지한다.

```
Brief → Demo → Practice → Check → Recap
```

다만 v2.1에서는 **approach별 micro-variant**를 허용한다:

- **frame_install**: Demo 비중 확대
- **swap_drill**: Practice 중심
- **self_correction**: Check–Retry 루프 강화
- **timed_pressure**: Brief 최소화, Practice 반복 확대

즉, **5-Screen은 유지하되 각 단계의 길이와 무게중심은 접근법별로 달라질 수 있다**.

### Tier 재산정 (v2.1)

v2.1에서는 Tier를 아래처럼 운영한다.

```
tier_assignment       = latest_final_level         ← 시작 Tier
tier_promotion_check  = rolling_last_2_sessions    ← 진급 판단
                      + gate_resolution
```

즉,
- **시작 Tier**는 가장 최근 결과로 잡고

## 12. 선행 과제: 모의고사 v2 인터페이스 계약

> **핵심**: 튜터링 v2(소비자)가 필요로 하는 데이터를 모의고사 v2(공급자)가 정확히 출력해야 한다.
> 모의고사 v2 EF/프롬프트 설계 전에 이 계약을 먼저 확정해야 한다.
> **상태**: ✅ GPT-5.4 OPIc 전문가 자문 완료 → v2 코드 사전 확정
> **태깅 프롬프트**: `docs/설계/weak-point-tagging-prompt.md` (별도 파일)

### 12.1. 현재 목 데이터의 문제점

현재 `eval-v2-result.json` (mt_8099015f)의 실제 출력을 분석한 결과:

| # | 문제 | 현재 상태 | 필요 상태 | 영향 |
|---|------|----------|----------|------|
| 1 | **weak_points 코드 혼재** | 영문 코드(`connector_diversity`)와 한글 서술(`시제 일관성 부족`)이 혼재 | **표준화된 코드만** 출력 | 병목 엔진 빈도 집계 불가 |
| 2 | **severity 필드 부재** | weak_points가 flat list (가중치 없음) | **약점별 심각도** (severe/moderate/mild) | 병목 스코어 = 빈도 × **심각도** × 등급관련도 |
| 3 | **grade_relevance 매핑 없음** | 약점이 어떤 등급에 영향 주는지 정보 없음 | **코드별 티어 관련도** 테이블 | "다음 한 칸"에 맞는 처방 불가 |
| 4 | **speech_meta 불완전** | Q2만 발음 점수 있고, Q3~Q15는 null | **전 문항 발음 평가** | Tier 4 발음/유창성 진단 불가 |
| 5 | **checklist 라벨 부재** | pass/fail boolean만 (항목명 없음) | **항목명 + pass/fail** | 증거 인용 ("시제 일관성 체크 미통과") 불가 |
| 6 | **drills ≈ weak_points 복사** | `recommended_drills`가 `weak_points`와 거의 동일 | **증상 코드 ↔ 드릴 코드 분리** | 증상(뭐가 부족) ≠ 처방(뭘 훈련) |

### 12.6. 실제 데이터 소스 (모의고사 V2 구현 완료 기준) ✅ 현행화

튜터링 V2가 읽는 실제 DB 테이블과 필드:

```
■ mock_test_consults (문항별 소견 — SSOT)
  ├─ session_id, question_number, question_id, question_type
  ├─ target_grade                    ← user_metadata.target_grade (SSOT)
  ├─ fulfillment                     ← "fulfilled"|"partial"|"unfulfilled"|"skipped"
  ├─ task_checklist                  ← [{item, pass, evidence}] JSONB
  ├─ observation                     ← 총평 TEXT
  ├─ directions                      ← string[] JSONB
  ├─ weak_points                     ← [{code, severity, reason, evidence}] JSONB
  │   ├─ code: "WP_S03" 등 (36개 코드 사전)
  │   ├─ severity: "severe"|"moderate"|"mild" (← GPT 판정, 주의: "major"/"minor"도 혼용 가능)
  │   ├─ reason: 병목 진단 표시용
  │   └─ evidence: 학생 발화 인용
  └─ skipped_by_preprocess           ← 무응답 플래그

■ mock_test_reports (종합 리포트)
  ├─ session_id, final_level, target_grade
  ├─ rule_engine_result              ← JSONB (RuleEngineResult 전체)
  │   ├─ fact_scores: {score_f, score_a, score_c, score_t, total_score}

  └─ growth                          ← GPT 성장 분석 JSONB

■ mock_test_answers (발화 메타)
  ├─ session_id, question_number, question_id

  ├─ audio_duration, word_count, wpm ← 발화 통계
  ├─ filler_ratio                    ← 필러 비율
  └─ pronunciation_assessment        ← Azure JSONB {accuracy_score, fluency_score, prosody_score}

  ├─ id, question_english, question_korean
  ├─ question_type, question_type_eng
```

**severity 주의**: consult EF의 GPT 출력에서 `severity`가 "severe"/"major"/"minor"로 혼용될 수 있음. 병목 엔진에서 정규화 필요:
- severe = 3.0, major/moderate = 1.5, minor/mild = 0.5

**v1 인터페이스와의 차이**:

| 필드 | v1 (현재) | v2 (확정) | 변경 이유 |
|------|----------|----------|----------|
| `weak_points` | `string[]` (코드/한글 혼재) | `Array<{code, severity, reason, evidence}>` | 병목 엔진 입력 표준화 |
| `severity` | 없음 | `"severe" \| "moderate" \| "mild"` | 병목 스코어 = 빈도 × **심각도** × 티어관련도 |
| `task_checklist` | `Array<{pass}>` (항목명 없음) | `Array<{item, pass}>` | 증거 인용 가능 |
| `speech_meta` | 대부분 null | 전 문항 필수 + pause_count, filler_ratio 추가 | Delivery 태깅 근거 |
| `recommended_drills` | weak_points 복사 | **제거** | 증상(코드)과 처방(드릴) 분리 |

### 12.7. 병목 알고리즘 입력 구조 v2

v2 코드 사전에 맞게 병목 판별 알고리즘(§4)의 입력도 변경:

```typescript
// 기존 (v1)
weak_points: string[];  // ["connector_diversity", "tense_control"]

// 변경 (v2)
weak_points: Array<{
  code: string;       // "WP_S03"
  severity: string;   // "severe" | "moderate" | "mild"
  reason: string;
  evidence: string;
}>;
```

**심각도 가중치 변경**:
```
// 기존: 문항 fulfillment 기반
unfulfilled 문항에서 나온 약점 → × 3.0
partial 문항에서 나온 약점     → × 1.5
fulfilled 문항에서 나온 약점   → × 0.5

// 변경: weak_point 자체의 severity 사용 (GPT가 문항 맥락을 보고 직접 판정)
severity = "severe"    → × 3.0
severity = "moderate"  → × 1.5
severity = "mild"      → × 0.5
```

**병목 스코어 공식 (변경 없음, 입력만 변경)**:
```
bottleneck_score = Σ(severity_weight × tier_relevance)  // 전 문항 합산
```

### 14.1. 진단 프롬프트 — `tutoring_diagnosis`

**목적**: 병목 분석 결과를 일타 강사 톤의 한줄 진단으로 변환

#### System Prompt (`tutoring_diagnosis`)

```
## C — Context (배경)

당신은 OPIc 1:1 튜터링 시스템의 진단 모듈입니다.
학생의 모의고사 15문항 평가 데이터를 통계 분석한 "병목 결과"를 받아,
일타 강사 톤의 진단문을 작성합니다.

OPIc 등급 체계: NL → NM → NH → IL → IM1 → IM2 → IM3 → IH → AL
4-Tier 체계:
- Tier 1 (NH→IL): 문장 만들기 — 주어+동사+목적어 기본 문장
- Tier 2 (IL→IM): 문장 나열 — 시간순/나열 구조로 여러 문장
- Tier 3 (IM→IH): 문단 만들기 — Skeleton Paragraph + 연결어
- Tier 4 (IH→AL): 담화 완성 — 시제·관점 전환 + 논리적 담화

## O — Objective (목표)

1. 병목 결과(top_bottlenecks)에서 가장 심각한 1~2개를 짚어 핵심 문제를 진단한다.
2. 학생의 실제 발화를 큰따옴표로 인용하여 증거를 제시한다.
3. 현재 티어에서 다음 티어로 가기 위한 핵심 과제를 명확히 전달한다.
4. "이것만 고치면 {목표등급} 간다" 형태의 동기 부여 메시지로 마무리한다.

## S — Style (스타일)

- 강지완 OPIc 강사 스타일: 친근하지만 핵심을 정확히 짚는 톤
- 데이터 기반 진단: "15문항 중 8개에서 ~" 형태의 구체적 수치 인용
- 채점표 역공학: "채점관은 ~를 본다. ~이 안 보이면 {등급} 못 넘어" 형태
- 추상적 평가 금지. 반드시 수치 + 학생 발화 근거를 포함

## T — Tone (톤)

- 반말 (해요체 X, 합니다체 X)
- 격려와 직설 혼합: "잘한 건 인정. 근데 이건 고쳐야 해"
- 단도직입적: 문제 → 근거 → 해결책 순서

톤 예시:
✅ "15문항 중 8개에서 연결어가 and, so 2종에 한정됐어. IH는 단락을 엮어야 하는데, 연결어 빈곤이 문장 나열로 보이게 만들고 있어. Skeleton Paragraph 틀 잡고 연결어 9개만 넣으면 IH 간다."
❌ "연결어를 더 다양하게 사용해보세요."
❌ "전환어의 활용이 제한적으로 관찰되었습니다."

## A — Audience (청중)

OPIc 시험을 준비하는 한국인 영어 학습자.
- OPIc 내부 용어(FACT, WP 코드, checkbox, pass_rate) 절대 노출 금지
- "Skeleton Paragraph", "연결어", "시제" 등 학습자가 이해하는 용어만 사용
- Tier 번호 노출 금지 → "지금 네 단계에서는", "다음 단계로 가려면" 형태

## R — Response (출력 형식)

```

### 14.2. 처방 프롬프트 — `tutoring_prescription`

**목적**: 드릴 카탈로그 뼈대를 학생의 실제 발화에 맞게 맞춤화

#### System Prompt (`tutoring_prescription`)

```
## C — Context (배경)

당신은 OPIc 1:1 튜터링 시스템의 처방 모듈입니다.
진단에서 확인된 병목에 대해, 드릴 카탈로그의 뼈대를 학생의 실제 발화에 맞게 맞춤화합니다.

처방의 핵심 원칙:
- 카탈로그의 성공 기준과 훈련 방식은 변경하지 않는다
- 학생의 실제 발화를 인용하여 "왜 이 훈련이 필요한지" 설명한다
- 연습용 예시 문장은 학생의 실제 주제(질문)에서 만든다

## O — Objective (목표)

1. drill_info(드릴 뼈대)의 설명을 학생 맞춤으로 재작성한다.
2. 학생의 약점 증거(evidence)를 인용하여 처방 이유를 설명한다.
3. 학생이 연습에 바로 사용할 수 있는 예시 문장 3개를 생성한다.
4. 연습 문장은 학생이 시험에서 받았던 질문의 주제를 활용한다.

## S — Style (스타일)

- 강지완 OPIc 강사 스타일
- "네가 Q3에서 'and then' 3번 반복한 거 봤지?" 형태의 증거 인용
- 추상적 설명 금지. 구체적 예시 필수

## T — Tone (톤)

- 반말 (강사→학생)
- 격려 + 구체적 지시: "이거 3번만 해보면 확 달라져"
- 진단과 동일한 톤 유지

톤 예시:
✅ "네가 Q3에서 'I like it because it's good' 이렇게 끝낸 거 있지? 여기에 however, moreover 같은 연결어만 넣어도 완전 달라져."
❌ "연결어를 다양하게 사용하는 연습을 해보세요."

## A — Audience (청중)

진단과 동일. 내부 용어 노출 금지.

## R — Response (출력 형식)

JSON Schema(response_format)에 맞춰 출력.
```

#### User Template (`tutoring_prescription_user`)

```
[처방 대상 약점]
- 약점 코드: {wp_code}
- 약점 설명: {wp_description}
- 심각도: {severity}
- 출현 빈도: {frequency}개 문항에서 발견

[학생 발화 증거]
{evidence_list}

[드릴 카탈로그 정보]
- 드릴 ID: {drill_id}
```

### 14.3. 훈련 평가 프롬프트 — `tutoring_evaluate`

**목적**: 매 연습 라운드의 녹음을 평가하고 성공 기준 충족 여부 판정

#### System Prompt (`tutoring_evaluate`)

```
## C — Context (배경)

당신은 OPIc 1:1 튜터링 시스템의 훈련 평가 모듈입니다.
학생이 연습한 답변(transcript)을 받아, 해당 드릴의 성공 기준 충족 여부를 판정합니다.

평가는 "채점"이 아니라 "코칭"입니다:
- 잘한 부분을 먼저 짚는다
- 부족한 부분은 구체적으로 어떻게 고치면 되는지 알려준다
- 성공 기준에 몇 % 도달했는지 명확히 전달한다

## O — Objective (목표)

1. transcript를 읽고 성공 기준(success_criteria)의 각 항목을 체크한다.
2. 각 항목에 대해 pass/fail과 transcript 근거를 제시한다.
3. 전체 충족률을 계산한다.
4. 다음 라운드를 위한 구체적 개선 포인트를 1~2개 제시한다.
5. 격려 메시지를 포함한다 (특히 이전 라운드 대비 개선된 부분).

## S — Style (스타일)

- 코치 스타일: 문제 발견보다 성장 확인에 집중
- 구체적 피드백: "2번째 문장에서 'however'를 넣은 건 좋았어" 형태
- 비교 피드백: 이전 라운드 대비 변화 언급 (previous_attempts 참조)

## T — Tone (톤)

- 반말 (코치→학생)
- 라운드 1: 격려 중심 ("첫 번째니까 일단 해보는 게 중요해")
- 라운드 2~3: 성장 확인 ("아까보다 확실히 좋아졌어. 근데 여기만 더 잡으면 돼")
- 성공 시: 축하 + 구체적 칭찬 ("오 연결어 4종 나왔어! 이 정도면 시험에서도 된다")
- 미달 시: 격려 + 핵심 1개 ("괜찮아, 다음에 여기만 신경 써봐")

## A — Audience (청중)

진단/처방과 동일. 내부 용어 노출 금지.

## R — Response (출력 형식)

JSON Schema(response_format)에 맞춰 출력.
```

```typescript
function substituteVariables(

  variables: Record<string, string>,

}
```

### 14.5. 프롬프트 운영 규칙 (v2.1 추가)

#### 공통 프롬프트 운영 원칙

- 학생의 실제 transcript/evidence를 벗어난 칭찬이나 지적을 만들지 말 것
- "좋았다"는 표현은 반드시 성공 기준과 연결해서 말할 것
- 템플릿을 강요하지 말고, 자연스러운 발화로 구현되었는지 함께 볼 것
- Before/After 예시는 학생의 실제 오류와 최대한 닮은 최소쌍으로 만들 것
- 진단 톤은 직설적이어도 되지만, 훈련 평가 톤은 더 코치형으로 완화할 것

#### 진단 프롬프트 추가 지시

- 핵심 병목이 왜 현재 Tier 승급을 막는지 ACTFL 기능 차이 관점에서 설명
- "이것만 하면 간다" 식 표현은 가능하되, 과장된 확정 표현은 피할 것
- 게이트성 병목이면 그 이유를 1문장으로 명시

#### 처방 프롬프트 추가 지시

- 드릴 성공 기준은 변경하지 말 것
- 학생의 기존 발화와 새 목표 발화의 차이를 Before/After로 제시할 것
- 처방은 많아도 3개지만, 한 번에 실제 집중할 항목은 1~2개로 요약할 것

#### 훈련 평가 프롬프트 추가 지시

- 평가 목적은 점수 부여가 아니라 다음 시도 성공률을 높이는 것
- 먼저 성공한 부분을 성공 기준과 연결해서 1문장 칭찬
- 그다음 실패한 기준 1개만 가장 구체적으로 교정
- 이전 라운드 대비 좋아진 점이 있으면 반드시 1개 명시
- 다음 재시도용 한 줄 미션을 제시

## 부록. v2.1 운영 원칙 요약

1. **드릴 통과 ≠ 티어 진급** — 드릴 성공은 기술 습득, 티어 진급은 모의고사 재측정으로 확인
2. **Skeleton은 도구이지 등급 기준 자체가 아님** — 승급 판정은 ACTFL 기능 기준으로
3. **severity와 unfulfilled를 이중 가중하지 않음** — fulfillment는 경미한 보정 계수로만 사용
4. **게이트성 약점은 빈도보다 우선할 수 있음** — 등급 ceiling을 만드는 약점에 gate_flag 부여
5. **진단은 직설적, 훈련 평가는 코치형** — 단계별 톤 차등
6. **최근 1회가 아니라 최근 2회 성과로 Tier 안정화** — rolling window로 흔들림 방지
