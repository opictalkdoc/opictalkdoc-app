# 튜터링 설계서

> **문서 목적**: 모의고사 이후 학생을 목표 등급까지 실제로 끌어올리는 AI 튜터링 시스템의 설계 기준을 정의한다.
> **작성 근거**: GPT 오픽 일타강사 전문가 자문 + 소리담 실데이터 시뮬레이션 (2026-03-25)
> **참조 데이터**: `docs/설계/튜터링-시뮬레이션-데이터.md`

---

## 오픽톡닥 적용 결정 사항

> **상태**: 큰 그림 논의 완료. 세부 구현(DB 스키마, Prompt 상세)은 구현 착수 시 상의.
> **최종 업데이트**: 2026-03-25

### 용어 원칙

| 문맥 | 표기 | 예시 |
|------|------|------|
| UI (학생에게 보이는 문구) | **주제** | "집 주제에서는 안정적" |
| 설계/로직 (이 문서 내) | **topic** | "같은 topic에서 반복되는 병목" |
| DB/코드 | **`questions.topic`** | `WHERE topic = '집'` |
| answer_type도 동일 원칙 | UI: **유형**, 설계: **type**, DB: **`question_type_eng`** | |

### 확정된 결정 상세

#### D-1. 튜터링 모듈 성격
- **독립 모듈**: 스크립트/쉐도잉과 완전 별개
- **진입점**: 모의고사 응시 완료 후
- **크레딧**: 튜터링 전용 크레딧. **세션 1회 = 크레딧 1회 소모**, 해당 세션 내 모든 훈련 포함

#### D-2. 튜터링 진입 조건
- **최소 모의고사 3회 완료** 후 튜터링 진입 가능
- **근거**: 소리담 실데이터 시뮬레이션 결과, 1회 데이터로는 반복 병목과 일회성 실수를 구분할 수 없으나, 3회 누적이면 5회와 거의 동일한 진단 정확도 달성 (confidence 0.72 vs 0.78)

#### D-3. 분석 데이터 범위
- **최근 N회 분석**: 3회면 3회, 4회면 4회, 5회면 5회, **6회 이상이면 최근 5회만** 분석
- **재진입 시**: 이전 튜터링에 사용된 세션 데이터는 **제외**. 튜터링 이후 새로 응시한 세션만 대상
- **재진입 조건**: 튜터링 실행 후 **최소 3회 새 모의고사 완료** 필요

#### D-4. 기존 모의고사 평가 데이터 활용
- **세션별 종합 등급(`final_level`)**: 기존 모의고사 평가가 산출한 예측 레벨 → Prompt C 입력으로 활용
- **raw data**: transcript + speech meta → Prompt C가 직접 읽고 독립적으로 패턴 분석
- **Prompt C가 자체 판단하는 것**: weak points(약점) 추출 + 반복 패턴에서 병목 확정 + stable level 해석
- **사용하지 않는 것**: FACT 점수, 체크박스(74개), consults 전체(fulfillment/observation/weak_points/directions) — 모의고사 결과 페이지 전용

> **핵심 원칙**: Prompt C는 `final_level`을 참고 신호로 사용하되, weak_points와 병목은 transcript + speech meta에서 직접 판단한다. 기존 consults의 weak_points는 사용하지 않는다.

#### D-5. Prompt C 입력 데이터

> **원칙**: mock_test_sessions + mock_test_reports + mock_test_answers + questions JOIN만 사용. consults/evaluations 테이블 의존 없음.

```
세션별 (mock_test_sessions + mock_test_reports):
- session_id         -- 세션 식별자 (e.g. "mt_8099015f")
- completed_at       -- 완료 시점
- mode               -- training/test
- final_level        -- 세션별 종합 예측 등급 (참고 신호)
- target_grade       -- 학생 목표 등급

문항별 (mock_test_answers + questions JOIN):
- question_id        -- questions.id (e.g. "HSG_GEN_SEL_DESC_01")
- question_number    -- 문항 번호 (1~15)
- answer_type        -- questions.question_type_eng (e.g. "description", "rp_12", "adv_14")
- topic              -- questions.topic (e.g. "집", "렌터카", "음악") — 한국어
- transcript         -- STT 결과 텍스트
- audio_duration     -- 발화 길이 (초)
- word_count         -- 단어 수
- wpm                -- 분당 단어 수
- filler_word_count  -- 필러 단어 수
- filler_ratio       -- 필러 비율
- long_pause_count   -- 긴 멈춤 횟수
- pronunciation_summary -- 발음 요약 점수 {accuracy, fluency, prosody} (Azure 상세 데이터에서 총점 3개만 추출)
- skipped            -- 무응답 여부
- meta_only          -- 메타 정보만 있는 경우 (음성 없음)
- unfinished_end     -- 시간 내 미완성 답변
```

> **Prompt C 산출물**: weak_points 자체 판단 → 3~5회 반복 패턴 → current stable level + ceiling candidate + next step level + **target gap 해석** + 병목 Top 3 + type/topic mastery

#### D-6. 튜터링 프롬프트 파이프라인

**오픽톡닥 운영 파이프라인 = C → D → QSE → E → [Layer 1] → F(조건부)**

> Prompt A(문항별 절대 평가)와 Prompt B(학생용 코칭 변환)에 해당하는 기능은 기존 모의고사 평가 시스템(eval→consult→report)에서 이미 수행되므로, 튜터링 모듈의 직접 호출 대상이 아니다.

| 단계 | 역할 | 입력 | 산출물 | 실행 방식 |
|------|------|------|--------|----------|
| **Prompt C** | 종합 병목 분석 | final_level + target_grade + transcript + speech meta (3~5회) | stable level, target gap 해석, 병목 Top 3, type/topic mastery | LLM |
| **Prompt D** | 처방 생성 | C 결과 + drill library metadata | focus 3개 + **selection policy** + success/retest 기준 | LLM |
| **QSE** | 질문 선택 | D의 selection policy + question bank | question_pool (Q1/Q2/Q3 후보) | **DB 규칙 기반** |
| **Prompt E** | 드릴 세션 생성 | focus + question_pool + level context | Q1/Q2/Q3 세션 계획 (frame, 예시, pass 기준) | LLM |
| **Layer 1** | 경량 피드백 | transcript + pass_criteria_json | PASS/RETRY + 체크리스트 | **규칙 엔진** |
| **Prompt F** | 상세 피드백 | drill goal + transcript + Layer 1 결과 | praise + fix + retry/pass | LLM (조건부) |

**최종 파이프라인**
```
기존 모의고사 결과 (final_level + target_grade + transcript + speech meta) × 3~5회분
    ↓
Prompt C — weak_points 자체 판단 → 반복 패턴 → stable level + target gap 해석 + 병목 Top 3
    ↓
Prompt D — focus 3개 + selection policy
    ↓
Question Selection Engine — DB에서 실제 질문 후보 검색 (LLM 아님)
    ↓
Prompt E — question pool을 받아 Q1/Q2/Q3 세션 구성
    ↓
Layer 1 — 규칙 기반 경량 피드백 (LLM 아님)
    ↓ (escalation 시에만)
Prompt F — 상세 재발화 피드백
```

#### D-7. 탭 구조

**메인 페이지 (`/tutoring`) — 3탭**

| 탭 ID | 이름 | 역할 |
|--------|------|------|
| `diagnosis` | 진단 | 종합 진단 대시보드 — Current/Next/Final, 병목 Top 3, 유형별·topic별 mastery, 처방 CTA |
| `training` | 훈련 | 드릴 목록 — 이번 처방의 focus별 드릴 카드, 진행 상태, 시작 버튼. 미니 테스트 포함 |
| `history` | 나의 튜터링 | 튜터링 이력 — 회차별 병목 변화, 졸업/active/relapsed 추이, 성장 비교 |

**드릴 세션 (별도 immersive 라우트 `/tutoring/drill`)**
- 모의고사 응시와 동일한 패턴: immersive 레이아웃으로 전환
- Rule → Frame → Speak → Feedback → Retry → Pass

**설계 근거:**
- 진단 탭이 허브 역할: 유형별·topic별 mastery를 섹션으로 배치
- 문항 리뷰는 모의고사 결과 페이지에 이미 존재 → 링크로 연결, 중복 제거
- 모의고사와 동일한 UX 패턴 (메인 탭 + immersive 세션)으로 학생에게 익숙

#### D-8. 드릴 UX 설계

**기본 원칙: "Full Answer First, Slot Rescue Second"**

> 학생은 기본적으로 전체 답변을 한 번에 녹음한다. Frame은 발화 트리거(구조 리마인더)이지 답안 제공 UI가 아니다. slot별 녹음은 구조 붕괴 시에만 보조 장치로 개입한다.

**Frame (말하기 구조)**
- 기본: 전체 구조를 카드형으로 보여주고 → 학생이 한 번에 전체 답변 녹음
- 보조: 첫 시도에서 PASS 핵심 조건 누락 시 → 빠진 slot만 하이라이트 + slot별 연습 모드 제안
- slot rescue 후에는 **반드시 전체 답변 재녹음**

**Feedback (피드백 2단계)**
- **Layer 1 (기본)**: 규칙 기반 경량 피드백 — LLM 호출 없이 transcript + speech meta로 `pass_criteria_json` 충족 여부 판정. PASS/RETRY + 체크리스트 + 빠진 요소 1~2개 + retry 지시 1줄. 목표 응답 시간 1초 내외.
- **Layer 2 (조건부)**: Prompt F 호출 — transcript 분석 + 구체적 교정 + 짧은 model + 주의점 1개

**Layer 1 → Layer 2 escalation 조건**
1. 동일 문항에서 2회 이상 retry 실패
2. foundation issue가 심함 (transcript 극단적으로 짧거나 구조 붕괴)
3. 규칙 충돌 (word count 충분한데 핵심 slot 탐지 안 됨)
4. pronunciation/fluency가 너무 낮아 내용 판정 불안정
5. 학생이 "자세히 보기" 버튼을 직접 누름
6. Q3 독립 수행 실패 후 회고 단계

> **Layer 1 상세 스펙은 12장, Layer 2 = Prompt F 상세는 16장 참조**

**3문항 점진적 자립 구조**

| 문항 | 단계 | Frame 강도 | Feedback | Slot Rescue |
|------|------|-----------|----------|-------------|
| Q1 | 학습 | Frame 전체 노출 + 예시 1개 | 경량 (실패 시 상세) | ✅ 허용 |
| Q2 | 적용 | Frame 접힌 요약형 | 경량 (빠진 slot 힌트만) | 부분만 (빠진 slot 힌트) |
| Q3 | 독립 수행 | 규칙 1줄만 | 최소 피드백 | ❌ 없음 |

> Q1은 배운다. Q2는 붙인다. Q3는 혼자 한다.

**레벨별 Frame 강도**

| 레벨 | Frame 표시 | 한국어 설명 | 예시 | Slot Rescue |
|------|-----------|-----------|------|-------------|
| IL~IM1 | slot 전체 노출 | 표시 | 허용 | 적극 허용 |
| IM2 | slot 전체 노출 | 짧게 | 토글 | 실패 시만 |
| IM3 | slot 제목만 | 없음 | 숨김 | 빠진 slot 힌트만 |
| IH | 접힌 상태 기본 | 없음 | 없음 | 거의 없음 |
| AL | 거의 없음 | 없음 | 없음 | 없음 |

**Frame 카드 구성 (각 slot)**
- slot 이름
- 1줄 영어 frame (예: `In the past, ...`)
- 1줄 한국어 의미 (레벨에 따라 표시/숨김)
- optional 힌트 1개 (과도한 예시 금지)

**드릴 플레이어 화면 레이아웃**
- 상단: 오늘의 목표 1줄 + 문항 진행 (Q1/Q2/Q3) + PASS 기준 보기
- 중단: 질문 원문 + Frame 카드 + 예시 토글
- 하단: 녹음 버튼 → 녹음 후 체크리스트 → Retry/Next
- 실패 시 추가: "빠진 slot 연습하기" + "자세히 보기"
- **처음에는 질문+목표+frame+녹음만 표시**, 결과 후에만 피드백 영역 열림

#### D-9. 미니 재평가 (Embedded Mini Retest)

**핵심 원칙: "학생은 연습 중이라고 느끼고, 시스템은 중간부터 평가 중으로 전환된다"**

> UX는 부드럽게 이어지고, 내부 상태만 바뀐다. "시험"이라는 단어는 사용하지 않는다.

**전체 흐름**
```
드릴 시작 화면
  → Q1 학습형 (frame 전체, retry 허용, slot rescue 허용)
  → Q2 적용형 (frame 축소, 힌트만)
  → Q3 독립 수행 (frame 거의 없음)
  → 전환 화면 ("힌트 없이 확인해볼게요")
  → Mini Retest 2~3문항 (frame 없음, 피드백 없음, retry 없음)
  → 결과 화면 (graduated / improving / hold)
```

**드릴 구간 vs Retest 구간 규칙 차이**

| | 드릴 (Q1~Q3) | Retest |
|---|---|---|
| Frame/힌트 | 허용 (레벨별 차등) | **금지** |
| Retry | 허용 | **금지** |
| 즉시 피드백 | 허용 | **금지** (문항 간 "다음으로" 만) |
| Slot Rescue | 조건부 허용 | **금지** |
| 판정 용도 | 학습 증거 | **졸업 판정 증거** |

**전환 화면 문구 예시**
> "좋아요. 이제 정말 내 것이 됐는지 짧게 확인해볼게요. 이번에는 힌트 없이 2문항만 해봅니다."

**결과 3상태**

| 상태 | 의미 | 학생 메시지 | CTA |
|------|------|-----------|-----|
| **Graduated** | 병목 졸업 | "이 병목은 통과했습니다. 도움 없이도 구조가 나옵니다." | 다음 focus 시작 / 오늘은 여기까지 |
| **Improving** | 개선 중 | "거의 됐어요. 아직 매번 안정적이진 않습니다." | 비슷한 문항 1세트 더 / 나중에 다시 확인 |
| **Hold** | 재훈련 필요 | "아직 도움 없이 하기엔 이릅니다." | 이 병목 다시 연습 / 다른 병목으로 이동 |

**3가지 Retest Mode**

| Mode | 진입 | 문항 수 | 용도 |
|------|------|---------|------|
| **Bottleneck Retest** | 드릴 끝에 embedded | 2문항 | 병목 졸업 판정 (기본형) |
| **Type Retest** | 허브에서 별도 진입 가능 | 3문항 | 유형 안정성 점검 |
| **Topic Retest** | 허브에서 별도 진입 | 4~5문항 | topic 안정성 점검 (여러 type 혼합) |

> **Topic Retest는 병목 졸업의 주된 판정 수단이 아니라, topic mastery와 transfer 안정성을 점검하는 보조 평가다.** 병목 졸업 판정은 Bottleneck Retest(embedded)가 담당한다.

#### D-10. 네비게이션 흐름

**모의고사 → 튜터링 진입 동선**

**Case 1: 모의고사 3회 미만**
- 모의고사 결과 성장 탭에 튜터링 CTA **비활성** 표시
- 안내 문구: "모의고사 3회 완료 후 AI 튜터링을 시작할 수 있어요 (현재 N/3회)"
- 진행 바로 시각적 피드백 (1/3, 2/3)

**Case 2: 모의고사 3회 이상, 튜터링 미사용**
- 성장 탭 + 나의 모의고사 탭에 튜터링 CTA **활성**
- 문구: "3회 이상 데이터가 쌓였어요. AI 튜터링으로 병목을 고쳐보세요"
- CTA 클릭 → 크레딧 확인 → `/tutoring` 진단 탭 이동
- 크레딧 없으면 스토어 안내 (기존 패턴과 동일)

**Case 3: 튜터링 사용 후, 새 모의고사 3회 미만**
- 튜터링 CTA 비활성
- 문구: "다음 튜터링을 위해 모의고사 N회 더 필요해요 (현재 N/3회)"

**Case 4: 네비바에서 직접 `/tutoring` 진입**
- 조건 미충족 시: 진단 탭에 안내 화면 (모의고사 N회 더 필요)
- 조건 충족 + 진단 미생성: 진단 생성 CTA ("진단 시작하기" 버튼 → Prompt C 호출)
- 조건 충족 + 진단 있음: 기존 진단 결과 표시

**진단 생성 시점**
- `/tutoring` 진입 시 자동 생성하지 않음
- 학생이 "진단 시작하기" 버튼을 누를 때 크레딧 소모 + Prompt C 호출
- 진단 완료 후 → 처방 자동 생성 → 훈련 탭에 드릴 목록 노출

## 1. 시스템 철학

### 한 줄 정의

> **기출 question bank를 중심으로, 학생의 현재 안정권 수준을 절대 기준으로 진단하고, 목표 등급까지의 갭을 상대 기준으로 해석한 뒤, 유형·topic·병목을 결합한 맞춤 드릴과 재평가 루프로 실제 OPIc 등급 상승을 만드는 AI 튜터링 시스템**

### 핵심 원칙

1. **병목 제거형 시스템**: 영어 전반을 다루지 않고, 목표 등급을 막는 핵심 병목만 제거한다.
2. **같은 Universe**: 모의고사, 튜터링, 재평가가 같은 question bank 안에서 순환한다.
3. **절대 진단 + 상대 처방 분리**: 현재 수준은 절대 기준으로 판정하고, 처방은 목표 대비 상대 기준으로 설계한다.
4. **Top 3 원칙**: 학생에게는 언제나 핵심 병목 3개만 노출한다.
5. **설명보다 재발화**: 이해보다 실제 말하기 수행 변화가 우선이다.
6. **다음 계단 중심**: 최종 목표가 높아도 실제 훈련은 늘 Next Step Level 기준으로 설계한다.
7. **안정권 우선**: 한두 문항의 고점이 아니라 sustained performance를 더 중요하게 본다.
8. **졸업 기준 명확화**: 학생은 무엇을 하면 통과인지 알아야 하고, 시스템은 무엇을 근거로 병목 졸업을 선언하는지 명확해야 한다.

## 2. 학생이 목표 등급에 도달하지 못하는 6가지 원인

| # | 원인 | 설명 |
|---|------|------|
| 1 | 구조 부재 | 답변 틀이 없어 즉흥적 나열로 끝남 |
| 2 | 유형 수행 부재 | description, routine, comparison, role-play를 같은 방식으로 답함 |
| 3 | 낮은 텍스트 타입 | 상위 등급이 요구하는 connected discourse / paragraph discourse가 안 나옴 |
| 4 | 반복 약점 | 시간 프레임, 연결, 디테일, 문장 안정성이 여러 문항에서 반복적으로 무너짐 |
| 5 | 자동화 실패 | 읽으면 알지만 말할 때 바로 안 나옴 |
| 6 | 시험 운영력 부족 | role-play, 돌발, 회복 멘트, 첫 문장 시작 능력이 약함 |

## 3. 3단 등급 구조

### 3-1. Current Stable Level

지금 안정적으로 재현 가능한 수준. 한두 문항의 고점이 아니라 14문항 전체 패턴과 핵심 유형 수행을 기준으로 결정한다.

### 3-2. Next Step Level

이번 루프에서 현실적으로 도전할 다음 단계. 처방과 드릴 난이도는 여기에 맞춘다.

### 3-3. Final Target Level

학생이 최종적으로 원하는 목표 등급. Final Target Level은 단순한 장기 목표가 아니라, **튜터링 처방과 훈련 우선순위를 결정하는 핵심 입력값**이다.

단, Final Target Level은 Current Stable Level을 직접 판정하는 기준으로 사용하지 않는다. 즉 시스템은 먼저 학생의 현재 안정권을 절대 기준으로 진단한 뒤, Current → Next → Final의 갭을 해석하여 어떤 병목을 먼저 고쳐야 하는지 결정한다.

### 3-4. 목표 등급 사용 원칙

1. Current Stable Level은 절대 기준으로 판정한다.
2. Next Step Level은 이번 루프의 직접 훈련 수준이다.
3. Final Target Level은 병목 우선순위와 장기 훈련 방향을 결정한다.
4. 실제 드릴 난이도와 pass 기준은 Next Step Level 기준으로 설계한다.
5. 동일한 Current Level 학생이라도 Final Target Level이 다르면 focus 우선순위는 달라질 수 있다.

### 3-5. Gap 전략

| Gap | 예시 | 전략 |
|-----|------|------|
| 0.5~1 | IM3→IH | 직접 목표형 |
| 2 | IM2→IH | 브릿지형: 이번 루프 IM3 안정화 |
| 3+ | IM2→AL | 계단형 로드맵: foundation → intermediate floor → advanced entry |

## 4. Current Stable Level 판정 기준

### 4-1. 판정 원칙

1. 안정권은 **반복성**으로 판정한다.
2. **핵심 유형 floor**를 충족해야 상위 레벨을 인정한다.
3. 고급 유형 고점이 있어도 intermediate floor가 약하면 상위 레벨을 확정하지 않는다.
4. 한 문항의 exceptional performance는 ceiling 참고자료일 뿐, stable level의 직접 근거가 아니다.

### 4-2. 문항 집계 단위

- Q1 자기소개는 진단 참고만 하고 최종 stable 판정의 핵심 근거로 쓰지 않는다.
- Q2~Q15의 14문항을 기반으로 집계한다.
- 각 문항은 다음 세 층위로 판정된다:
  - task completion
  - type contract fulfillment
  - level evidence

### 4-3. 안정권 판정 로직

**IL / IM1 안정권**
- 완성 문장 유지가 여러 문항에서 가능
- description / routine에서 최소 구조 유지
- 문장 붕괴가 전체의 과반을 넘지 않음

**IM2 안정권**
- description / routine / basic experience에서 최소 이유·디테일이 반복적으로 나타남
- 14문항 중 중간 난이도 핵심 문항군에서 IM2 수행이 다수 확인됨
- comparison이나 role-play가 약해도 intermediate core가 유지되면 IM2 stable 가능

**IM3 안정권**
- experience 계열에서 시간 흐름이 반복적으로 유지됨
- comparison에서 difference만이 아니라 reason이 일정 수준 반복됨
- 문장 나열을 넘어 connected discourse가 다수 문항에서 확인됨

**IH 안정권**
- 다음 4개 핵심 유형군 중 최소 3개에서 IH급 수행이 안정적으로 확인되어야 함:
  1. description / routine
  2. past experience
  3. comparison
  4. rp_11 / rp_12
- paragraph-like discourse가 한두 번이 아니라 반복적으로 나타나야 함
- comparison과 role-play 중 하나가 계속 붕괴하면 IH stable 판정을 보수적으로 제한

**AL 안정권**
- adv_14 / adv_15에서 고급 수행이 반복적으로 확인되어야 함
- 변화/원인/영향, 이슈/반응/의견 구조가 단발성 문장이 아니라 논리적으로 이어져야 함
- intermediate 및 IH floor가 동시에 무너지지 않아야 함
- advanced 응답이 화려하더라도 core types 안정성이 부족하면 AL stable로 확정하지 않음

### 4-4. Floor / Ceiling 규칙

- **Floor 실패**: lower band 필수 수행이 무너지면 상위 band는 ceiling 후보로만 기록
- **Ceiling 후보**: 상위 수행 흔적이 있으나 지속성 부족 시 `ceiling_candidate_level`로만 저장
- **Stable 판정**: floor 충족 + 핵심 유형 반복 수행 + severe foundation issue가 통제된 경우에만 부여

### 4-5. 저장 필드

```json
{
  "current_stable_level": "IM2",
  "ceiling_candidate_level": "IM3",
  "stable_confidence": 0.82,
  "floor_status": {
    "intermediate_floor_met": true,
    "ih_floor_met": false,
    "al_floor_met": false
  },
  "stability_notes": [
    "description/routine stable",
    "comparison reason inconsistent",
    "rp_12 alternatives weak"
  ]
}
```

## 5. Question Bank 기반 시스템 모델

### 5-1. 3축 분석 프레임

시스템은 언제나 **Type × Topic × Skill** 3축으로 문항을 읽는다.

> **핵심**: 본 설계서에서 Topic은 `questions.topic` 컬럼을 의미하며, 채점의 핵심 축이 아니라 **같은 이야기 세계를 공유하는 질문들을 연결하여 전이 학습, story reuse, topic retest, topic mastery 분석을 가능하게 하는 튜터링 축**으로 사용한다.

#### Type 축 (10개 — DB `questions.question_type_eng`)

- `description` — 묘사
- `routine` — 루틴/절차
- `past_childhood` — 과거경험(어린시절)
- `past_recent` — 과거경험(최근)
- `past_special` — 과거경험(기억에남는일) ※ 115개로 가장 많음
- `comparison` — 비교
- `rp_11` — 롤플레이(질문하기)
- `rp_12` — 롤플레이(문제해결)
- `adv_14` — 고급(변화/비교)
- `adv_15` — 고급(이슈/의견)

#### Topic 축 (DB `questions.topic` — 한국어, 69개)

현재 시스템에서 `topic`은 `questions.topic` 컬럼을 의미한다. 질문의 세부 wording이 아니라, **질문이 속한 주제 세계(topic universe)**를 나타낸다.

예: 집, 국내여행, 병원, 렌터카, 음악, 전화기, 음식점, 은행, 인터넷, 건강, 쇼핑, 재활용, 가전제품, 산업, 모임, 가족/친구, 자유시간, 치과, 헬스클럽, 여행, 호텔, 영화 등 (총 69개)

**Topic의 역할**: 단순한 질문 분류 라벨이 아니라, **같은 이야기 세계를 공유하는 질문들을 묶는 학습 단위**다. 학생은 같은 topic 안에서:
- 다른 유형으로 확장해 답할 수 있고
- 자신의 경험과 표현을 재활용할 수 있으며
- 같은 topic에서 반복되는 병목을 노출할 수 있다

예를 들어 `집` topic이라면:
- description: 집 묘사
- routine: 집에서의 일상
- comparison: 예전 집 vs 지금 집

처럼 같은 topic 안에서 여러 type으로 확장 가능하다.

**튜터링에서 Topic을 사용하는 이유:**

1. **같은 topic 안에서 다른 유형으로 확장** — topic은 익숙하지만 특정 type에서만 무너지는지 확인
2. **학생의 이야기 자산 재활용** — 같은 topic의 경험/표현을 여러 질문에 반복 활용 (OPIc story reuse)
3. **같은 topic에서 반복되는 병목 추적** — topic 세계 안에서 운영력이 약한지 해석
4. **Topic Retest 구성** — 같은 주제 세계 안의 질문 묶음으로 재평가
5. **추천 질문 난이도 조절** — 익숙한 topic에서 성공 경험 → 다른 topic으로 transfer
6. **Topic 기준 취약도 시각화** — 학생이 직관적으로 약점 이해

**Topic은 채점의 핵심 변수는 아니다.** 문항별 절대 평가와 stable level 판정은 `question_type` + `weak_points` + `bottleneck`이 핵심이다. Topic은 추천 질문 구성, 전이 학습, topic retest, recurrence 해석, topic mastery 시각화에 사용한다.

#### Skill 축 (8대 스킬)

1. 구조/답변 틀
2. 시간 프레임
3. 연결/논리 전개
4. 내용 확장/디테일
5. 문장 안정성/정확성
6. 유창성/발화 유지
7. role-play 운영
8. 발음/전달력

### 5-2. Bank 활용 3단계

| 단계 | 용도 |
|------|------|
| 모의고사 조합 | answer type distribution과 frequency를 반영하여 출제 |
| 튜터링 조합 | 같은 type 유사 문항, 같은 topic의 다른 type, 같은 병목 자극 질문 추천 |
| 재평가 조합 | type retest, topic retest, bottleneck retest |

## 6. 병목 Ontology 정규화

> 병목을 **발견 단위**와 **처방 단위**로 분리한다.

### 6-1. 용어 정의

- **Weak Point**: 개별 문항에서 발견된 구체적 실패 신호
- **Bottleneck**: 여러 weak point를 묶은 훈련 가능한 대표 병목
- **Focus**: 이번 주에 실제로 노출할 상위 1~3개의 병목 과제

```
question weak points → normalized bottleneck → weekly focus
```

### 6-2. 병목 카테고리

**1. Foundation Bottleneck**
- sentence incomplete
- structure collapse
- pause overload
- severe intelligibility issue
- answer too short

**2. Type Bottleneck**
- description detail poverty
- routine sequence missing
- experience timeline weak
- comparison reason missing
- rp_11 info-category missing
- rp_12 alternatives missing
- advanced14 cause/impact weak
- advanced15 public reaction / why-it-matters weak

**3. Skill Bottleneck**
- connector poverty
- time-frame instability
- grammar instability
- fluency breakdown
- pronunciation delivery issue

**4. Target-Gap Bottleneck**
- IM2→IM3 connected discourse gap
- IM3→IH type stability gap
- IH→AL analytical depth gap

### 6-3. 대표 병목 1개 원칙

하나의 weak point는 여러 축에 걸칠 수 있지만, 실제 저장 시에는:

- `primary_bottleneck_code`: 1개만 부여
- `secondary_tags`: 보조 태그로 보존

```json
{
  "weak_point_code": "WP_CP02",
  "primary_bottleneck_code": "BT_COMPARISON_REASON",
  "secondary_tags": ["reasoning", "target_gap_im3_ih"]
}
```

### 6-4. 병목 우선순위 규칙

1. **foundation > type > skill > target-gap**
2. 같은 우선순위 내에서는 recurrence가 높은 것을 우선
3. severity가 높아도 단발성이면 우선순위를 다소 낮춤
4. student-facing 노출은 최대 3개
5. internal 저장은 최대 5개까지 허용

## 7. 등급별 튜터링 전략

### IL ~ IM1

- **목표**: 완성 문장 유지
- **집중**: SVO, 기본 묘사, 기본 루틴, 말문 열기
- **운영**: advanced 노출 최소화, 성공 경험 강화

### IM2

- **목표**: 이유·디테일이 붙은 답변
- **집중**: because 1문장, 4~5문장 유지, 기본 past sequencing
- **운영**: explanation depth보다 기본 구조 완성 우선

### IM3

- **목표**: connected discourse 진입
- **집중**: time flow, comparison reason, roleplay 구조화
- **운영**: 문장 개수보다 흐름 일관성에 초점

### IH

- **목표**: 유형별 수행 안정화
- **집중**: comparison, experience, role-play, paragraph 유지
- **운영**: "좋아 보이는 문장"이 아니라 "핵심 유형에서 반복 가능한 수행"

### AL

- **목표**: 분석·반응·영향의 안정적 전개
- **집중**: adv_14/15, 고급 comparison, 설득형 role-play
- **운영**: 과잉 학술체 금지, 말 가능한 고급 수행 유지

## 8. 전체 서비스 흐름

```
모의고사 응시 → 절대 평가 → 종합 진단 → 처방 → 훈련 → 미니 재평가 → 병목 졸업/유지 판정 → 다음 루프
```

### STEP 1. 모의고사 응시 (3~5회 누적)

수집 데이터: transcript, speech_meta, question metadata, answer_type, target_level

### STEP 2. 기존 모의고사 평가 (이미 완료)

세션별 final_level 산출 (기존 eval→consult→report 파이프라인)

### STEP 3. 종합 진단 (Prompt C)

- 입력: final_level(참고) + **target_grade** + transcript + speech meta (3~5회분)
- Prompt C가 **weak_points를 자체 판단** → 반복 패턴 → 병목 정규화
- 산출:
  - current stable level + ceiling candidate
  - next step level
  - **current → next → final gap 해석**
  - 병목 Top 3
  - type/topic mastery

### STEP 4. 처방 (Prompt D)

- **target_grade를 반영하여 focus 우선순위 조정**
- focus 3개 이하
- drill 추천 + 추천 question 묶음
- success criteria / retest criteria

### STEP 5. 훈련 (Prompt E + F)

- 실제 드릴 난이도는 **Next Step Level 기준**으로 설계
- 단, **Final Target Level에 필요한 유형/병목은 우선적으로 배치**
- rule → frame → speak → feedback → speak. 반드시 재발화 포함

### STEP 6. 미니 재평가

embedded bottleneck retest / type retest / topic retest

### STEP 7. 졸업 판정

병목 유지 / 부분 개선 / 졸업 결정

## 9. 추천 로직 명세

### 9-1. 추천 공식

```
추천 훈련 = Current Stable Level + Next Step Level + Final Target Level + 취약 Type + 취약 Topic + 반복 병목 + 최근 재평가 상태
```

### 9-2. 목표 등급을 추천 로직에 반영하는 방식

- Current Stable Level은 학생이 지금 안정적으로 할 수 있는 수준이다.
- Next Step Level은 이번 루프에서 직접 훈련할 현실적 목표다.
- Final Target Level은 **어떤 병목을 먼저 해결해야 목표 등급 달성에 실제로 연결되는지** 결정하는 상위 기준이다.

즉 추천 로직은 아래 두 질문을 동시에 본다:
1. 지금 학생을 가장 빨리 다음 단계로 올리려면 무엇을 먼저 고쳐야 하는가?
2. 그 다음 단계가 **최종 목표 등급 달성에 실제로 연결**되는가?

예:
- Current = IM2, Final Target = IM3 → intermediate core 안정화 중심
- Current = IM2, Final Target = IH → comparison / role-play / connected discourse 반복성 우선
- Current = IH, Final Target = AL → adv_14/15, cause-impact, issue reaction 병목 우선

### 9-3. Priority Score

```
Priority Score = (Foundation Weight × 0.30)
               + (Type Weakness × 0.20)
               + (Target Gap Weight × 0.20)
               + (Skill Severity × 0.15)
               + (Frequency/Recurrence × 0.10)
               + (Recent Retest Failure × 0.05)
```

### 9-3. 운영 보정 규칙

이 수식은 고정 진리가 아니라 **초기 운영 가중치**다. 실제 서비스에서는 매월 보정한다.

- foundation 병목이 있으면 상위 target-gap 병목은 추천 후보에서 한 단계 뒤로 민다.
- 최근 retest에서 이미 pass한 병목은 동일 강도로 재추천하지 않는다.
- 주간 처방에는 서로 지나치게 겹치는 focus를 동시에 넣지 않는다.
- 같은 focus를 3주 이상 반복 추천할 경우, drill 포맷을 바꾸거나 topic transfer를 강제한다.
- **Final Target Level이 높을수록 target-gap 병목의 우선순위를 더 강하게 반영한다.**
- **단, foundation 병목이 심할 경우 target-grade 관련 상위 병목보다 foundation 복구를 먼저 추천한다.**

### 9-5. Topic을 추천 로직에 사용하는 방식

추천 로직에서 `topic`은 다음 네 가지 방식으로 사용한다.

1. **같은 topic, 같은 type** — 가장 직접적인 반복 연습 (예: 집 + comparison)
2. **같은 topic, 다른 type** — topic은 유지한 채 수행 유형만 바꿔 전이 확인 (예: 집 + description → 집 + comparison)
3. **다른 topic, 같은 bottleneck** — 병목이 topic을 넘어 transfer되는지 확인 (예: 집 + comparison reason → 국내여행 + comparison reason)
4. **익숙한 topic → 덜 익숙한 topic** — 추천 난이도를 조절하여 성공 경험 후 확장 (예: 집 → 병원 → 렌터카 순)

### 9-5. 추천 결과 구조

```json
{
  "weekly_focuses": [
    {
      "focus_code": "FOCUS_COMPARE_REASON",
      "label": "comparison에서 차이 뒤에 이유 붙이기",
```

## 11. Role-play 세부 계약

> roleplay를 하나의 큰 틀로 보지 않고, **상황 subclass**를 둔다.

### 11-1. rp_11 subclass

1. **schedule/time inquiry** — 시간, 날짜, 운영 시간
2. **price/payment inquiry** — 가격, 할인, 결제 방식
3. **location/access inquiry** — 위치, 주차, 교통
4. **requirement/policy inquiry** — 준비물, 규칙, 제한
5. **service/feature inquiry** — 서비스 내용, 옵션, 기능

**평가 규칙**
- 질문 개수뿐 아니라 **정보 범주 다양성**을 본다.
- 같은 범주를 표현만 바꿔 반복하면 고득점 불가.
- opening과 closing은 IH 이상에서 가산 요소가 아니라 사실상 안정성 요소다.

### 11-2. rp_12 subclass

1. **reservation/booking problem**
2. **device/service malfunction**
3. **document/access issue**
4. **change/cancel/reschedule request**
5. **compensation/alternative request**

**공통 구조**
- opening/apology
- problem statement
- reason/detail
- alternative 1
- alternative 2
- confirmation / preferred option

**고급 평가 규칙**
- IM3: 대안 1개면 부분 충족
- IH: 대안 2개 이상, 상대가 처리하기 쉽게 명확해야 함
- AL: 대안 비교, 선호 근거, 협상적 조정 가능

## 12. Type Template + Level Modifier + Layer 1 시스템

### 12-1. 설계 원칙

드릴 스펙은 `Type Template`와 `Level Modifier`를 분리하여 관리한다.

- `Type Template`는 각 question_type의 고유 구조, core slots, 기본 pass criteria를 정의한다.
- `Level Modifier`는 각 등급에서 요구되는 text type, 길이, 담화 강도, 힌트 강도, pass strictness를 정의한다.
- `Bottleneck Overlay`는 이번 focus에 맞춘 추가 요구/강조를 정의한다.
- 런타임에서는 `type_template + level_modifier + bottleneck_overlay + question_binding`을 merge하여 최종 drill spec을 생성한다.

```
final_drill_spec = type_template + level_modifier + bottleneck_overlay + question_binding
```

**Merge 우선순위**:
1. type_template 기본 구조
2. level_modifier로 난이도 보정
3. bottleneck_overlay로 focus-specific 강화
4. 특정 drill에서 개별 override

### 12-6. Layer 1 규칙 기반 피드백 엔진

> **한 줄 정의**: Layer 1 = 드릴 중 즉시 PASS/RETRY를 판정하고, 빠진 slot 1~2개를 짚어 재발화를 유도하는 규칙 기반 엔진. LLM 호출 없음.

#### Layer 1 설계 원칙

1. **빠름**: 목표 응답 시간 1초 내외. LLM 호출 없음. deterministic rule evaluation.
2. **설명 가능**: 왜 RETRY인지 바로 보여줄 수 있고, 내부 판정 trace가 남아야 함.
3. **보수적**: 확실히 잡히는 것만 PASS. 애매하면 RETRY 또는 Layer 2 escalation. "틀릴 수도 있는 PASS"보다 "보수적 RETRY"가 낫다.
4. **구조 중심**: semantic beauty를 보지 않고, 구조적 slot 충족 여부를 본다.
5. **타입별**: 공통 로직 + type-specific rules.

#### Layer 1 판정 흐름

```
Step 0. 전처리
  - transcript normalize (lowercase, punctuation)
  - question_type별 rule set 로드
  - drill별 pass_criteria_json 로드

Step 1. 기본 유효성 검사
  - transcript 없음 → 즉시 RETRY
  - word_count < min * 0.4 → 즉시 RETRY
  - skipped / meta_only → 즉시 RETRY
  - STT 품질 불안정 → escalate_l2

Step 2. 공통 구조 체크
  - 최소 길이 / 발화 시간
  - filler 과다 여부 (soft warning)
  - long pause 과다 여부 (soft warning)
  - unfinished_end 참고

Step 3. Type-specific slot 체크
  - pass_criteria의 required_flags 각각에 대해 marker 매칭
  - 예: comparison → past/present/difference/reason

Step 4. PASS / RETRY / ESCALATE 결정
  - required_flags 전부 충족 + hard meta gate 통과 → PASS
  - 일부 미충족 → RETRY
  - 해석 불가 / 반복 실패 → ESCALATE_L2

Step 5. 피드백 문구 생성 (template 기반, LLM 없음)
  - praise 1 + missing point 1~2 + retry instruction 1
```

#### Layer 1 marker 예시

| 카테고리 | markers |
|---------|---------|
| **past** | `in the past`, `before`, `when i was younger`, `back then`, `used to` |
| **present** | `these days`, `now`, `nowadays`, `today`, `currently` |
| **difference** | `different`, `the biggest difference`, `more than`, `less than`, `compared to`, `unlike` |
| **reason** | `because`, `since`, `that's why`, `the reason is`, `so now`, `due to` |
| **sequence** | `first`, `then`, `next`, `after that`, `finally`, `later` |
| **alternative** | `instead`, `another option`, `alternatively`, `or i can`, `could you`, `maybe i can` |
| **problem** | `there is a problem`, `it doesn't work`, `i can't`, `something is wrong`, `was canceled` |

## 13. 드릴 시스템 설계

### 13-1. 드릴의 본질

**이해가 아니라 수행 자동화.** 학생이 규칙을 안다고 끝이 아니라, 같은 질문에 대해 더 나은 답변을 실제로 다시 말할 수 있어야 한다.

### 13-2. 훈련 사이클

```
Rule → Frame → Speak → Feedback(Layer 1) → Speak → Pass/Retry/Escalate(Layer 2)
```

### 13-3. 한 세션 = 3문항

| 문항 | 목적 |
|------|------|
| Q1 | 구조 학습 + 즉시 재말하기 |
| Q2 | 같은 병목, 다른 문항 적용 |
| Q3 | 변형 문항에서 최소 피드백으로 독립 수행 확인 |

### 13-4. 절대 금지 운영

- ❌ 피드백만 주고 종료
- ❌ AL 모범답안 보여주기
- ❌ 병목 5개 이상 한 번에 교정
- ❌ 새 질문만 계속 던지기
- ❌ pass 기준 없이 무한 재시도 시키기

## 14. PASS / Retry / 졸업 규칙

### 14-1. 드릴 PASS 규칙

PASS는 사전에 정의된 **목표 체크리스트 충족**으로 판정한다.

예: `comparison_core_01`
- ✅ 과거 언급
- ✅ 현재 언급
- ✅ 핵심 차이 진술
- ✅ 이유 1문장

위 4개 중 **4개 충족 시 pass**, 3개 이하이면 retry.

### 14-2. 병목 졸업 규칙

병목 하나는 아래 **3단계를 통과해야 졸업**된다.

**Stage A. Drill Pass**
- 동일 focus drill 2회 pass

**Stage B. Transfer Pass**
- 같은 병목을 다른 질문 또는 다른 topic에서 1회 이상 pass

**Stage C. Mini Retest Pass**
- 미니 재평가에서 해당 병목 관련 조건 충족

```
병목 졸업 = Drill Pass 2회 + Transfer Pass 1회 + Mini Retest Pass 1회
```

### 14-3. 졸업 상태

| 상태 | 의미 |
|------|------|
| active | 현재 핵심 병목 |
| improving | 개선 중이나 졸업 전 |
| graduated | 해당 루프에서 졸업 |
| relapsed | 졸업 후 재발 |

### 14-4. relapsed 규칙

- 졸업 후 다음 2회 재평가 중 동일 병목이 다시 high severity로 나타나면 relapsed
- relapsed는 무조건 Top 1 focus로 올리지 않고, recurrence와 severity를 재계산한다.

## 15. 5개 엔진 아키텍처

### A. Assessment Engine (기존 모의고사)

- 문항별 절대 평가 (기존 eval→consult→report 파이프라인)
- 세션별 final_level 산출
- 튜터링에 raw data(transcript + speech meta) 제공

### B. Diagnosis Engine (Prompt C)

- current stable / ceiling candidate / next step 산출
- weak point 자체 판단 → bottleneck 정규화
- target gap 해석
- type/topic mastery 산출

### C. Prescription Engine (Prompt D)

- 우선순위 스코어링
- weekly focus 3개 도출
- selection policy 생성 (질문 검색 조건)

### D. Question Selection Engine (QSE — DB 규칙 기반)

- Prompt D의 selection policy를 받아 question bank에서 실제 후보 검색
- topic/type 필터링, recent history 제외, candidate 랭킹
- Prompt E에 question_pool 제공
- **LLM이 아닌 DB 질의 + 규칙 엔진**

### E. Practice & Reassessment Engine (Prompt E + Layer 1 + Prompt F)

- Prompt E: question_pool을 받아 Q1/Q2/Q3 세션 구성
- Layer 1: 규칙 기반 경량 피드백 (PASS/RETRY/ESCALATE)
- Prompt F: Layer 2 상세 피드백 (escalation 시에만 호출)
- speak-feedback-speak 루프
- drill pass 판단
- mini retest 및 졸업/재발 판정

## 16. 프롬프트 아키텍처

### 16-1. 설계 원칙

1. 절대 평가와 목표 기반 처방 분리
2. 내부 출력과 학생용 출력 분리
3. Type Template + Level Contract 기반 판단
4. JSON 고정 출력 (설명문/마크다운/코드펜스 금지)
5. versioned prompt 운영
6. 모든 프롬프트는 CO-STAR 형식 (Context-Objective-Style-Tone-Audience-Response)

### 16-2. 오픽톡닥 운영 파이프라인 (실사용 기준)

> Prompt A(문항별 절대 평가), Prompt B(학생용 코칭 변환)에 해당하는 기능은 기존 모의고사 평가 시스템(eval→consult→report)에서 이미 수행되므로, 튜터링 모듈의 직접 호출 대상이 아니다.

```
Prompt C (종합 병목 분석)
    ↓
Prompt D (처방 생성 + selection policy)
    ↓
Question Selection Engine (DB 규칙 기반, LLM 아님)
    ↓
Prompt E (드릴 세션 생성)
    ↓
Layer 1 (규칙 기반 경량 피드백, LLM 아님)
    ↓ (escalation 시에만)
Prompt F (상세 재발화 피드백)
```

| 단계 | 역할 | 입력 | 산출물 | 실행 방식 |
|------|------|------|--------|----------|
| **Prompt C** | 종합 병목 분석 | final_level + target_grade + transcript + speech meta (3~5회) | stable level, target gap 해석, 병목 Top 3, type/topic mastery | LLM |
| **Prompt D** | 처방 생성 | C 결과 + drill library metadata | focus 3개 + selection policy + success/retest 기준 | LLM |
| **QSE** | 질문 선택 | D의 selection policy + question bank | question_pool (Q1/Q2/Q3 후보) | DB 질의 |
| **Prompt E** | 드릴 세션 생성 | focus + question_pool + level context | Q1/Q2/Q3 세션 계획 (frame, 예시, pass 기준) | LLM |
| **Layer 1** | 경량 피드백 | transcript + pass_criteria_json | PASS/RETRY + 체크리스트 | 규칙 엔진 |
| **Prompt F** | 상세 피드백 | drill goal + transcript + Layer 1 결과 | praise + fix + retry/pass | LLM (조건부) |

### 16-3. Question Selection Engine (QSE)

> Prompt E는 question bank 전체를 직접 탐색하지 않는다. topic 내 다른 질문, 다른 topic의 질문, transfer용 질문 선택은 **Question Selection Engine**이 담당한다.

**역할**: DB의 `questions` metadata와 사용자 이력을 기반으로 실제 question candidate pool을 생성하는 **retrieval layer**. Prompt E는 이 candidate pool을 입력으로 받아 세션을 구성하는 **instructional generation layer**.

**입력** (Prompt D가 생성):
```json
{
  "focus_code": "BT_COMPARISON_REASON",
  "question_type": "comparison",
  "primary_topic": "집",
  "topic_mode": "same_then_transfer",
  "q1_rule": "same_type_same_topic",
  "q2_rule": "same_type_same_or_adjacent_topic",
  "q3_rule": "same_type_transfer_topic",
  "exclude_recent_question_ids": ["HSG_GEN_SEL_CMP_05"],
  "candidate_limit": 12
}
```

**질의 규칙**:
- 같은 topic, 같은 type: `WHERE question_type_eng = :type AND topic = :topic AND id != :exclude`
- 다른 topic, 같은 type: `WHERE question_type_eng = :type AND topic != :topic`
- 같은 topic, 다른 type: `WHERE topic = :topic AND question_type_eng != :type`
- 인접 topic: `topic_adjacency_map` 활용 (향후 확장)

**출력**:
```json
{
  "q1_candidates": [{"question_id": "HSG_GEN_SEL_CMP_05", "topic": "집", "question_type": "comparison", "question_english": "..."}],
  "q2_candidates": [{"question_id": "HSP_GEN_COM_CMP_04", "topic": "병원", ...}],
  "q3_candidates": [{"question_id": "DTR_GEN_SEL_CMP_03", "topic": "국내여행", ...}]
}
```

**향후 확장 — topic_adjacency_map** (구현 시 결정):
```json
{
  "집": ["가구", "가전제품", "집에서 휴가"],
  "병원": ["병원예약", "약국", "치과"],
  "국내여행": ["여행", "호텔", "여행(공항)"]
}
```

```
- 출력은 반드시 JSON만 반환하라.
```

#### Output Schema

```json
{
  "praise_one": "",
  "fix_one_or_two": ["", ""],
  "correction_examples": [""],
  "retry_instruction": "",
  "target_connection_hint": "",
  "pass_or_retry": "retry",
  "confidence": 0.85
}
```

## 17. 화면 체계

### 17-1. 종합 진단 대시보드 (진단 탭)

- Current / Next / Final 카드
- 병목 Top 3
- 유형별 mastery map
- topic별 취약도
- 처방 CTA

### 17-2. 훈련 허브 (훈련 탭)

- 이번 처방의 focus별 드릴 카드
- 진행 상태
- 미니 테스트 진입

### 17-3. 드릴 플레이어 (immersive)

- 오늘의 목표 1개
- slot 기반 frame
- 녹음 → 즉시 피드백 → PASS 체크리스트 → 재도전
- embedded mini retest

### 17-4. 나의 튜터링 (이력 탭)

- 회차별 병목 변화
- graduated / active / relapsed 추이
- 성장 비교

### 17-5. Topic 기준 화면의 목적

Topic 관련 화면은 단순 주제 목록이 아니라, 학생이 **어떤 이야기 세계에서 강하고 약한지**를 보여주는 학습 지도 역할을 한다.

예를 들어 학생은 다음과 같이 이해할 수 있어야 한다:
- `집` topic에서는 안정적으로 말할 수 있음
- `병원` topic에서는 과거 경험 답변이 자주 무너짐
- `렌터카` topic에서는 role-play 운영이 약함

즉 topic UI의 목적은 질문을 분류하는 것이 아니라, **학생의 topic별 mastery와 transfer 상태를 시각화하는 것**이다.

## 19. UX 원칙

1. 내부 코드와 pass rate는 숨기고 코칭 언어로 번역한다.
2. 학생에게는 늘 행동형 피드백을 준다.
3. IM2 학생에게 IH/AL 모범답안을 강요하지 않는다.
4. 같은 이야기를 여러 질문에 재활용하게 설계한다.
5. low level 학생에게 advanced overload를 주지 않는다.
6. high level 학생에게는 쉬운 유형도 깊이를 바꿔 훈련한다.
7. 학생은 지금 무엇을 고치고 있는지, 언제 통과인지 분명히 알아야 한다.

## 20. 운영 원칙

1. 질문 universe를 모의고사와 튜터링에서 통일한다.
2. Current / Next / Final을 분리한다.
3. 레벨 계약서를 type별로 운영한다.
4. 병목은 내부 5개, 외부 3개 체계로 운영한다.
5. foundation 병목을 먼저 해결한다.
6. 같은 질문을 다른 목표로 반복 훈련시킨다.
7. 설명보다 재발화와 재평가를 중시한다.
8. stable level은 보수적으로, coaching은 구체적으로 운영한다.
9. 병목 졸업과 재발 규칙을 반드시 기록한다.
10. monthly calibration으로 추천 가중치를 조정한다.

## 21. MVP 우선순위

> D-8(드릴 UX), D-9(embedded mini retest)의 상세 UX는 **목표 상태 기준 설계**이다. MVP Phase 1에서는 Prompt C~D 중심의 진단/처방 로직을 우선 구현하고, Prompt E~F 기반 실시간 드릴 경험은 Phase 2에서 확장한다.

### Phase 1 — 진단 + 처방 + 기본 드릴

- Current / Next / Final 구조
- current stable / ceiling candidate 로직 (Prompt C)
- **target_grade 입력 및 current → next → final gap 해석**
- 병목 Top 3 도출 + weak point → bottleneck 정규화
- 처방 생성 (Prompt D) — **target_grade 반영 focus 우선순위** + 드릴/질문 추천
- Type Template 4종 + Level Modifier 6종 (structure / timeline / comparison / rp_12 우선)
- **Layer 1 규칙 기반 피드백 엔진** (위 4종 type에 대해 우선 적용)
- 3문항 미니 테스트
- PASS / graduation 기본 로직
- 진단/훈련/나의 튜터링 3탭 기본 UI

### Phase 2 — 실시간 드릴 + 성장

- Prompt E (드릴 생성) + Prompt F (재발화 피드백)
- D-8 Full Answer First + Slot Rescue UX 구현
- D-9 embedded mini retest 구현
- topic mastery
- rp_11 전용 UI
- adv_14/15 전용 훈련
- 성장 리포트
- relapsed 로직

### Phase 3

- 완전 자동 개인화
- 장기 성장 예측
- 추천 가중치 자동 보정
- 병목 전이 분석

## 23. 최종 요약

이 튜터링 시스템은 단순히 좋은 피드백을 주는 시스템이 아니다.

이 시스템은:
- **안정권 수준을 보수적으로 판정**하고
- **반복 약점을 대표 병목으로 정규화**하며
- **같은 question bank 안에서 드릴과 재평가를 돌리고**
- **명시적 pass 및 graduation 규칙으로 성장 여부를 판정**하는

**실행형 OPIc 튜터링 운영체계**다.

핵심은 세 가지다:

1. **Stable Level은 고점이 아니라 반복성으로 판정한다.**
2. **병목은 분석 단위가 아니라 훈련 가능한 단위로 정규화해야 한다.**
3. **학습은 설명으로 끝나지 않고, pass와 graduation으로 닫혀야 한다.**
