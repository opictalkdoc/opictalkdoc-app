## 1. 핵심 원칙: 의사 → 처방전 → 물리치료

### 전문가 피드백 핵심 결론

> "분석은 훌륭하다. 학습자가 바로 행동으로 옮길 수 있게 만들어라."

현재 시스템은 **"무엇이 잘못됐는지"**까지는 최고 수준이지만, **"그래서 지금 당장 뭘 하라"**가 부족하다. 단, 이 "실행"은 평가 모듈이 아닌 **튜터링 모듈**의 역할이다.

### 3개 모듈 역할 정의

| 모듈 | 핵심 질문 | 역할 | 비유 | 사용자 체감 |
|------|----------|------|------|-----------|
| **개별 평가** (Stage B) | "이 답변에서 뭐가 문제야?" | 진단 + 교정 | 의사의 진찰 | "아, 이게 틀렸구나" |
| **종합 평가** (Stage C) | "전체적으로 어떤 상태야?" | 진단서 + 처방전 | 의사의 소견서 | "전체적으로 이 수준이고, 이걸 해야 하는구나" |
| **튜터링** | "그래서 지금 당장 연습하자" | 치료 + 재활 + 추적 | 물리치료사 | "지금 이 연습을 하면 실력이 올라가는구나" |

### 경계 원칙

```
개별 평가 ─── 진단은 하되, 연습은 시키지 않음
종합 평가 ─── 전략과 방향을 알려주되, 실행은 시키지 않음
튜터링   ─── 실제로 연습시키고, 측정하고, 추적함
```

**금지 사항:**
- ❌ 개별 평가에서 "오늘 3회 녹음하세요" 같은 연습 과제 제공
- ❌ 종합 평가에서 실제 녹음/연습 UI 제공
- ❌ 튜터링에서 등급 판정이나 체크박스 평가 재실행

**연결 원칙:**
- ✅ 개별 평가의 교정 데이터 → 종합 평가가 집계하여 패턴 발견
- ✅ 종합 평가의 처방 → 튜터링 시작 시 자동으로 약점 과제 생성
- ✅ 튜터링의 성장 데이터 → 다음 모의고사 종합 평가에서 변화 추적

## 3. 개별 평가 (Stage B) — 확정 설계

> **상태: ✅ 논의 완료, 구현 대기**
> **결정일: 2026-03-08**
> **핵심 철학**: 품질에 타협 없음. 6개 Layer 전부 제공, 각 Layer가 세계 최고 품질.

### 3-0. 개별 평가의 존재 이유

- **경쟁사에 없는 기능**: 문항별 개별 피드백을 제공하는 OPIc 모의고사 서비스는 없음
- OPIc 15문항, 10가지 question_type → 각 답변에 대해 "당신은 이렇게 답했고, 이게 문제이고, 이렇게 말했어야 한다"
- 종합 평가는 전체 패턴만 볼 수 있음. **개별 문항의 구체적 교정은 여기서만 가능**
- 응시자 관점: "15문항 중 어디서 잘했고 어디서 망했는지"를 알아야 다음에 고칠 수 있음

### 3-1. 6-Layer 구조 (확정)

| Layer | 이름 | 목적 | 표시 | 읽는 시간 |
|-------|------|------|------|----------|
| **1** | 한줄 인사이트 | 첫인상, 3초 파악 | 항상 펼침 | 3초 |
| **2** | 핵심 교정 (1~3개) | 구체적으로 뭐가 틀렸고 어떻게 고치나 | 항상 펼침 | 30초 |
| **3** | 답변 개선 | 내 답변 vs 교정 vs 목표 등급 | 항상 펼침 | 1분 |
| **4** | 영역별 점수 + 구조 + 등급 추정 | 객관적 지표 + 이 답변의 수준 | 항상 펼침 | 30초 |
| **5** | 심층 분석 | L1 원인 + 목표 답변 + 유형 공략법 | 항상 펼침 | 2분 |
| **6** | 말하기 습관 | 시간 + 속도 + 필러 + 발음 | 항상 펼침 | 30초 |

### 3-2. Layer 1 — 한줄 인사이트 (확정: 하이브리드)

**형태**: 칭찬 + 핵심 약점 + 실제 교정 예시를 한 문장에

```
"어휘는 좋았지만 구조가 약해요 — 'I like go to...' 대신
'I usually go to... after work'처럼 말하면 훨씬 자연스러워요."
```

**프롬프트 지시:**
```
one_line_insight 작성 규칙:
- 형식: [칭찬 1개] + [핵심 약점 1개] + [실제 교정 예시 1개]를 한 문장으로
- 교정 예시는 key_corrections 중 impact가 가장 높은 것의 original → better를 인용
- 추상적 코멘트("구조적 답변 연습이 필요해요") 금지. 반드시 구체적 표현 포함
- 톤: target_level에 맞게 (NH~IM2: 격려, IM3~IH: 균형, AL: 도전적)
```

**JSON 스키마:**
```jsonc
"one_line_insight": "어휘는 좋았지만 구조가 약해요 — 'I like go to...' 대신 'I usually go to... after work'처럼 말하면 훨씬 자연스러워요."
```

### 3-3. Layer 2 — 핵심 교정 (확정: 1~3개 유동 + 최대 정보)

**선별 기준 (의미전달 우선):**
1. 의미전달(meaning) 오류 중 등급 영향 높은 것 → 무조건 포함
2. 의미전달(meaning) 오류 중 등급 영향 중간 → 포함
3. 형태(form) 오류 중 등급 영향 높은 것 → 남은 자리에 포함
4. 형태(form) 오류 중 등급 영향 중/하 → 제외

**개수**: 1~3개 유동. 오류가 적으면 1개만. "교정할 것이 1개뿐이에요!"가 강력한 칭찬.

**교정 1건당 포함 정보:**

```jsonc
{
  "sentence_index": 0,
  "error_type": "meaning",              // "meaning" | "form"
  "original": "I like go to internet cafe",
  "corrected": "I like going to an internet cafe",
  "better": "I usually go to an internet cafe near my house after work",
  "why": "go to → going to: 동명사가 필요한 자리예요.",
  "l1_cause": "한국어에서 '가는 것을 좋아해'처럼 동사 원형을 쓰는 습관이 영어에 옮겨진 거예요.",
  "impact": "high",
  "impact_detail": "동명사/to부정사 오류는 문장 기본 구조를 흔들어 등급 안정성을 떨어뜨려요."
}
```

**프롬프트 지시:**
```
key_corrections 작성 규칙:
- 개수: 의미전달(meaning) 오류가 없고 형태(form) 오류만 사소하면 1개도 가능. 최대 3개.
- 각 교정에 반드시 포함: error_type, original, corrected, better, why, l1_cause, impact, impact_detail
- error_type 분류:
  * "meaning": 시제 혼동, 어순 오류, 어휘 오류, 주어-동사 불일치 등 의미가 달라지는 오류
  * "form": 관사 누락, 전치사, 3인칭 -s 등 의미 전달은 가능하지만 형태가 틀린 오류
- l1_cause: "한국어에서 ~하니까 이런 실수가 나와요" 형태로 한 문장
- better: {target_level} 수준으로 재구성. 구체적 수치/빈도 필수 포함
- impact_detail: 이 오류가 OPIc 등급에 구체적으로 어떤 영향을 주는지
```

### 3-4. Layer 3 — 답변 개선 (확정: 3단계 + 핵심 표현 하이라이트)

**구조:**
1. 학생 답변 요약
2. 교정 답변 (corrected) — 문법만 수정
3. 더 나은 답변 (better) — 목표 등급 수준
4. 무엇이 달라졌는지 3가지
5. **핵심 표현 3~5개** — better에서 배울 표현 + 뜻 + 사용 상황

**JSON 스키마:**
```jsonc
"answer_improvement": {
  "student_summary": "인터넷 카페에 자주 간다는 내용. 문장 3개 나열.",
  "corrected_version": "I like going to an internet cafe. I play games there. It's really fun.",
  "better_version": "I usually go to an internet cafe near my house after work. For example, last Friday I played my favorite game for about two hours. Because of that, I felt really refreshed and ready for the weekend.",
  "what_changed": [
    "문장 나열 → 시간순 이야기 구조로 전환",
    "구체적 예시 추가 (지난주 금요일, 약 2시간)",
    "연결어 사용 (For example, Because of that)"
  ],
  "key_expressions": [
    {
      "expression": "I usually go to...",
      "type": "습관 표현",
      "usage": "루틴/묘사 질문에서 도입부로 사용"
    },
    {
      "expression": "For example, last Friday...",
      "type": "구체적 예시 연결",
      "usage": "일반 진술 → 구체적 에피소드 전환 시"
    },
    {
      "expression": "Because of that, I felt...",
      "type": "인과 + 감정 마무리",
      "usage": "답변 마무리 시 느낀 점으로 마감"
    }
  ]
}
```

**프롬프트 지시:**
```
answer_improvement 작성 규칙:
- corrected_version: 학생 답변의 문법/어휘만 수정. 구조는 바꾸지 않음. 학생의 의도 유지.
- better_version: {target_level} 수준으로 완전히 재구성.
  * 필수: 구체적 수치/빈도("일주일에 2-3번", "약 2시간"), 연결어 3개 이상, 도입-전개-마무리 구조
  * 추상적 표현("가끔", "종종") 금지 → 구체적 표현 필수
- what_changed: corrected → better 사이의 핵심 차이점 3가지
- key_expressions: better_version에서 학습자가 반드시 배워야 할 표현 3~5개
  * 각 표현에 type(표현 유형)과 usage(어떤 상황에서 쓰는지) 포함
  * 이 question_type에서 특히 유용한 표현 위주로 선별
```

### 3-5. Layer 4 — 영역별 점수 + 구조 + 등급 추정 (확정)

**구조:**
1. 영역별 점수 5개 (유창성/정확성/내용/구조/발음, 1-5점)
2. 구조 분석 (도입/전개/마무리 비율 + 구조 점수 + 스토리 밀도)
3. **이 답변의 추정 수준** (신규)

**JSON 스키마:**
```jsonc
"skill_summary": {
  "fluency":       { "score": 3, "label": "보통", "brief": "문장 사이 멈춤이 잦음" },
  "accuracy":      { "score": 2, "label": "부족", "brief": "관사, 동명사 오류 반복" },
  "content":       { "score": 3, "label": "보통", "brief": "주제 관련 답변이지만 구체성 부족" },
  "structure":     { "score": 2, "label": "부족", "brief": "도입/마무리 없이 문장 나열" },
  "pronunciation": { "score": 3, "label": "괜찮음", "brief": "대체로 이해 가능" }
},

"structure_evaluation": {
  "has_intro": false,
  "has_body": true,
  "has_conclusion": false,
  "structure_score": 2,
  "structure_label": "문장 나열",
  "time_distribution": { "intro_pct": 0, "body_pct": 95, "conclusion_pct": 5 },
  "story_density": { "example_count": 0, "detail_level": "low" },
  "tip": "도입부에서 'I usually...'로 시작하고, 마무리에 'Overall, I think...'를 넣어보세요."
},

// 🆕 이 답변의 추정 수준
"answer_level_estimate": {
  "estimated_level": "IM2",                  // GPT가 이 답변만 보고 추정
  "confidence": "medium",                    // high/medium/low
  "reason": "문장은 이어가지만 구조가 없고, 기본 문법 오류가 반복됨",
  "gap_to_target": {
    "target": "{target_level}",
    "key_gaps": [
      { "area": "구조", "current": 2, "needed": 4, "tip": "도입-전개-마무리 3단 구조" },
      { "area": "정확성", "current": 2, "needed": 4, "tip": "동명사/관사 오류 수정" }
    ]
  },
  "disclaimer": "※ 이 추정은 단일 문항 기반입니다. 확정 등급은 전체 평가에서 확인하세요."
}
```

**프롬프트 지시:**
```
answer_level_estimate 작성 규칙:
- 이 답변 하나만 보고 OPIc 등급을 추정하세요 (NH/IL/IM1/IM2/IM3/IH/AL)
- confidence: 답변이 짧거나 판단하기 어려우면 "low", 명확하면 "high"
- reason: 왜 이 등급으로 추정하는지 한 문장
- gap_to_target: {target_level}까지 부족한 영역과 필요 점수
  * 현재 점수와 필요 점수 차이가 큰 영역 2~3개만
- disclaimer는 고정 텍스트 (프론트엔드에서 표시)
```

## 4. 종합 평가 (Stage C) — 확정 설계

> **상태: ✅ 논의 완료, 구현 대기**
> **결정일: 2026-03-08**
> **핵심 철학**: 14문항 전체 데이터에 근거한 맞춤 처방. 일반적 조언이 아닌 "이 사람의 실제 데이터 기반" 진단서.

### 4-0. 종합 평가의 존재 이유

- **개별 평가 = 각 문항 진찰**: "이 답변에서 뭐가 틀렸고 어떻게 고치나"
- **종합 평가 = 진단서 + 처방전**: "14문항 전체를 관통하는 패턴 발견 + 다음에 뭘 해야 하나"
- 개별 평가가 아무리 좋아도, **전체적인 그림**은 종합 평가에서만 가능
  - "관사 누락이 14문항 중 9문항에서 반복된다"
  - "묘사는 잘하는데 비교에서 무너진다"
  - "기초 질문은 안정적인데 심화 질문에서 급격히 약해진다"
- **데이터 소스**: 14문항의 coaching_feedback (6-Layer) + 체크박스 합산 + 규칙엔진 FACT + 확정 등급

### 4-1. 내부 용어 차단 원칙 (확정)

> **종합 평가 전체에 적용되는 필수 규칙**

사용자에게 보이는 모든 텍스트에서 **내부 평가 용어를 절대 노출하지 않는다**.

| 내부 용어 | 사용자 친화 표현 | 적용 |
|----------|----------------|------|
| `INT` | "기초" 또는 표시 안 함 | UI + GPT 출력 |
| `ADV` | "심화" 또는 표시 안 함 | UI + GPT 출력 |
| `AL` | "고급" 또는 표시 안 함 | UI + GPT 출력 |
| `FACT` | 각 이름만 (말하기흐름, 문법정확성, 내용풍부도, 질문수행) | UI |
| `checkbox`, `pass_rate` | 절대 노출 안 함 | UI + GPT 출력 |
| `aggregated_*` | 절대 노출 안 함 | UI |
| `int_pass_rate`, `adv_pass_rate` | 절대 노출 안 함 | UI |
| `floor_status`, `ceiling_status` | 절대 노출 안 함 | UI |

**프롬프트 지시 (eval_comprehensive에 추가):**
```
## 내부 용어 사용 금지 (필수)
- 출력에 INT, ADV, AL, FACT, checkbox, pass_rate, floor, ceiling 등 내부 평가 용어를 절대 사용하지 마세요.
- "기초 질문", "심화 질문", "고급 질문"으로 표현하세요.
- "통과율", "체크박스"가 아닌 "정확도", "평가 영역"으로 표현하세요.
- 학습자는 평가 시스템의 내부 구조를 모릅니다. 코치가 알기 쉬운 말로 설명하듯 작성하세요.
```

### 4-2. 현재 vs 개선 후

| 항목 | 현재 V2 설계 | 개선 후 |
|------|------------|--------|
| 내부 용어 | INT/ADV/FACT 노출 | 사용자 친화 표현으로 전환 |
| 읽기 경로 | 없음 | 30초/이번 주/4주 3단계 행동 가이드 |
| 시험 전략 | 없음 | **14문항 실제 데이터 기반** 시간분석 + 돌발 대처 |
| 답변 공식 | 없음 | 확정 등급 기반 도입/마무리/연결어 + "이것만 외우세요" |
| 실천 계획 CTA | "스크립트/쉐도잉 연습" | "→ 튜터링에서 바로 시작" 버튼 |

### 4-3. GPT 출력 스키마 — 신규 3개 섹션

## 5. 튜터링 모듈 — v3 완전 재설계 ✅

> **기존 v2 4-Level 설계 폐기** → v3 "세션 + 프로토콜" 구조로 완전 대체 (2026-03-08)

### 5-1. v2 → v3 핵심 변경

| 항목 | 기존 v2 (4레벨) | 새 v3 (세션+프로토콜) |
|------|---------------|-------------------|
| 훈련 구조 | answer_type별 L1→L2→L3→L4 | **단일 세션 6 Screen + 5 프로토콜** |
| 훈련 단위 | "교정 답변 하나를 외우기" | **"능력을 여러 문항으로 체화"** |
| 새 질문 연습 | 없음 | **매 세션 471개 DB에서 새 질문 제시** |
| 오류 패턴 교정 | 없음 | **Oral Transformation + Self-repair** |
| 시간 압박 훈련 | 없음 | **15-60-15 타이머** |
| 변형 대응력 | 없음 | **Forced Variation (강제 변주)** |
| 과제 생성 | pass_rate 기반 수동 | **종합평가 처방에서 자동 변환** |
| 등급별 차이 | 동일 경로 | **파라미터만 자동 조절** |
| DB 구조 | 6테이블 | **7테이블** |

### 5-2. 세션 구조 요약

```
Screen 0 (Brief) → 1 (Warmup) → 2 (EPP) → 3 (Variation+Transformation) → 4 (Timed) → 5 (Self-repair) → 6 (Recap)
```

- **5 프로토콜**: EPP(핵심 표현 팩), Forced Variation, Oral Transformation, 15-60-15 Timer, Self-repair
- **핵심 원칙**: "피드백은 1회, 수행은 5회" — 같은 능력을 형태를 바꿔 5번 수행
- **등급별 파라미터**: 동일 세션 구조, IL→IM / IM→IH / IH→AL별 난이도만 자동 조절

### 5-3. 전문가 피드백 반영 현황
