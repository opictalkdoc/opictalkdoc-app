## 1. 현황 진단

### 현재 관리자 기능
| 항목 | 현황 | 문제 |
|------|------|------|
| 역할 체계 | ✅ `app_metadata.role = 'admin'` | JWT 기반 역할 확인 |
| 관리자 식별 | ✅ `requireAdmin()` + `getAdminUser()` | DB 조회 없이 JWT에서 role 확인 |

### 3-2. 감사 로그 테이블

```sql
CREATE TABLE admin_audit_log (
  id          BIGSERIAL PRIMARY KEY,
  admin_id    UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  action      TEXT NOT NULL,            -- 'user.credit_adjust', 'eval.retrigger', 'content.update' 등
  target_type TEXT,                      -- 'user', 'session', 'question', 'prompt' 등

-- 인덱스
CREATE INDEX idx_audit_admin_id ON admin_audit_log(admin_id);
CREATE INDEX idx_audit_action ON admin_audit_log(action);
CREATE INDEX idx_audit_created_at ON admin_audit_log(created_at DESC);

-- RLS: admin만 조회, service_role만 INSERT
ALTER TABLE admin_audit_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY "admin_audit_log_select"
  ON admin_audit_log FOR SELECT
  USING (auth.jwt() -> 'app_metadata' ->> 'role' = 'admin');

```

## 4. 권한 체계

### 4-1. 인증 헬퍼 (`lib/auth.ts` 확장)

```typescript
// 기존 getUser(), getAuthClaims() 유지

// 레이아웃용: getUser() 후 app_metadata.role 확인
export const getAdminUser = cache(async () => {
  const user = await getUser();
  if (!user) return null;
  if (user.app_metadata?.role !== "admin") return null;
  return user;
});

// Server Actions용: admin 아니면 에러, service client 반환
export async function requireAdmin() {
  const user = await getUser();
  if (!user) redirect("/login");
  if (user.app_metadata?.role !== "admin") throw new Error("관리자 권한이 필요합니다");
  const supabase = createClient(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY);
  return { supabase, userId: user.id };
}
```

### 4-2. 미들웨어 확장 (`middleware.ts`)

```typescript
// 기존 protectedRoutes, authRoutes 유지

// 추가: admin 라우트 보호

  // 레이아웃의 서버 컴포넌트에서 getAdminUser()로 검증
```

> **설계 결정**: 미들웨어에서는 인증 여부만 확인하고, admin 역할 검증은 `(admin)/layout.tsx` 서버 컴포넌트에서 수행한다. 이유:
> - 미들웨어에서 DB 쿼리(profiles.role)는 모든 요청에 지연을 추가
> - 레이아웃에서 검증하면 admin 페이지 접근 시에만 1회 확인
> - Server Actions에서 `requireAdmin()`으로 이중 검증

## 7. 페이지별 설계

### 7-1. 관리자 대시보드 (`/admin`)

**목적**: 전체 서비스 현황을 한눈에 파악

**주요 지표 카드 (4열 그리드)**:
| 카드 | 데이터 소스 | 설명 |
|------|-----------|------|
| 전체 회원 수 | `profiles` COUNT | 총 가입자 |
| 오늘 활성 사용자 | `mock_test_sessions` + `scripts` (today) | DAU 근사치 |
| 이번 달 매출 | `orders` SUM(amount) WHERE status='paid' | 월 매출 |
| 평가 대기 | `mock_test_answers` WHERE eval_status NOT IN ('completed','skipped','failed') | 처리 중 답변 |

**하단 섹션**:
- 최근 가입 회원 (5명)
- 최근 결제 내역 (5건)
- 평가 실패 세션 (최근 5건)
- 모듈별 사용 현황 (시험후기/스크립트/모의고사/튜터링)

### 7-2. 사용자 관리 (`/admin/users`)

**기능**:
1. **회원 목록**: 테이블 (이메일, 이름, 가입일, 플랜, 크레딧 잔량)
2. **검색/필터**: 이메일 검색, 플랜별 필터
3. **크레딧 조정**: 모달 → 수동 지급/차감 + 사유 입력 → 감사 로그 기록
4. **사용자 상세**: 클릭 시 상세 패널 (후기/스크립트/모의고사/튜터링 이력 요약)

**Server Actions**:
```typescript
// lib/actions/admin/users.ts
getUsers(page, search, planFilter)    // 페이지네이션 + 검색 + 필터
getUserDetail(userId)                  // 사용자 상세 (크레딧 + 활동 요약)
adjustCredit(userId, type, amount, reason)  // 크레딧 수동 조정 + 감사 로그
```

### 7-3. 결제 관리 (`/admin/payments`)

**기능**:
1. **주문 목록**: 테이블 (결제일, 사용자, 상품, 금액, 상태, PG TxID)
2. **필터**: 상태별 (paid/refunded/cancelled), 기간별
3. **상세 조회**: 결제 상세 정보 + 관련 크레딧 변동

**Server Actions**:
