## 1. 핵심 원칙: 의사 → 처방전 → 물리치료

| 모듈 | 역할 | 비유 | 핵심 질문 |
|------|------|------|----------|
| **개별 평가** (Stage B) | 진단 + 교정 | 의사의 진찰 | "이 답변에서 뭐가 문제야?" |
| **종합 평가** (Stage C) | 진단서 + 처방전 | 의사의 소견서 | "전체적으로 어떤 상태야?" |
| **튜터링** | 치료 + 재활 + 추적 | 물리치료사 | "지금 당장 연습하자" |

### 경계 원칙

- ❌ 개별 평가에서 연습 과제/스크립트 암기 지시
- ❌ 개별 평가에서 다른 문항까지 묶는 총평/등급 예측
- ❌ 종합 평가에서 실제 녹음/연습 UI
- ❌ 튜터링에서 등급 판정/체크박스 재실행
- ✅ 개별 평가 교정 데이터 → 종합 평가가 집계하여 패턴 발견
- ✅ 종합 평가 처방 → 튜터링 자동 과제 생성
- ✅ 튜터링 성장 데이터 → 다음 모의고사에서 변화 추적

### 내부 용어 차단 (전 모듈 적용)

| 내부 용어 | 사용자에게 보이는 표현 |
|----------|---------------------|
| INT, ADV, AL | "기초", "심화", "고급" (또는 표시 안 함) |
| FACT | 각 이름만 (말하기흐름, 문법정확성, 내용풍부도, 질문수행) |

## 2. 개별 평가 (Stage B) — v3 확정 설계

> **v2 → v3 핵심 변경**: 6-Layer 데이터 구조 유지 + **학습자 관점 5단계 표시 순서** 도입 + **타입별 체크리스트** + **피드백 3분기** (품질 판정은 GPT에 위임)
>
> **근거**: GPT-5.4 OPIc 전문가 자문 (2026-03-08)
> **원칙**: "학습자가 궁금한 순서대로" — 제대로 답했나? → 뭘 고쳐야 돼? → 구조가 맞아? → 문법? → 발음?

### 2-1. v2 → v3 변경 요약

| 항목 | v2 | v3 |
|------|----|----|
| 표시 순서 | 6-Layer 순서대로 | **학습자 관점 5단계** (과제충족 최상위) |
| 과제충족 판정 | 없음 (한줄 인사이트에 암묵적) | **명시적 충족/부분충족/실패 판정** |
| 과제충족 기준 | - | **🔴필수/🟡심화 분리 + 목표 등급별 threshold** |
| 처방 우선순위 | 없음 | **최우선 처방 1~2개** (2번째 표시) |
| 타입별 차별화 | question_type 공략법 (Layer 5에 포함) | **10개 타입별 체크리스트 + 교정 템플릿** |
| 무응답 감지 | skip-detector (15초/15자/환청) | **동일** — process 단계 정량 3조건만. 품질 판정은 GPT(B-1)에 위임 |
| 피드백 분기 | 단일 피드백 | **3분기** (무응답/부분응답/정상응답) |
| 구제 메시지 | 없음 | **타입별 시작틀 + 동기 보호 메시지** |
| 구현 방식 | - | **별도 specs 테이블 불필요** — evaluation_prompts에 섹션 추가 |
| GPT 모델 | gpt-4.1 단일 | **B-1 판정: gpt-4.1-mini** + **B-2 코칭: gpt-4.1** (역할별 분리) |
| EF 구조 | 3-Stage (process→eval→report) | **4-Stage** (process→eval-judge→eval-coach→report) |
| 비용 | $0.53/test | **$0.41/test (-24%)**, 무응답 포함 시 ~$0.35 (-34%) |

### 2-2. 피드백 5단계 표시 구조 (v3)

학습자가 문항별 피드백을 볼 때의 **표시 순서**:

| 순서 | 단계 | 내용 | v2 매핑 |
|------|------|------|---------|
| **1** | 과제충족 판정 | 충족/부분충족/실패 + 근거 1줄 | 🆕 (v2에 없음) |
| **2** | 최우선 처방 1~2개 | "이것만 고치면 등급 UP" 행동 지침 | 🆕 (v2에 없음) |
| **3** | 구조 + 교정 | 구조 진단 + 언어 교정 TOP 3 (Before→After) | v2 Layer 2+3+5 통합 |
| **4** | 답변 개선 | 내 답변 → 교정 → 목표 등급 + 핵심 표현 | v2 Layer 3 유지 |
| **5** | 전달 + 습관 | 영역별 점수 + WPM/필러/발음 + 등급 추정 | v2 Layer 4+6 통합 |
| *선택* | 좋았던 점 | 이 답변에서 유지할 강점 1개 | v2 Layer 5 strengths |

> **데이터 생성(GPT)**은 v2의 6-Layer JSON 스키마를 그대로 사용. **표시 순서만** v3로 재배치.
> v2 JSON 스키마 상세: `모의고사-평가-v2-개선방향.md` §3-1 ~ §3-7

### 2-3. 과제충족 판정 시스템 (🆕 v3 신규)

> **핵심 설계 결정**: 체크리스트 항목을 🔴 필수 / 🟡 심화로 분리하고, 목표 등급별로 충족 기준선을 다르게 적용한다.
> **근거**: OPIc 실제 채점에서 Task Fulfillment 기대치가 등급별로 다름 (IM=기본 과제 수행, IH=디테일 포함, AL=풍부한 수행)
> **구현**: 별도 specs 테이블 불필요 — 기존 `evaluation_prompts` 8개 프롬프트에 과제충족 섹션 추가

#### 체크리스트 구조

| 태그 | 의미 | 대상 등급 |
|------|------|----------|
| 🔴 **필수** (Required) | 이것 없으면 과제 자체를 수행하지 않은 것 | **전 등급** (IM 이하 포함) |
| 🟡 **심화** (Advanced) | 있으면 답변 품질이 한 단계 올라감 | **IH 이상** 요구 |

#### 등급별 충족 기준선 (Threshold)

| 목표 등급 | 충족 조건 | 부분충족 | 실패 |
|----------|----------|---------|------|
| **IM 이하** (IL, IM1~3) | 🔴 필수 전부 충족 | 🔴 필수 1개 이상 미충족 | 🔴 필수 절반 이상 미충족 |
| **IH** | 🔴 필수 전부 + 🟡 심화 1개 이상 | 🔴 필수 전부 but 🟡 심화 0개 | 🔴 필수 미충족 |
| **AL** | 🔴 필수 전부 + 🟡 심화 전부 | 🔴 필수 전부 + 🟡 심화 일부 | 🔴 필수 미충족 |

> **효과**: IM 목표 학습자가 필수 3개를 다 하면 "충족" → 동기 보호. 심화 항목은 "최우선 처방"으로 성장 방향 제시.
> **효과**: IH 목표 학습자가 필수만 하면 "부분충족" → 적절한 성장 압박.

#### question_type별 과제충족 체크리스트

GPT eval 프롬프트에서 타입별 체크리스트를 판정하고, 목표 등급 기준선에 따라 충족/부분충족/실패를 자동 판정한다.

##### 1) description (묘사)

**체크리스트:**
- 🔴 묘사 대상이 분명함 (무엇/어디/누구)
- 🔴 핵심 특징 2개 이상
- 🔴 현재시제 중심으로 안정적
- 🟡 예시·감각·개인화 1개 이상 (단순 형용사 나열 아님)
- 🟡 개인 반응/선호 이유 있음

**이상적 전개:** 대상 소개 → 전체 인상 → 세부 특징 2~3개 → 예시/경험 1개 → 짧은 마무리

**자주 범하는 실수 TOP 3:**
1. 형용사만 나열 (big, nice, beautiful 반복)
2. 구체성 없음 (색, 분위기, 사용 방식 빠짐)
3. 묘사 중 이야기로 새거나 너무 단답

**핵심 처방:** "특징 1개 + 예시 1개" 묶음으로 선명하게
**피드백 톤:** 격려형 코칭 (4~5문장)

##### 2) routine (루틴/습관)

**체크리스트:**
- 🔴 어떤 루틴인지 분명함
- 🔴 빈도 표현 있음 (usually, every day, on weekends)
- 🔴 현재시제로 안정적
- 🟡 순서 연결어 사용 (first, then, after that)
- 🟡 이유/느낌/예외 상황 짧게 있음

**이상적 전개:** 루틴 소개 → 빈도 제시 → 순서대로 3단계 설명 → 보통/예외 구분 → 짧은 느낌

**자주 범하는 실수 TOP 3:**
1. 빈도 없이 행동만 나열
2. 순서가 흐려짐
3. 현재 습관인데 과거시제로 흔들림

**핵심 처방:** 빈도 + 순서 연결어를 먼저 고정
**피드백 톤:** 직설적 코칭 (4~5문장)

##### 3) comparison (비교/변화)

**체크리스트:**
- 🔴 비교의 두 축이 분명함 (예전 vs 지금)
- 🔴 비교 포인트 최소 2개
- 🔴 대비 표현 사용 (compared to, unlike, these days)
- 🟡 변화의 원인/배경이 있음
- 🟡 변화에 대한 본인 의견 있음

**이상적 전개:** 비교 기준 제시 → 과거 상태 → 현재 상태 → 왜 바뀌었는지 → 개인적 평가

**자주 범하는 실수 TOP 3:**
1. 한쪽만 말함 (지금만 길게, 과거 없음)
2. 원인 없이 차이만 말함
3. 비교 표현 없이 나열식 설명

**핵심 처방:** 첫 문장에서 비교축 세우기 — "In the past…, but these days…"
**피드백 톤:** 분석형 코칭 (5문장 전후)

##### 4) past_childhood (어릴 때 경험)

> ⚠️ DB question_type 키: `past_childhood` (설계 초안의 `past_experience_childhood`에서 변경)

**체크리스트:**
- 🔴 "어릴 때"라는 시간 앵커 분명 (when I was a child)
- 🔴 배경 정보 있음 (어디서, 누구와, 어떤 상황)
- 🔴 실제 사건/행동이 과거시제로 나옴
- 🟡 감정/반응 있음
- 🟡 지금 돌아보는 의미 또는 지금과 어떻게 연결되는지

**이상적 전개:** 어린 시절 시점 → 배경 → 무슨 일이 있었는지 → 그때 반응 → 지금 돌아보는 의미

**자주 범하는 실수 TOP 3:**
1. childhood 앵커 없이 그냥 과거 경험처럼 말함
2. 과거시제가 약함
3. 사건보다 배경 설명만 길어짐

**핵심 처방:** 첫 문장에 childhood marker, 마지막에 지금 의미 붙이기
**피드백 톤:** 스토리 코칭형 (5~6문장)

##### 5) past_special (기억에 남는/특별한 경험)

> ⚠️ DB question_type 키: `past_special` (설계 초안의 `past_experience_memorable`에서 변경)

**체크리스트:**
- 🔴 어떤 경험이 기억에 남는지 명확
- 🔴 배경 있음 (시간/장소/함께한 사람)
- 🔴 핵심 사건이 분명함
- 🟡 감정이나 반응 있음
- 🟡 왜 기억에 남는지 이유 분명

**이상적 전개:** 기억에 남는 경험 소개 → 배경 → 핵심 사건 → 감정/반응 → 왜 기억나는지

**자주 범하는 실수 TOP 3:**
1. "재밌었다"만 있고 사건이 없음
2. 시간순 전개가 불분명
3. 왜 memorable인지 설명이 약함

**핵심 처방:** 사건 1개를 중심으로 서사화 — "무슨 일이 있었는지"가 핵심
**피드백 톤:** 격려 + 구조 교정 (5~6문장)

##### 6) past_recent (최근 경험)

> ⚠️ DB question_type 키: `past_recent` (설계 초안의 `past_experience_habitual`에서 변경)
> 설계 초안의 "과거 습관"에서 "최근 경험"으로 변경됨. DB questions 테이블의 실제 분류 기준을 따름.

**체크리스트:**
- 🔴 최근 경험이라는 시간 앵커 분명 (recently, last week, a few days ago)
- 🔴 배경 정보 있음 (어디서, 누구와, 어떤 상황)
- 🔴 과거시제 안정적
- 🟡 감정/반응 있음
- 🟡 경험에서 배운 점 또는 영향 있음

**이상적 전개:** 최근 경험 소개 → 배경 → 무슨 일이 있었는지 → 반응/느낌 → 배운 점

**자주 범하는 실수 TOP 3:**
1. 단발성 사건처럼 말함
2. 현재 루틴과 섞임
3. 시제 흔들림

**핵심 처방:** 시간 앵커 + 과거시제 고정 + 감정/배운 점 마무리
**피드백 톤:** 직설형 교정 (4~5문장)

##### 7) roleplay_11 (Q11 정보 요청 롤플레이)

**체크리스트:**
- 🔴 목적/상황이 드러남
- 🔴 질문 개수 3개 이상
- 🔴 질문이 WH-질문 중심 (단순 yes/no 아님)
- 🟡 정중 표현 사용 (Could you~, Would you~)
- 🟡 상대 답변을 지어내지 않음

**이상적 전개:** 인사/상황 제시 → 질문 1 → 질문 2 → 질문 3 → 짧은 마무리

**자주 범하는 실수 TOP 3:**
1. 질문이 아니라 진술문으로 말함
2. 질문 수가 부족함
3. 혼자 질문하고 혼자 답까지 함

**핵심 처방:** 문장 품질보다 질문 개수 확보가 먼저
**피드백 톤:** 체크리스트형, 직설적 (3~4문장)

##### 8) roleplay_12 (Q12 상황 대응 롤플레이)

**체크리스트:**
- 🔴 사과 또는 상황 인정
- 🔴 이유/원인 설명
- 🔴 해결 옵션 2개 이상 제시
- 🟡 현재 상태 설명
- 🟡 상대의 선택/확인 유도

**이상적 전개:** 사과 → 이유 설명 → 현재 상황 → 옵션 A / 옵션 B → 어떤 게 괜찮은지 묻기

**자주 범하는 실수 TOP 3:**
1. 사과만 하고 끝남
2. 옵션이 1개뿐이거나 없음
3. 상대 선택 유도 없이 독백으로 끝남

**핵심 처방:** 사과 + 이유 + 옵션 2개 — 이 3개가 필수
**피드백 톤:** 문제 해결 코칭형 (4~5문장)

##### 9) advanced_14 (Q14 사회 비교/변화)

**체크리스트:**
- 🔴 사회적 주제가 분명함
- 🔴 과거 vs 현재 또는 집단 간 비교 있음
- 🔴 이유/배경 최소 2개
- 🟡 개인 경험만이 아닌 사회적 관점
- 🟡 예시 또는 관찰 사례 있음

**이상적 전개:** 주제 제시 → 사회적 비교/변화 → 원인 1 → 원인 2 → 예시/시사점

**자주 범하는 실수 TOP 3:**
1. 개인 이야기만 하고 사회적 관점 없음
2. 비교는 있는데 이유가 약함
3. 추상적 주장만 있고 예시가 없음

**핵심 처방:** 개인 경험을 사회 현상으로 끌어올리기
**피드백 톤:** 논리 코칭형 (5~6문장)

##### 10) advanced_15 (Q15 의견 제시/주장)

**체크리스트:**
- 🔴 입장이 명확함
- 🔴 이유 최소 2개
- 🔴 논리 연결어 사용 (because, therefore, for example)
- 🟡 예시 1개 이상 (구체적)
- 🟡 결론/재진술 있음

**이상적 전개:** 입장 제시 → 이유 1 + 예시 → 이유 2 + 예시/설명 → 결론

**자주 범하는 실수 TOP 3:**
1. 입장이 모호함
2. 같은 이유를 반복
3. 예시 없이 추상적으로만 말함

**핵심 처방:** 첫 문장에 입장 + 이유 2개 분리하여 말하기
**피드백 톤:** 명확하고 코칭적 (5~6문장)

### 2-4. 스킵 판정 + 피드백 분기 (v3 확정)

> **핵심 원칙**: "스킵 = 사용자가 문제를 풀지 않겠다는 의사." 답변 품질이 낮아도 사용자가 답했으면 무조건 평가한다.
> regex 기반 NLP(meta_only, 질문 개수 감지, 구조 감지)로 품질을 판정하면 false positive가 발생한다 (실증: "Yeah, I remember..."로 시작하는 78단어 정상 답변이 무응답 처리됨). 품질 판정은 GPT(B-1)에 위임.

#### 스킵 판정 (process 단계, 3조건만)

스킵은 **사용자가 안 푼 경우**만 감지한다. `skip-detector.ts` 기존 3가지:

| # | 조건 | 이유 | 생략 범위 |
|---|------|------|-----------|
| 1 | `audio_duration ≤ 15초` | 아예 안 말함 | STT + Azure + GPT 전체 |
| 2 | `transcript ≤ 15자` | 사실상 안 말함 | Azure + GPT |
| 3 | Whisper 환청 (이모지만) | STT 아티팩트 | Azure + GPT |

> **삭제된 것**: v3 초안의 process 단계 파생 지표(meta_only, filler_ratio, unfinished_end) 기반 스킵 판정, 타입별 failWords/minWords, rp_11 질문 개수 감지, rp_12 구조 감지 — 전부 제거. 답변 품질 판정은 B-1(GPT)이 담당.

#### feedback_branch 결정 (B-1 단계)

GPT eval-judge가 `task_fulfillment.status`를 판정하면, **그 값을 그대로 `feedback_branch`로 사용**한다.

```typescript
// eval-judge에서 feedback_branch 결정
function determineFeedbackBranch(taskFulfillment) {
  if (taskFulfillment.status === "failed") return "failed";
  if (taskFulfillment.status === "partial") return "partial";
  return "fulfilled";
}
```

> **삭제된 것**: v3 초안의 3축 종합 판정 (내용+전달+발화질 각 축 심각도 계산 후 2축 이상이면 failed 등). GPT가 transcript, wpm, filler_count, 발음 점수를 모두 볼 수 있으므로 별도 정량 축 계산이 불필요.

| 분기 | 조건 | B-2 호출 | 피드백 톤 |
|------|------|----------|----------|
| **fulfilled** | 🔴필수 전부 + 등급별 🟡심화 충족 | eval_coach_standard | 표준 5단계 |
| **partial** | 🔴필수 일부 미충족 | eval_coach_partial | 빠진 요소 강조 |
| **failed** | 🔴필수 대부분 미충족 또는 과제 이탈 | 호출 안 함 (룰 기반 구제 메시지) | 축소+구제 |

### 2-5. 피드백 3분기 체계 (🆕 v3 신규)

#### A. 무응답/스킵 피드백

**목적:** 답변 붕괴 원인 진단 + 학습 의욕 보호 + 최소 행동 1개 제시

**구성 (축소형):**
1. **과제충족**: 실패
2. **최우선 처방**: 1개만 (시작 진입 방법)
3. **구조 진단**: 전체 구조가 아닌 "시작 진입 실패" 지적
4. **언어 교정**: 오류 TOP3 대신 **시작 문장 + 복구 문장 1~2개**만
5. **전달 지표**: "긴 침묵으로 수치 해석 제한" 명시

**생략/축소하는 것:**
- 긴 문법 설명
- 발음 상세 분석
- TOP3 오류를 억지로 뽑아내는 것
- "아예 못했다" 같은 낙인 표현

#### B. 부분 응답 피드백

**목적:** "무엇이 빠졌는가"를 명확히 보여주기

**구성 (표준 + 빠진 요소 강조):**
1. **과제충족**: 부분 충족 + 빠진 필수 요소 명시
2. **최우선 처방**: 빠진 필수 요소 1~2개
3. **구조 진단**: 누락 단계 명시
4. **언어 교정 TOP 3**: 실제 발화 기반
5. **전달 지표**: pause/filler가 구조 붕괴와 연결된다는 점 설명

#### C. 정상 응답 피드백

**목적:** 과제 충족 확인 + 정교화에 집중

**구성:** 표준 5단계 전부 적용
1. 과제충족 판정 (충족)
2. 최우선 처방 1~2개
3. 구조 + 언어 교정
4. 답변 개선 (corrected → better)
5. 전달 + 습관
+ 좋았던 점 1개

### 2-6. 무응답 구제 메시지 설계 (🆕 v3 신규)

#### 톤 원칙

| 해야 할 톤 | 피해야 할 톤 |
|-----------|------------|
| 차분함 | 단정적 비난 |
| 사실 전달 | 실력 전체를 평가하는 말 |
| 회복 가능성 제시 | "준비 안 됐다", "기초 부족" |
| 행동 1개만 제시 | 장황한 문법 강의 |

#### 구제 메시지 구조

1. **상태를 사실적으로** — "답변이 짧았다"가 아니라 "긴 멈춤 때문에 과제 수행이 중단된 상태"
2. **원인을 좁혀주기** — "영어가 부족해서"가 아니라 "시작 문장을 잡지 못해 뒤가 무너진 케이스"
3. **회복 포인트 1개** — "첫 문장 + 포인트 2개만 잡아도 바로 회복됩니다"
4. **즉시 쓸 수 있는 시작틀 제시**

#### 타입별 시작틀 (무응답 구제용)

| question_type | 시작틀 |
|--------------|--------|
| description | "One place I can describe is…" |
| routine | "I usually… First,…" |
| comparison | "In the past…, but these days…" |
| past_childhood | "When I was a child,…" |
| past_special | "One memorable experience I had was…" |
| past_recent | "Recently, I…" |
| roleplay_11 | "Hi, I'd like some information. First,…" |
| roleplay_12 | "I'm sorry, but… Would you prefer A or B?" |
| advanced_14 | "Compared to the past, these days…" |
| advanced_15 | "I think … for two reasons." |

> ⚠️ **구현 파일**: `supabase/functions/_shared/question-type-map.ts` — `buildRescueMessage()` 함수에서 타입별 시작틀 생성

#### 구제 메시지 예시 (GPT 프롬프트 참고용)

> "이번 답변은 영어 지식 부족이라기보다 **긴 멈춤으로 인해 답변이 중단된 경우**에 가깝습니다. 이 유형은 완벽하게 말하려고 버티기보다, **첫 문장으로 바로 진입하는 것**이 더 중요합니다. 다음에는 '{시작틀}'로 시작해보세요. 이것만으로도 스킵 상태는 바로 벗어납니다."

## 3. 종합 평가 (Stage C) — v3 확정 설계

> **v2 → v3 전면 재설계.** GPT-5.4 OPIc 전문가 자문 (2026-03-08) + Claude 아키텍처 설계
> **핵심 변경**: v2의 9개 섹션 → v3 **9개 섹션 재구성** + **튜터링 처방 데이터** 통합 + **GPT/규칙엔진/클라이언트 역할 분리**
> **원칙**: "데이터를 많이 보여주는 것"이 아니라 "다음 2주 동안 무엇을 연습해야 하는지 명확히 정해주는 것"

### 3-0. 종합 평가의 존재 이유

사용자가 모의고사 후 궁금한 건 4가지뿐이다:
1. **내 등급이 뭐냐**
2. **왜 거기서 막혔냐**
3. **뭘 고치면 올라가냐**
4. **지금 바로 연습할 수 있냐**

종합 평가는 **성적표가 아니라 "처방전"**이다. 개별 평가가 이미 문항별 상세를 다 해주므로, 종합 평가는 **"14문항을 관통하는 패턴 + 우선순위 + 다음 행동"**에 집중한다.

**개별 평가와의 경계:**
- 개별 평가: "이 문항에서 뭐가 틀렸는지" (진찰)
- 종합 평가: "14문항 전체를 봤을 때 패턴/우선순위/다음 행동" (진단서 + 처방전)
- 중복 금지: 개별 평가 내용을 종합 평가에서 반복하지 않음

### 3-1. 데이터 소스 역할 분리: GPT vs 규칙엔진 vs 클라이언트

> **핵심 결정**: GPT는 "해석/패턴/처방"에만 집중. 숫자/정렬/파라미터는 기존 시스템이 담당.

| 데이터 | 생성 주체 | 이유 |
|--------|----------|------|
| 한줄 총평, 진단 태그 | **GPT** | 해석이 필요 |
| FACT 해석 문장 | **GPT** | 숫자 → 의미 번역 |
| FACT 점수 숫자, 통과율 | **규칙엔진** | 이미 계산 완료 |
| 등급 결정 핵심 사유 3개 | **GPT** | 패턴 분석 + 해석 |
| Top 3 처방 카드 | **GPT** | 핵심 가치 — 우선순위 판단 |
| 목표 등급 로드맵 (기본 조건) | **DB 템플릿** | 등급별 조건은 OPIc 구조상 거의 불변 |
| 목표 등급 로드맵 (개인화) | **GPT** | 이 사람의 경우 특히 어디가 부족한지 |
| 유형별 진단 맵 | **GPT** | 10개 타입별 상태 분석 |
| 반복 오류 패턴 | **GPT** | 14문항 관통 패턴 추출 |
| 발음/습관 수치 (WPM, 필러, 침묵) | **클라이언트 집계** | answers 테이블에서 집계 |
| 발음/습관 해석 문장 | **GPT** | 목표 등급 기준으로 번역 |
| 강점 1~2개 | **GPT** | 해석 필요 |
| 문항별 정렬 (우선순위순) | **클라이언트** | 개별 평가 데이터로 정렬 |
| CTA (훈련 시작) | **클라이언트** | UI 컴포넌트 |
| 튜터링 파라미터 (hint, timer 등) | **튜터링 모듈** | 등급별 파라미터 종합표가 source of truth |

모의고사 14문항의 개별 평가 데이터를 종합하여, 학습자에게 "진단서 + 처방전"을 작성합니다.

## 출력 규칙

### 언어
- 모든 텍스트는 **한국어**로 작성합니다.
- 예문(before/after)만 영어로 작성합니다.

### 내부 용어 차단 (절대 준수)
- INT, ADV, AL → "기초", "심화", "고급"
- FACT → 각 영역명만 사용 (말하기흐름, 문법정확성, 내용풍부도, 질문수행력)
- checkbox, pass_rate, floor, ceiling → 절대 노출 안 함
- 등급 약어(NH, NM, IL, IM1~3, IH, AL) → 사용자에게는 그대로 사용 가능 (OPIc 공식 표기)

### 목표 등급 기준 번역 원칙
- 모든 수치/상태를 **목표 등급({target_level}) 기준**으로 해석합니다.
- ❌ "긴 침묵 8회" → ✅ "목표 {target_level} 기준에서 침묵이 상향을 막고 있습니다"
- ❌ "답변 22초" → ✅ "목표 등급 기준에서 의견형 답변이 짧은 편입니다"

### 톤
- 차분하고 전문적인 코칭 톤
- 학습자를 {nickname}으로 호칭
- 부정적 결과도 회복 가능성과 함께 전달
- "망했다"가 아니라 "이 순서대로 하면 됩니다"

### 구조화 데이터 규칙
- drill_tag는 반드시 아래 목록에서만 선택:
  description_detail, routine_sequence, comparison_frame, past_narrative,
  tense_consistency, roleplay_questions, roleplay_recovery, opinion_support,
  social_perspective, detail_expansion, example_insertion, transition_words,
  vocabulary_variety, polite_expression, opening_entry, closing_wrap,
  pause_control, filler_reduction, sentence_completion

### top3_priorities 구성 규칙
- 가능하면 3개 영역(task_performance, content_structure, delivery)에서 각 1개씩 구성
- 단, 특정 영역에 심각한 문제가 없으면 **가장 개선 여지가 큰 것**을 선택. 억지로 약점을 만들지 않음
- 3개 미만은 불가. 반드시 3개를 채우되, 경미한 영역은 "개선하면 좋은 점"으로 작성

### roadmap optional 필드 처리
- next_to_target: 현재→목표가 1단계 차이이면 **null** 출력
- long_term_goals: next_to_target이 null이면 **빈 배열 []** 출력
- 키 자체는 절대 생략하지 않음 (JSON 파싱 안정성)

### 토큰 예산 (중요)
- 전체 JSON 출력: **4,000~5,000 토큰** 이내 목표
- 각 텍스트 필드: 2문장 이내 (headline, comment, reason 등)
- before/after 예문: 각 1~2문장
- recurring_patterns: 3~5개 (5개 초과 금지)
- question_type_map comment: 15자 이내

### 3-8a. tutoring_prescription 자동 변환 로직 (🆕 v3)

> **핵심 결정**: GPT에게 tutoring_prescription을 직접 생성하게 하면 top3_priorities와 중복+불일치 위험.
> EF 코드가 GPT 출력(top3/patterns/type_map) + 규칙엔진 데이터에서 **결정론적으로 변환**.

```typescript
// mock-test-report EF에서 실행
function buildTutoringPrescription(coachingReport, ruleEngineResult) {
  const { top3_priorities, recurring_patterns, question_type_map } = coachingReport;

  // 1. priority_weaknesses: top3에서 직접 추출
  const priority_weaknesses = top3_priorities.map((p, i) => ({
    rank: p.rank,
    area: p.area,
    subtype: p.where?.[0] ? getQuestionType(p.where[0]) : p.area,
    drill_tag: p.drill_tag
  }));

  // 2. error_drill_tags: recurring_patterns에서 추출
  const error_drill_tags = [...new Set(
    recurring_patterns.map(p => p.drill_tag)
  )];

  // 3. weak_types: question_type_map에서 weak/very_weak 추출
  const weak_types = question_type_map
    .filter(t => t.status === 'weak' || t.status === 'very_weak')
    .sort((a, b) => (a.status === 'very_weak' ? 0 : 1) - (b.status === 'very_weak' ? 0 : 1))
    .map(t => t.type);

  // 4. training_order: priority 기준 정렬
  const training_order = question_type_map
    .filter(t => t.priority)
    .map(t => t.type);

  // 5. must_fix_for_next_grade: top3의 drill_tag (등급 직결)
  const must_fix_for_next_grade = top3_priorities.map(p => p.drill_tag);

  return {
    priority_weaknesses,
    error_drill_tags,
    weak_types,
    training_order,
    must_fix_for_next_grade
  };
}
```

**장점**: GPT 토큰 ~800개 절감 + top3↔prescription 불일치 원천 차단 + drill_tag 닫힌 세트 보장

### 3-9. 튜터링 처방 연결

> 튜터링 모듈(docs/설계/튜터링.md)의 **처방 탭**이 소비하는 데이터.

```
coaching_report (JSONB) — GPT 출력
  ├─ top3_priorities                   → 진단 탭 약점 Top 3 표시
  ├─ question_type_map                 → 진단 탭 유형별 진단 표시
  ├─ recurring_patterns                → 반복 오류 드릴 자동 생성
  ├─ roadmap                           → 진단 탭 목표 로드맵 표시
  └─ training_recommendation           → CTA 텍스트/세션 수

tutoring_prescription (JSONB) — EF 자동 변환 (§3-8a)
  ├─ priority_weaknesses (Top 3)       → 약점 Top 2 추출 → 세션 자동 구성
  ├─ error_drill_tags                  → 오류 패턴 드릴 포함
  ├─ weak_types                        → 약한 유형 Top 2~3 → 해당 유형 문항 선택
  ├─ training_order                    → 훈련 순서 결정
  └─ must_fix_for_next_grade           → drill_tag 기반 처방 우선순위

규칙엔진 결과 (reports 테이블)
  ├─ final_level + target_level        → 튜터링 등급별 파라미터 자동 세팅
  └─ fact_scores                       → 진단 탭 FACT 표시

개별 평가 (evaluations 테이블)
  └─ 구조점수 ≤ 2 문항                  → 추가 훈련 대상 문항 추출
```

**핵심 1**: tutoring_prescription은 GPT가 아닌 **EF 코드가 자동 변환** → top3↔prescription 불일치 차단.
**핵심 2**: 튜터링 파라미터(hint_mode, timer_mode, repetition_count)는 **튜터링 모듈 자체의 등급별 파라미터 종합표**가 source of truth. `confirmed_grade`만 넘기면 자동 결정. GPT가 이 값을 출력하지 않음.
**핵심 3**: must_fix_for_next_grade는 drill_tag 택소노미(§3-4)에서만 선택. 별도 형식 없음.

#### 성장 패턴 4종 감지 (`detectGrowthPattern()`)

3회 이상 세션 이력이 있을 때 자동 감지:

| 패턴 ID | 설명 | 감지 조건 |
|---------|------|----------|

| `t_unstable` | T(질문수행력)가 불안정 | T 점수가 연속 오르내림 |
| `a_plateau` | A(문법)가 정체 | A 점수 변화 3회 연속 ±0.5 이내 |
| `score_up_level_same` | 점수는 올랐는데 등급은 그대로 | total_score ↑ but final_level 동일 |
