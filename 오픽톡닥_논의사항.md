## 논의 1: 스크립트 생성 — 등급 설정 방식

### 현황
- 마이페이지에 `target_grade` 필드 이미 구현됨 (IL, IM1, IM2, IM3, IH, AL)
- `user_metadata`에 저장 중 (Supabase Auth)
- master_questions에는 명시적 난이도 컬럼 없음 (topic_category로 간접 구분만)

### 핵심 질문
> 사용자가 등급을 설정해야 하는 걸 모를 수 있다. 어떻게 자연스럽게 설정을 유도할 것인가?

### 결정 ✅

- **선택**: 넛지 배너 + 인라인 모달
- **결정일**: 2026-02-21

### 결정 내용

**1) 넛지 배너**
- 등급 미설정 시, 모든 주요 페이지 상단에 안내 배너 표시
- 적용 범위: 대시보드, 시험후기, 스크립트, 모의고사, 튜터링 (전 페이지)
- 설정 완료 시 배너 자동 사라짐

**2) 인라인 모달 (배너의 [설정하기] 버튼 클릭 시)**
- 마이페이지로 이동하지 않고, 그 자리에서 바로 모달로 설정
- 모달 항목 (2개만, 간결하게):
  - 현재 등급 (IL, IM1, IM2, IM3, IH, AL)
  - 목표 등급 (IL, IM1, IM2, IM3, IH, AL)
- 시험 예정일: 모달에서 제외 (마이페이지에서 별도 설정)

**3) 데이터 저장**
- 기존 `user_metadata`의 `target_grade` 활용
- `current_grade` 필드 추가 필요 (현재 등급)

### 구현 시 참고
- 배경설문 주제는 오픽톡닥이 고정 제공 → 사용자 선택 불필요
- 마이페이지의 기존 등급 설정 UI도 유지 (모달과 동기화)

## 논의 2: 시험후기 제출 — 보상 체계

### 현황
- 시험후기 데이터 = 데이터랩 빈도 분석의 핵심 자산
- 현재 submissions 테이블 미구현 (Step 1 대기 중)
- 크레딧 시스템은 이미 구축됨 (user_credits 테이블)

### 핵심 질문
> 사용자가 시험후기를 제출하면 어떤 보상을 줄 것인가?

### 선택지

#### A안: 크레딧 보상
- 후기 제출 1건당 모의고사 또는 스크립트 크레딧 지급
- 예: 후기 1건 = 모의고사 1회 or 스크립트 3회

| 장점 | 단점 |
|------|------|
| 기존 크레딧 시스템 활용, 구현 간단 | 크레딧 남용 가능 (허위 제출) |
| 직접적 금전 가치 → 동기부여 강력 | 유료 상품 가치 희석 우려 |
| 사용자가 보상 체감을 즉시 느낌 | 크레딧 경제 밸런스 조정 필요 |

#### B안: 기능 해금
- N건 제출 시 특정 기능 해금
- 예: 3건 제출 → 데이터랩 상세 분석 열람, 5건 → AI 튜터 1회 체험

| 장점 | 단점 |
|------|------|
| 크레딧 경제에 영향 없음 | 해금할 기능 설계가 복잡 |
| 서비스 탐색 유도 효과 | 보상 체감이 느림 (N건 모아야) |
| 유료 전환 유도 가능 | 기능이 아직 구현 안된 게 많음 |

#### C안: 포인트/배지 (게이미피케이션)
- 제출 시 포인트 적립 + 누적 배지
- 예: 후기 10건 → "데이터 기여자" 배지, 리더보드

| 장점 | 단점 |
|------|------|
| 커뮤니티 참여 동기 | 실질적 보상 부족 → 동기부여 약할 수 있음 |
| 장기 사용자 리텐션 효과 | 포인트/배지 시스템 추가 구현 필요 |
| 사회적 인정 욕구 자극 | 초기 사용자 적을 때 효과 미미 |

#### D안: 크레딧 + 배지 혼합 (권장)
- 즉시 보상: 후기 1건당 소량 크레딧 (예: 스크립트 1회)
- 누적 보상: N건 달성 시 배지 또는 추가 크레딧 보너스
- 품질 보상: 다른 사용자에게 "도움됨" 평가 받으면 추가 크레딧

| 장점 | 단점 |
|------|------|
| 즉시 + 장기 동기부여 모두 확보 | 구현 복잡도 증가 |
| 품질 관리 메커니즘 내장 | 밸런스 조정 필요 |
| 단계적 구현 가능 (즉시 보상 먼저) | - |

### 결정 ✅

- **선택**: A안 — 크레딧 보상
- **결정일**: 2026-02-21

### 결정 내용

**1) 보상 규모**
- 시험후기 1건 제출 = 스크립트 생성 크레딧 2회 지급
- `user_credits.script_credits`에 +2 (영구 크레딧)

**2) 남용 방지**
- 월 2건까지만 보상 지급 (월 최대 스크립트 4회 획득)
- 동일 시험일 중복 제출 불가
- 시험 날짜, 출제 문항 최소 3개 이상 등 필수 입력 항목 설정

### 구현 시 참고
- 월별 제출 횟수 추적 필요 (submissions 테이블에서 카운트)
- 보상 지급 시 `script_credits` (영구)에 추가 — 플랜 크레딧과 분리

## 논의 3: 대시보드 — 학습 현황 표시

### 현황
- 대시보드 완전 하드코딩 상태 (통계, 로드맵 모두 정적)
- user_credits, target_grade 등 데이터는 이미 DB에 존재
- 학습 이력 테이블(모의고사 결과, 스크립트 생성 이력 등)은 아직 미구현

### 핵심 질문
> 대시보드에서 어떤 학습 지표를 어떤 방식으로 보여줄 것인가?

### 선택지

#### A안: 최소 정보 (현재 상태 중심)
표시 항목:
- 현재 플랜 & 잔여 크레딧 (user_credits 연동)
- 목표 등급 & 시험일 D-day (user_metadata 연동)
- 학습 모듈 바로가기 (현행 유지)

| 장점 | 단점 |
|------|------|
| 즉시 구현 가능 (데이터 이미 존재) | 학습 동기부여 효과 약함 |
| 깔끔하고 단순한 UI | "학습 플랫폼" 느낌 부족 |
| 핵심 모듈 이관에 집중 가능 | - |

#### B안: 학습 진행률 중심
A안 + 추가 항목:
- 학습 로드맵 진행률 (5단계 체크리스트, 실제 활동 기반)
- 최근 활동 타임라인 (최근 생성한 스크립트, 모의고사 결과 등)
- 연속 학습일 (streak)

| 장점 | 단점 |
|------|------|
| 학습 흐름 안내 효과 | 활동 이력 테이블 필요 (추가 DB 설계) |
| 다음에 뭘 할지 명확 | 각 모듈 구현 후에야 실제 데이터 채워짐 |
| 게이미피케이션 요소 (streak) | - |

#### C안: 데이터 분석 중심
B안 + 추가 항목:
- 모의고사 점수 추이 차트
- 약점 영역 분석 (answer_type별 취약 지표)
- 주간/월간 학습 통계
- 등급별 예상 달성률

| 장점 | 단점 |
|------|------|
| 전문적 학습 플랫폼 느낌 | 구현 복잡도 높음 (차트 라이브러리 등) |
| 데이터 기반 서비스 차별화 | 모의고사/튜터링 모듈 완성 후에야 의미 있음 |
| "데이터 기반" 브랜드 강화 | 초기엔 빈 차트 → 오히려 역효과 |

#### 권장: A안 즉시 구현 → 모듈 이관에 따라 B→C 점진 확장
- **Phase 3 현재**: A안 (크레딧 + 목표 등급 + D-day 연동)
- **모듈 이관 완료 시**: B안 추가 (로드맵 진행률 + 최근 활동)
- **전체 완성 후**: C안 추가 (점수 추이 + 약점 분석)

### 결정 ✅

- **선택**: 점진 확장 (A → B → C)
- **결정일**: 2026-02-21

### 결정 내용

**1단계 — Phase 3 현재 (즉시 구현)**
- 현재 플랜 & 잔여 크레딧 (user_credits 연동)
- 목표 등급 & 시험일 D-day (user_metadata 연동)
- 학습 모듈 바로가기 (현행 유지)

**2단계 — 핵심 모듈 이관 완료 후**
- 학습 로드맵 진행률 (5단계 체크리스트, 실제 활동 기반)
- 최근 활동 타임라인
- 연속 학습일 (streak)

**3단계 — 전체 서비스 완성 후**
- 모의고사 점수 추이 차트
- 약점 영역 분석 (answer_type별)
- 주간/월간 학습 통계

### 구현 시 참고
- 1단계는 기존 데이터만으로 즉시 가능
- 2~3단계는 각 모듈 이관 시 활동 이력 테이블을 함께 설계
- 차트 라이브러리, streak 기준 등은 해당 단계 진입 시 결정

## 논의 4: 브랜딩 — "오픽톡닥" 네이밍 재검토

### 현황
- **소리담**: "나의 목소리에 나의 스토리를 담다" → 스토리텔링 + 음성의 콘셉트
- **오픽톡닥**: "톡닥이 AI 주치의" → 진단/처방 메타포
- **핵심 가치**: 후기 데이터 기반 시험 범위 축소 → 효율적 학습
- **외부 피드백**: "데이터 기반 접근이 영리하고 신뢰감 있다"

### 핵심 질문
> "오픽톡닥"이 서비스의 핵심 가치를 잘 전달하는가?

### 선택지

#### A안: "오픽톡닥" 유지
- 현재 브랜드 그대로 진행
- "톡닥이" AI 주치의 캐릭터 = 진단 + 처방 메타포

| 장점 | 단점 |
|------|------|
| 이미 도메인/인프라/PG심사 모두 세팅 완료 | "데이터 기반" 핵심 가치가 이름에 안 담김 |
| 변경 비용 0 | "톡닥" = 의료? 대화? 모호할 수 있음 |
| "AI 주치의" 콘셉트는 차별점 있음 | 소리담의 직관적 네이밍 대비 약함 |
| 발음하기 쉽고 기억하기 좋음 | - |

#### B안: 네이밍 변경 (데이터 키워드 반영)
- 데이터 기반 학습을 이름에 반영
- 예시: "오픽데이터랩", "오픽스코프", "오픽인사이트" 등

| 장점 | 단점 |
|------|------|
| 핵심 가치(데이터 기반)가 이름에서 드러남 | 도메인/인프라/PG심사 전부 재설정 |
| 마케팅 메시지와 일치 | 비용과 시간 소모 큼 |
| 전문적/신뢰감 있는 이미지 | 현 단계에서 브랜드 변경은 리스크 |

#### C안: 이름 유지 + 슬로건/부제로 보완 (권장)
- "오픽톡닥" 이름은 유지
- 슬로건으로 핵심 가치 전달
- 예: "오픽톡닥 — 데이터가 좁혀주는 OPIc 학습"
- 예: "오픽톡닥 — 수험생 후기 데이터로 시험 범위를 좁혀드립니다"
- 예: "오픽톡닥 — AI 주치의가 데이터로 진단하고 처방합니다"

| 장점 | 단점 |
|------|------|
| 인프라 변경 없음 (비용 0) | 이름 자체의 한계는 남음 |
| 슬로건으로 핵심 가치 명확 전달 | 슬로건이 길어질 수 있음 |
| "AI 주치의 + 데이터 기반" 두 마리 토끼 | - |
| 런칭 후 반응 보고 추후 판단 가능 | - |

### 결정 ✅

- **선택**: 이름 유지 (가칭) + 브랜딩 전면 재검토
- **결정일**: 2026-02-21

### 결정 내용

**1) 이름**
- "오픽톡닥"은 가칭으로 유지
- 도메인/인프라/PG심사는 현행대로 진행 (변경 없음)
- 정식 네이밍은 서비스 완성 후 재검토 가능

**2) 브랜드 컨셉: "나는 내 삶의 주인공이다"**

핵심 메시지:
- 화려한 스토리가 아닌, 나의 평범한 일상이 가장 완벽한 대본
- 남의 삶을 흉내 내는 억지 연기 대신, 나의 진짜 이야기를 할 때 가장 나답고 돋보인다
- 오픽톡닥은 데이터로 "무대"를 좁혀주고, 사용자는 그 위에서 자기 이야기를 펼친다

브랜드 메타포: **무대/연극** (대본, 조명, 명대사, 리허설, 디렉팅, 큐 사인, 주인공)

태그라인: **"말하다, 나답게."**
